How It Works section: Explain /etc/passwd format (7 colon-separated fields: username, x placeholder, UID, GID, GECOS, home, shell), /etc/shadow format (hashed passwords, aging fields), /etc/group format. Callout type="deep-dive" bridging to file permissions: reference back to Linux Fundamentals lesson 4 — "Remember that file ownership uses UID/GID numbers, not names. The username-to-UID mapping in /etc/passwd is what ls -l uses to display human-readable names."

How It Works section: Explain what an init system does (PID 1, process parent, dependency management). Brief history — SysV init scripts were sequential and fragile; systemd parallelizes boot and manages dependencies declaratively. Explain unit file anatomy: [Unit] section (Description, After, Requires, Wants), [Service] section (Type=simple|forking|oneshot|notify, ExecStart, ExecStop, Restart, RestartSec, StandardOutput), [Install] section (WantedBy). Explain targets as systemd's replacement for runlevels (multi-user.target, graphical.target). Explain the service lifecycle: loaded -> inactive -> activating -> active -> deactivating -> inactive. Explain journald as systemd's logging daemon — structured binary logs, not text files.

Include the unit file example from RESEARCH.md (hello.service). Include systemctl commands: start, stop, restart, reload, enable, disable, status, is-active, is-enabled, daemon-reload. Include journalctl basics: -u (unit), -f (follow), -b (boot), --since/--until, -p (priority).

Callout type="warning": "In this course we practice systemd inside Docker containers using a special systemd-enabled image. In production you manage systemd on real Linux hosts — containers give us a safe sandbox to practice the commands without risking a real system."

Exercise: ExerciseCard difficulty="Foundation", scenario "Your team deployed a Node.js API but it keeps crashing. You need to create a systemd service that automatically restarts it." Steps: write a hello.sh script, create /etc/systemd/system/hello.service unit file, daemon-reload, start and enable, verify with status and journalctl. VerificationChecklist items: service file exists, service is active, service is enabled, journalctl shows output.

**04-disk-management.mdx (SYS-04)**
Frontmatter: title "Disk Management: Block Devices, Filesystems, and Mounts", module "System Administration", moduleSlug "04-sysadmin", lessonSlug "04-disk-management", order 4, difficulty "Intermediate", estimatedMinutes 30, prerequisites ["01-linux-fundamentals/03-linux-filesystem", "04-sysadmin/01-user-management"], tags ["disk", "fdisk", "mount", "fstab", "lvm", "df", "du", "block-devices", "filesystems"].

How It Works section: Explain block devices — how the kernel presents physical/virtual disks as /dev/sdX or /dev/vdX. Explain partitions (MBR vs GPT partition tables), filesystem creation (mkfs.ext4), and the mount operation (connecting a filesystem to the directory tree). Explain /etc/fstab format (6 fields: device/UUID, mount point, fs type, options, dump, pass). Explain df vs du — df reports filesystem-level usage, du reports directory-level usage. Brief LVM overview (PV -> VG -> LV concept only, per locked decision — NOT full LVM deep-dive). Explain UUIDs vs device names — "Use UUIDs in fstab because /dev/sdX names can change between reboots when disks are added/removed."

Callout type="deep-dive" bridging to Linux filesystem lesson: "In Linux Fundamentals Lesson 3, you learned that everything is a file — including block devices. /dev/sda is a file that represents a disk. When you run fdisk /dev/sda, you are reading and writing that file's contents — which happen to be the partition table."

Callout type="warning": "Never add an fstab entry without testing with mount -a first. A bad fstab entry prevents the system from booting normally — you will end up in emergency mode. Also use the nofail option for non-critical mounts so the system can still boot if the disk is missing."

Exercise: ExerciseCard difficulty="Intermediate", scenario "Your team's server is running low on disk space. Identify what's consuming storage, and set up a new data partition for the application logs." Steps: use df -h to check usage, du -sh /var/log/* to find large dirs, create a loopback device with truncate + losetup, partition with fdisk, format with mkfs.ext4, mount, add fstab entry. VerificationChecklist: df shows new mount, fstab entry correct, data persists after umount/mount.
