**01-cloud-concepts.mdx (CLD-01)**
Frontmatter: title "Cloud Computing: What It Actually Is (Not Marketing Buzzwords)", module "Cloud Fundamentals", moduleSlug "07-cloud", lessonSlug "01-cloud-concepts", order 1, difficulty "Foundation", estimatedMinutes 25, prerequisites ["06-iac/01-iac-concepts"], tags ["cloud", "iaas", "paas", "saas", "regions", "availability-zones", "aws", "gcp", "azure"].

Overview: "Why This Matters" scenario — your team's app runs on a single Docker host. It works great until the machine fails at 3 AM, or you need to handle 10x traffic for a product launch. Cloud computing solves these problems — but understanding what it actually does (versus what vendors say it does) requires mapping it to concepts you already know.

How It Works section: Define cloud computing as "someone else's computers, managed via APIs." Break down service models with concrete analogies: IaaS = renting a bare-metal machine (like your Docker host, but you do not own the hardware), PaaS = a managed container service (like docker compose but the provider handles the host), SaaS = a fully managed application (Gmail — you don't run the mail server). Explain regions (physical datacenter locations), availability zones (isolated buildings within a region — if one catches fire, the others survive), and why multi-AZ deployment matters. Include the three major providers: AWS (2006, largest market share), GCP (Google infrastructure), Azure (Microsoft enterprise integration). Explain that this course is provider-agnostic — the concepts are identical, only the service names differ.

Include a CodeBlock showing a docker compose file that runs an app and explain: "This is your app on a single host. Cloud compute lets you run this same container across multiple AZs so a single failure does not take you down."

Callout type="deep-dive": "You already know IaC from the previous module. Cloud providers are just very large providers for OpenTofu — instead of provider 'docker' {} you write provider 'aws' {} or provider 'google' {}, and resources map to cloud services instead of local containers."

Exercise: ExerciseCard difficulty="Foundation", scenario "Map your local Docker setup to cloud equivalents. For each component of your Docker Compose stack, identify the cloud service that replaces it." Conceptual exercise — VerificationChecklist: can name the three service models (IaaS/PaaS/SaaS), can explain regions vs AZs, can identify which cloud service model docker compose + host = (IaaS with container runtime).

**02-compute.mdx (CLD-02)**

How It Works section: Three tiers of cloud compute, each mapped to what the learner knows:

1. Virtual Machines — "Remember learning about namespaces and cgroups in the Docker module? A VM goes further — it virtualizes the entire hardware, including CPU, memory, and network interfaces. You get a full OS kernel, not just an isolated process." Explain EC2 (AWS), Compute Engine (GCP), Azure VMs. Show that you SSH into a VM just like the Docker containers in this course, but it is a full machine.

2. Managed Containers — "docker compose up runs containers on your local Docker daemon. Cloud container services (ECS, Cloud Run, Container Apps) do the same thing but across multiple machines, with automatic load balancing and scaling." Explain ECS/Fargate (AWS), Cloud Run (GCP), Container Apps (Azure). Include CodeBlock showing a simple Dockerfile and explain how the same image runs locally or in the cloud.

3. Serverless — "No equivalent in Docker. You write a single function, upload it, and the cloud runs it only when triggered. You pay per invocation, not per hour." Explain Lambda (AWS), Cloud Functions (GCP), Azure Functions. Explain when serverless is appropriate (event-driven, low-frequency tasks) vs when containers are better (steady traffic, complex apps).

Include the compute rows from the cloud service mapping table.

Callout type="tip": "Start with containers. If you know Docker (and you do), managed container services are the fastest path to cloud deployment. VMs are for when you need full OS control. Serverless is for event-driven glue code."

Exercise: ExerciseCard difficulty="Foundation", scenario "Your team needs to deploy the exercise app from the Docker module to the cloud. **03-cloud-networking.mdx (CLD-03)**
Frontmatter: title "Cloud Networking: VPCs, Subnets, and Load Balancers", module "Cloud Fundamentals", moduleSlug "07-cloud", lessonSlug "03-cloud-networking", order 3, difficulty "Intermediate", estimatedMinutes 30, prerequisites ["07-cloud/01-cloud-concepts", "02-networking/02-tcp-ip-stack", "03-docker/05-docker-networking"], tags ["vpc", "subnets", "load-balancer", "security-groups", "cloud-networking", "aws", "gcp", "azure"].

How It Works section: Map every cloud networking concept to something the learner already knows:

1. VPC (Virtual Private Cloud) = docker network create — "A VPC is an isolated network, just like a Docker bridge network. Containers (or VMs) inside a VPC can talk to each other. Traffic from outside must be explicitly allowed." Explain AWS VPC, GCP VPC Network, Azure Virtual Network.

2. Subnets = Docker network --subnet — "In the networking module, you learned that subnets partition an IP range. Cloud subnets do the same thing, but each subnet lives in a specific availability zone." Explain public subnets (internet-reachable) vs private subnets (internal only). Show the Docker network create --subnet command alongside the cloud equivalent concept.

3. Load Balancers = nginx reverse proxy — "In the Docker Compose lesson, you used nginx as a reverse proxy distributing traffic to multiple app containers. Cloud load balancers do this at massive scale, with health checks and automatic failover." Explain ALB (AWS), Cloud Load Balancing (GCP), Azure Load Balancer.

4. Security Groups = iptables/ufw — "In the firewall lesson, you wrote iptables rules to control traffic. Cloud security groups are the same concept — inbound/outbound rules attached to instances. But they are stateful by default (if you allow inbound, the response is automatically allowed)."

Include the networking rows from the cloud service mapping table.

Include a CodeBlock showing a Docker Compose file with nginx lb + 2 app containers, then a diagram mapping each component to the cloud equivalent (VPC = the default network, nginx = ALB, app containers = EC2/ECS tasks, expose = security group rule).

Callout type="deep-dive": "Remember the DNS resolution chain from the networking module? Cloud providers have their own DNS services (Route 53, Cloud DNS, Azure DNS) that work the same way — authoritative nameservers responding to queries. The only difference is you manage records via API instead of zone files."

Exercise: ExerciseCard difficulty="Intermediate", scenario "Design a cloud network architecture for a web application with a load balancer, two application servers in different AZs, and a database in a private subnet." **04-cloud-storage.mdx (CLD-04)**
Frontmatter: title "Cloud Storage: Object, Block, and File Storage", module "Cloud Fundamentals", moduleSlug "07-cloud", lessonSlug "04-cloud-storage", order 4, difficulty "Intermediate", estimatedMinutes 25, prerequisites ["07-cloud/01-cloud-concepts", "03-docker/04-docker-volumes"], tags ["storage", "object-storage", "block-storage", "file-storage", "s3", "ebs", "gcs", "azure-blob"].

How It Works section: Three storage types, each mapped to what the learner knows:

1. Block Storage = Docker named volumes — "In the Docker module, named volumes are persistent storage attached to containers. Cloud block storage (EBS, Persistent Disk, Azure Disk) works the same way — a virtual disk attached to a VM or container. It persists when the instance stops." Explain that block storage is formatted with a filesystem (ext4, xfs) just like the disk management lesson in sysadmin. Show TerminalBlock with docker volume create and docker run -v, then explain the cloud equivalent.

2. Object Storage = no direct Docker equivalent — "This is where cloud adds something new. Object storage (S3, Cloud Storage, Azure Blob) stores files as key-value pairs accessible via HTTP URLs. No filesystem, no mount points, no directories — just a flat namespace of objects." Explain when to use object storage (static files, backups, logs, media) vs block storage (databases, application data). Include a CodeBlock showing curl to download from an S3-style URL to make it concrete.

3. File Storage = NFS/shared mount — "If block storage is a hard drive and object storage is an HTTP file server, file storage is a network drive. Multiple instances mount the same filesystem simultaneously." Explain EFS (AWS), Filestore (GCP), Azure Files. Note this maps to NFS concepts from the networking module.

Include the storage rows from the cloud service mapping table.

Callout type="deep-dive": "Remember Docker's storage driver and overlay filesystem from the container internals lesson? Cloud VMs use a similar layered approach — the root volume is an EBS/Persistent Disk snapshot, and additional volumes are attached as block devices. The df and mount commands you learned work the same way on cloud VMs."

Callout type="tip": "The most common mistake: using block storage for files that should be in object storage. If files are accessed via HTTP and do not need filesystem operations (ls, chmod), use object storage — it scales infinitely and costs less."

Exercise: ExerciseCard difficulty="Intermediate", scenario "Your application stores user uploads, database files, and shared configuration. Choose the correct storage type for each and explain why." VerificationChecklist: can explain why user uploads go in object storage (HTTP access, scales infinitely), can explain why database files go in block storage (filesystem operations, IOPS), can name the service in all three providers.

**05-iam.mdx (CLD-05)**
Frontmatter: title "Cloud IAM: Identity, Policies, and Least Privilege", module "Cloud Fundamentals", moduleSlug "07-cloud", lessonSlug "05-iam", order 5, difficulty "Intermediate", estimatedMinutes 25, prerequisites ["07-cloud/01-cloud-concepts", "01-linux-fundamentals/04-file-permissions", "04-sysadmin/01-user-management"], tags ["iam", "identity", "policies", "roles", "least-privilege", "permissions", "security", "aws", "gcp", "azure"].

How It Works section: Map every IAM concept to Linux permissions:

1. IAM Users/Groups = Linux users/groups — "In the Linux Fundamentals module, you created users with useradd and groups with groupadd. Cloud IAM has the same abstraction — users are identities, groups collect users for batch permissions." Explain AWS IAM Users/Groups, GCP Cloud IAM, Azure AD.

2. IAM Policies = chmod/chown — "chmod 640 file.txt gives the owner read/write, the group read-only, and others nothing. An IAM policy does the same thing for cloud resources: who (principal) can do what (action) on which resource (resource ARN/ID)." Show a side-by-side comparison: chmod command on the left, IAM policy JSON snippet on the right. Include a CodeBlock showing a simplified IAM policy JSON.

3. IAM Roles = sudo — "When you run sudo, you temporarily escalate to root privileges. IAM role assumption is the same concept — a user or service temporarily assumes a role with elevated permissions, and the assumption can be time-limited and audited." Explain AWS STS AssumeRole, GCP Service Accounts, Azure Managed Identity.

4. Least Privilege principle — "Grant the minimum permissions needed. On Linux, you learned not to run everything as root. In the cloud, the same principle applies — a web server should not have permission to delete databases."

Include the IAM rows from the cloud service mapping table.

Callout type="warning": "The most common cloud security mistake is granting * (all permissions) to get something working quickly and never tightening it. Treat cloud IAM like Linux permissions — start with no access and add only what is needed."

Callout type="deep-dive": "On Linux, /etc/sudoers controls who can sudo. In AWS, a Trust Policy on a role controls who can assume it. The concept is identical — a declaration of 'this identity can temporarily become this other identity.'"

Exercise: ExerciseCard difficulty="Intermediate", scenario "Your application needs to read from an S3 bucket and write to a database. Design the IAM policy using least privilege — no more permissions than required." VerificationChecklist: can map Linux user/group to IAM user/group, can write a conceptual policy allowing only S3 GetObject and DB write, can explain why role assumption is like sudo, can explain least privilege in both Linux and cloud terms.
