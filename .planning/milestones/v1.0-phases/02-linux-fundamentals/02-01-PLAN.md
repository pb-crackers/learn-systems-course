Content (15-25 min reading target):

- **How It Works**:
  - Users and groups: /etc/passwd, /etc/group, UID/GID, how the kernel identifies processes by UID not username
  - Permission model: read/write/execute for owner/group/other. What each permission means for FILES vs DIRECTORIES (execute on a directory = ability to cd into it and list contents)
  - Permission representation: `rwxr-xr--` breakdown, numeric (octal) representation (755, 644, 600, etc.), how to convert between symbolic and numeric
  - chmod: symbolic mode (u+x, g-w, o=r) vs numeric mode (chmod 755). When to use each.
  - chown and chgrp: changing ownership, recursive with -R, why only root can chown

  - Use `<Callout type="deep-dive">` for setuid/setgid security implications

**Lesson 5 — 05-processes.mdx (LNX-05):**
Frontmatter: title "Processes", description "PID, fork/exec, process states, signals, ps, top, kill — how Linux runs your programs", module "Linux Fundamentals", moduleSlug "01-linux-fundamentals", lessonSlug "05-processes", order 5, difficulty "Intermediate", estimatedMinutes 25, prerequisites ["01-linux-fundamentals/04-file-permissions"], tags ["processes", "pid", "fork", "exec", "signals", "ps", "top", "kill"].

Content (15-25 min reading target):
- **Overview**: "Why This Matters" — "A production service is consuming 100% of one CPU core and you need to figure out which process, what it's doing, and how to stop it gracefully without losing data. Understanding processes, signals, and process states is the difference between a clean fix and a hard reboot."
- **How It Works**:
  - What is a process: running instance of a program, has PID, parent PID (PPID), UID, environment, file descriptors, memory map
  - fork/exec model: fork() creates a copy of the parent process (new PID, same code), exec() replaces the process image with a new program. This two-step model is fundamental to Unix. Use `<Callout type="deep-dive">` for fork/exec with a shell pipeline example
  - PID 1 (init/systemd): the root of the process tree, adopts orphaned processes, why it matters for containers (Docker uses PID 1 for signal handling)
  - Process states: Running (R), Sleeping (S — waiting for I/O or event), Stopped (T — SIGSTOP/SIGTSTP), Zombie (Z — exited but parent hasn't called wait()). Show `ps aux` output and explain STAT column
  - /proc/[pid]/: cmdline, status, fd/, maps — each process has a directory in /proc
  - Signals: SIGTERM (15, graceful shutdown), SIGKILL (9, immediate, cannot be caught), SIGINT (2, Ctrl+C), SIGHUP (1, terminal hangover / config reload), SIGSTOP/SIGCONT (pause/resume). Use `<Callout type="warning">` about SIGKILL as last resort — process can't clean up
  - Process monitoring: ps aux (snapshot), top/htop (real-time), /proc/loadavg (load average = runnable + uninterruptible processes)
  - Background processes: &, jobs, fg, bg, nohup, disown
  - Use `<Callout type="tip">` for "kill -0 PID" to test if process exists without sending a signal
- **Hands-On Exercise**: ExerciseCard with difficulty "Intermediate", real-world scenario about investigating a runaway process on a server. Steps: launch background processes, examine them with ps and /proc, send signals (SIGTERM, SIGSTOP, SIGCONT), create and identify a zombie process, use top to monitor resource usage. Docker lab launch via TerminalBlock. VerificationChecklist items: "You can find a process's PID using ps aux | grep", "You can read a process's command line from /proc/PID/cmdline", "You successfully sent SIGTERM to a process".
- **Quick Reference**: ps aux, top, kill, kill -l, jobs, fg, bg, /proc/PID paths, signal names and numbers
