**01-observability-concepts.mdx (MON-01)**
Frontmatter: title "Observability: Metrics, Logs, and Traces", description "The three pillars of observability — what each tells you, when to use which, and how they connect to the monitoring tools you will learn", module "Monitoring & Observability", moduleSlug "08-monitoring", lessonSlug "01-observability-concepts", order 1, difficulty "Foundation", estimatedMinutes 25, prerequisites ["04-sysadmin/06-system-monitoring"], tags ["observability", "metrics", "logs", "traces", "monitoring", "three-pillars", "prometheus", "grafana"].

Overview: "Why This Matters" scenario — your Node.js app is slow. Users are complaining. You SSH in and run top — CPU is fine, memory is fine. The problem is not the machine; it is a database query taking 3 seconds. top cannot tell you that. You need application-level observability: metrics to see request latency, logs to see the slow query, traces to see which service in the call chain is the bottleneck. That is the gap between system monitoring (Phase 5) and observability.

How It Works section: Start with the explicit bridge: "In the System Administration module, you used top, htop, vmstat, and iostat. Those tools monitor the machine — CPU, memory, disk I/O. Observability goes further: it monitors the application running on the machine." Define the three pillars:

1. **Metrics** — numeric measurements collected over time. Counter (always goes up: total requests), Gauge (goes up and down: current memory), Histogram (distribution: request latencies). Metrics answer "how much" and "how fast." Tool: Prometheus. Analogy: a car's speedometer and fuel gauge — always updating, always numeric, always time-series.

2. **Logs** — timestamped text records of events. Structured (JSON) vs unstructured (plain text). Logs answer "what happened." Tool: Loki, ELK. Analogy: a flight recorder's black box — detailed record of everything, but expensive to search through at scale.

3. **Traces** — end-to-end request paths across services. Spans, trace IDs, parent-child relationships. Traces answer "where did the time go." Tool: Jaeger, Zipkin (conceptual only per CONTEXT.md — distributed tracing is v2). Analogy: tracking a package through shipping — each warehouse stamp is a span.

Explain when to use which pillar: start with metrics (cheap, fast alerting on thresholds), drill into logs when metric anomaly is found (what caused it), use traces when request crosses multiple services (where exactly is it slow).

Callout type="deep-dive": "The difference between monitoring and observability: monitoring tells you WHEN something is wrong (alert: latency > 500ms). Observability tells you WHY it is wrong (trace shows database service added 450ms). You need both."

Callout type="tip" bridging from Phase 5: "You already know process-level monitoring — top shows CPU and memory per process. Prometheus extends this to application-level metrics: request count, error rate, latency percentiles. Same concept, different granularity."

Steps: for each scenario, identify the primary pillar (metrics, logs, or traces), explain why, and describe what specific data you would look for. VerificationChecklist: can classify problems to correct pillar, can explain why metrics are cheapest to query, can explain when traces are necessary.

**02-prometheus.mdx (MON-02)**
Frontmatter: title "Prometheus: How Metrics Collection Actually Works", description "From counters and gauges to PromQL queries — how Prometheus scrapes, stores, and queries time-series metrics", module "Monitoring & Observability", moduleSlug "08-monitoring", lessonSlug "02-prometheus", order 2, difficulty "Foundation", estimatedMinutes 30, prerequisites ["08-monitoring/01-observability-concepts"], tags ["prometheus", "promql", "metrics", "counter", "gauge", "histogram", "alerting", "scrape"].

How It Works section: Explain the pull-based scrape model — Prometheus periodically GETs your app's /metrics endpoint (contrast with push-based systems where apps send metrics to a collector). Explain why pull is better for reliability: Prometheus controls the pace, knows immediately if a target is down (up metric), does not need apps to know where to push. Show the scrape cycle: every scrape_interval (default 15s), Prometheus fetches /metrics, parses the Prometheus text format, appends to TSDB. Explain Prometheus TSDB: time-series stored as metric_name{label1="val",label2="val"} + timestamp + value. Explain metric types in detail:
- Counter: monotonically increasing (http_requests_total). Only use rate() with counters — raw value is meaningless (it always goes up). rate(http_requests_total[5m]) = requests per second over last 5 minutes.
- Gauge: can go up or down (active_connections, process_resident_memory_bytes). Read the raw value directly.
- Histogram: tracks distribution of values in configurable buckets (http_request_duration_seconds_bucket). The canonical PromQL: histogram_quantile(0.95, sum(rate(http_request_duration_seconds_bucket[5m])) by (le)) = P95 latency.
- Summary: similar to histogram but calculated client-side (mention, but recommend histograms — they are aggregatable across instances).

Explain PromQL basics: instant vectors (http_requests_total{job="app"}), range vectors ([5m] suffix), functions (rate, increase, histogram_quantile, sum, avg), aggregation by label (sum by (route)).

Show the 8 essential PromQL patterns from RESEARCH.md code examples section, each with explanation of what it returns and when to use it.

Explain alerting rules: groups, rules with expr/for/labels/annotations. Show the AppDown, HighRequestLatency, and HighErrorRate alert rules from RESEARCH.md Pattern 4. Explain the "for" duration (pending vs firing state). Explain Alertmanager integration: Prometheus evaluates rules -> sends firing alerts to Alertmanager -> Alertmanager routes/groups/deduplicates -> sends notifications.

Callout type="warning": "prom/node-exporter does NOT work on macOS. It reads Linux /proc and /sys filesystems that do not exist on macOS. For container-level metrics, use gcr.io/cadvisor/cadvisor which uses the Docker API instead."

Callout type="tip": "To reload Prometheus config without restarting: curl -X POST http://localhost:9090/-/reload. This requires the --web.enable-lifecycle flag on the Prometheus container."

Callout type="deep-dive" on the up metric: "up{job=\"monitored-app\"} == 1 only means Prometheus successfully scraped the /metrics endpoint in the last scrape interval. The app could still be returning HTTP 500 errors to real users. Always combine up with application-level metrics like error rate."

Exercise: ExerciseCard difficulty="Foundation", scenario "Your team deployed a Node.js API instrumented with prom-client. Use the Docker Compose monitoring stack to write PromQL queries that answer real questions about the app's health." Steps: navigate to docker/monitoring/, run docker compose up -d, wait for stack to start, open Prometheus UI at http://localhost:9090, write PromQL queries: (1) check if app target is up, (2) total HTTP requests, (3) request rate over 5 minutes, (4) P95 request latency, (5) current heap memory usage. VerificationChecklist: up{job="monitored-app"} returns 1, rate(http_requests_total[5m]) returns a rate value, histogram_quantile query returns a latency value, process_resident_memory_bytes shows memory.

**04-log-aggregation.mdx (MON-04)**
Frontmatter: title "Log Aggregation: Centralizing Logs from Every Service", description "From scattered log files to centralized search — how Loki and ELK collect, store, and query logs at scale", module "Monitoring & Observability", moduleSlug "08-monitoring", lessonSlug "04-log-aggregation", order 4, difficulty "Intermediate", estimatedMinutes 25, prerequisites ["08-monitoring/03-grafana"], tags ["logs", "loki", "promtail", "elk", "logql", "centralized-logging", "log-aggregation"].

How It Works section: Start with the problem — "In the Linux Fundamentals module, you read log files with cat, grep, and tail. That works when you have one server. When you have 50 containers across 10 hosts, you need centralized logging." Explain the centralized logging pipeline: collect (agents on each host) -> ship (transport to central store) -> store (indexed, searchable) -> query (search UI).

Explain two major stacks:
1. **ELK (Elasticsearch + Logstash + Kibana)**: Elasticsearch stores and indexes logs (full-text search), Logstash transforms and routes logs, Kibana visualizes. Pros: powerful full-text search, mature ecosystem. Cons: resource-heavy (Elasticsearch needs 2+ GB heap), complex to operate. Conceptual only — no lab deployment.

2. **Loki + Promtail + Grafana**: Loki stores logs indexed only by labels (not full-text), Promtail ships logs to Loki, Grafana queries Loki with LogQL. Pros: lightweight (does not index log content), uses same Grafana you already know, works well with Prometheus labels. Cons: no full-text index — searches by label first, then grep through matching streams.

Explain the Loki model in detail: Promtail runs on each host, tails log files or reads Docker container logs via docker.sock, attaches labels (job, container_name, host), ships to Loki. Loki stores log streams indexed by label combinations. LogQL queries: {job="app"} |= "error" means "find all log lines from job=app containing the string error."

Show the Loki config (loki-config.yml from RESEARCH.md) and explain each section: auth_enabled: false (single-tenant dev mode), server port 3100, inmemory ring store (dev only — production uses consul/etcd), schema_config with tsdb store, filesystem storage for chunks.

Show the Promtail config (promtail-config.yml) and explain: scrape_configs targeting /var/log and Docker socket, label extraction from container names.

Callout type="deep-dive": "Loki's key design decision: index labels, not content. ELK indexes every word in every log line — powerful but expensive. Loki only indexes the label set {job=\"app\", container=\"web\"}. To find a specific error message, you first select the log stream by labels, then Loki greps through that stream. This is why Loki uses 10x less storage than Elasticsearch for the same log volume."

Callout type="tip": "The Docker Compose monitoring stack in docker/monitoring/ includes a Loki + Promtail example. Run docker compose --profile loki up -d to start the full stack with log aggregation. Query logs in Grafana by adding Loki as a data source at http://loki:3100."

Exercise: ExerciseCard difficulty="Intermediate", scenario "Your Docker Compose application generates logs from multiple containers. Steps: navigate to docker/monitoring/, start the monitoring stack with Loki profile (docker compose --profile loki up -d), open Grafana at http://localhost:3001, add Loki data source (http://loki:3100), go to Explore, select Loki data source, write LogQL queries: (1) {container=~"monitoring-app.*"} to see app logs, (2) {container=~"monitoring-app.*"} |= "error" to filter errors, (3) count_over_time({container=~"monitoring-app.*"}[5m]) for log volume. Frontmatter: title "Incident Response: From Alert to Resolution", description "Alerting workflows, runbook patterns, postmortem templates, and on-call basics — turning monitoring data into action", module "Monitoring & Observability", moduleSlug "08-monitoring", lessonSlug "05-incident-response", order 5, difficulty "Intermediate", estimatedMinutes 25, prerequisites ["08-monitoring/04-log-aggregation"], tags ["incident-response", "alerting", "alertmanager", "runbook", "postmortem", "on-call", "pagerduty", "opsgenie"].

How It Works section: Explain the alert lifecycle: metric crosses threshold -> Prometheus evaluates alert rule -> alert enters PENDING state -> after "for" duration, alert becomes FIRING -> Prometheus sends to Alertmanager -> Alertmanager groups/deduplicates/routes -> notification sent (Slack, PagerDuty, webhook, email).

Explain Alertmanager configuration: Show the alertmanager.yml from RESEARCH.md Pattern 5. Explain route tree: group_by (which labels to group alerts by), group_wait (how long to buffer before first notification), group_interval (how long between notifications for same group), repeat_interval (how long before re-sending same alert), receiver (where to send). Explain receivers: webhook_configs for the lab exercise (sends to app itself for visibility), note that production uses Slack/PagerDuty/OpsGenie integrations.

Explain the webhook receiver approach in the lab: "For learning purposes, the webhook receiver points to the app itself at http://app:3000/alert-webhook. When an alert fires, the app logs the alert body. This creates a visible notification path without requiring Slack or PagerDuty credentials."

Explain runbooks — structured response procedures:
```
# Runbook: High Request Latency
## Symptoms
- Alert: HighRequestLatency firing
- P95 latency > 500ms for 2+ minutes

## Diagnosis Steps
1. Check Grafana dashboard — is latency high on all routes or one?
2. Check database metrics — is query time elevated?
3. Check container resource usage — is CPU/memory saturated?
4. Check recent deployments — did a code change introduce the regression?

## Resolution Actions
- If database: check slow query log, optimize or add index
- If resource saturation: scale horizontally or increase limits
- If code regression: rollback to previous version

## Escalation
- If not resolved in 30 minutes: escalate to senior on-call
```

Explain postmortems — structured retrospectives after incidents:
```
# Postmortem: API Latency Spike — 2026-03-15

## Summary
API P95 latency exceeded 2s for 45 minutes, affecting ~1200 users.

## Timeline
- 14:00 — Deploy v2.3.1 with new ORM query
- 14:15 — HighRequestLatency alert fires
- 14:20 — On-call engineer acknowledges, opens Grafana
- 14:35 — Root cause identified: N+1 query in /api/orders
- 14:45 — Hotfix deployed (eager loading), latency recovers

## Root Cause
New /api/orders endpoint used lazy loading, generating 1 SQL query per order item instead of 1 query total.

## Action Items
- [ ] Add query count metric to all API endpoints
- [ ] Add load test step to CI pipeline for latency regression
- [ ] Update code review checklist: check for N+1 queries
```

Callout type="deep-dive": "Blameless postmortems: the goal is to prevent recurrence, not to assign blame. 'Engineer X deployed bad code' is not a root cause — 'our CI pipeline has no latency regression test' is. Focus on the system, not the person."

Callout type="tip" about on-call: "On-call rotations exist so that one person is always responsible for responding to alerts. Tools like PagerDuty and OpsGenie manage rotations, escalation policies, and schedule overrides. The key rule: if you are on-call and an alert fires, you own it until it is resolved or escalated."

Callout type="tip" with What's Next bridge: "You now have the full observability toolkit: metrics (Prometheus), dashboards (Grafana), logs (Loki), and incident response. In the Advanced Capstone, you will put it all together — deploying a full application with monitoring, then diagnosing an intentional failure using everything you have learned across the entire course."
