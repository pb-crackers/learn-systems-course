
Section structure:
1. **## Overview** — "Why This Matters" scenario: "Your API is timing out — is it a DNS problem, a routing issue, or is TCP retransmission burning time? Understanding the TCP/IP stack tells you which layer to investigate."

2. **## How It Works** — Deep mechanism:
   - ### The Four Layers: Application (HTTP, DNS, SSH), Transport (TCP, UDP), Internet (IP), Link (Ethernet). ASCII diagram showing the full stack with example protocols at each layer.
   - ### Encapsulation: How data gets wrapped — Application data → TCP segment (adds source/dest port) → IP packet (adds source/dest IP) → Ethernet frame (adds source/dest MAC). ASCII diagram showing the nesting.
   - ### IP Addressing and Subnets: IPv4 address structure, dotted-decimal notation, subnet masks, CIDR notation. How to determine if two IPs are on the same subnet. Example: 192.168.1.100/24 and 192.168.1.200/24 are on the same subnet; 192.168.1.100/24 and 10.0.0.1/24 are not.
   - ### TCP: The Reliable Protocol: Three-way handshake (SYN, SYN-ACK, ACK) with ASCII diagram showing the exchange. Sequence numbers, acknowledgments, windowing concept, retransmission on timeout. Connection teardown (FIN/ACK). Use `<Callout type="deep-dive">` for congestion control (slow start, congestion avoidance).
   - ### UDP: The Fast Protocol: No handshake, no guarantees, no ordering. When to use it: DNS queries, video streaming, gaming. Why it's faster (no overhead).
   - ### Ports: How the OS multiplexes connections. Well-known ports (22 SSH, 53 DNS, 80 HTTP, 443 HTTPS). Ephemeral ports for client connections.
   - Use `<Callout type="tip">` for "When you run `ss -tlnp` (which you'll use in the troubleshooting lesson), you're seeing the transport layer — which ports are listening and what process owns them."

3. **## Hands-On Exercise** — ExerciseCard with scenario: "You're debugging why two containers can't communicate — you need to inspect IP addresses, subnet membership, and active connections." Difficulty: Foundation. Tasks: Use `ip addr show` to find IP and subnet, calculate whether two IPs are on the same subnet, use `ss -tlnp` to see listening ports, use `cat /proc/net/tcp` to see raw TCP socket data. TerminalBlock examples. VerificationChecklist: identified container IP, determined subnet, listed listening ports.

4. **## Quick Reference** — QuickReference: TCP/IP layers, common port numbers, TCP vs UDP comparison, CIDR notation examples.

```yaml
title: "DNS: The Internet's Phone Book"

```

Section structure:
1. **## Overview** — "Why This Matters" scenario: "Your application can't connect to the database at db.internal.company.com — is the server down, or is DNS failing to resolve the hostname? Understanding DNS tells you where the name lookup is breaking."

2. **## How It Works** — Deep mechanism:
   - ### What DNS Does: Maps human-readable names to IP addresses. Every network request starts with a DNS lookup (unless the destination is already an IP).
   - ### The Resolution Chain: Recursive resolver → root nameserver → TLD nameserver → authoritative nameserver. ASCII diagram showing the full resolution of "app.example.com" step by step (client → recursive resolver → root → .com TLD → example.com authoritative → answer). Explain each step.
   - ### DNS Record Types: A (IPv4), AAAA (IPv6), CNAME (alias), MX (mail), NS (nameserver), PTR (reverse), TXT (arbitrary text), SOA (zone authority). Table format with type, purpose, and example value.
   - ### DNS Caching: Browser cache → OS resolver cache → recursive resolver cache. TTL (time-to-live) controls how long a record is cached. Explain why low TTLs are used for migrations and high TTLs for stable records.
   - ### Container DNS: How Docker's embedded DNS server (127.0.0.11) resolves container names to IPs on the same network. Use `<Callout type="tip">` explaining this is exactly what learners will use in the Docker module.
   - Use `<Callout type="deep-dive">` for DNSSEC — what it is, why it exists (preventing DNS spoofing), but mark it as an advanced topic not covered in exercises.
   - Use `<Callout type="warning">` for "DNS propagation delays — when you change a DNS record, some resolvers may cache the old record for up to the previous TTL. This is why DNS changes 'take time to propagate.'"

3. **## Hands-On Exercise** — ExerciseCard with scenario: "You're migrating a service to a new server and need to verify DNS records are correct before switching traffic. You need to query DNS directly and trace the resolution path." Difficulty: Foundation. Tasks: Use `dig app.learn.local @dns` to query the lab's CoreDNS server, inspect the ANSWER section, use `dig +trace` to see the resolution chain, use `dig -x` for reverse lookup, save a dig trace to /tmp/exercise/dns-trace.txt. TerminalBlock showing the docker compose launch commands and expected dig output. VerificationChecklist: resolved forward lookup, completed reverse lookup, saved dig trace.

   Include TerminalBlock for lab launch:

   ```
   docker compose -f docker/networking/03-dns/compose.yml up -d

   ```
   And teardown:
   ```
   docker compose -f docker/networking/03-dns/compose.yml down
   ```

4. **## Quick Reference** — QuickReference: DNS record types table, dig command cheat sheet (dig +short, dig +trace, dig -x, dig @server), resolution chain summary.
