Section structure:
1. **## Overview** — "Why This Matters" scenario: "Your deployment succeeded but users are getting 502 Bad Gateway errors. Understanding HTTP tells you the request reached the load balancer but the backend didn't respond correctly — the status code is the first clue."

2. **## How It Works** — Deep mechanism:
   - ### HTTP Request/Response Cycle: Client sends request (method + path + headers + optional body), server returns response (status code + headers + body). ASCII diagram showing a complete request/response exchange with labeled parts.
   - ### HTTP Methods: GET (read), POST (create), PUT (replace), PATCH (update), DELETE (remove). When to use each. Idempotency explained.
   - ### HTTP Headers: Host, Content-Type, Authorization, User-Agent, Accept. Request headers vs response headers. Use a CodeBlock showing a real curl -v output with headers annotated.
   - ### Status Codes: 2xx success (200 OK, 201 Created), 3xx redirect (301, 302, 304), 4xx client error (400, 401, 403, 404), 5xx server error (500, 502, 503). Table format. `<Callout type="tip">` for "Most debugging starts with the status code — a 502 means the proxy reached the server but got a bad response; a 504 means the proxy couldn't reach the server at all."
   - ### HTTPS and TLS: Why encryption matters. The TLS handshake: ClientHello → ServerHello → Certificate → Key Exchange → Finished. ASCII diagram of the handshake. Certificates, certificate authorities, self-signed certs. `<Callout type="deep-dive">` for how symmetric encryption is negotiated during the handshake (asymmetric for key exchange, symmetric for data transfer).
   - ### curl: The DevOps Swiss Army Knife: `curl -v` for verbose output, `curl -I` for headers only, `curl -X POST -d`, `curl -k` for self-signed certs. This is the primary HTTP debugging tool.

3. **## Hands-On Exercise** — ExerciseCard with scenario: "You're debugging a web application deployment and need to verify the server is responding correctly, inspect response headers, and confirm HTTPS is working with a self-signed certificate." Difficulty: Foundation. Tasks: Use `curl -I http://webserver/` to get response headers, save headers to /tmp/exercise/headers.txt, use `curl -v https://webserver/` to see TLS negotiation (with -k for self-signed), use `curl -X POST -d '{"test":"data"}' -H 'Content-Type: application/json' http://webserver/api` to send a POST request. Section structure:
1. **## Overview** — "Why This Matters" scenario: "You need to access a database server that's behind a firewall and only reachable from a bastion host. SSH tunneling lets you create a secure path through the bastion without exposing the database to the internet."

2. **## How It Works** — Deep mechanism:
   - ### How SSH Works: Client connects to sshd on port 22. The server proves its identity (host key), then the client authenticates (password or key). All traffic is encrypted after key exchange.
   - ### Key-Based Authentication: ssh-keygen generates a key pair (private stays on client, public goes to server). The server checks ~/.ssh/authorized_keys. ASCII diagram showing the authentication flow: client sends public key ID → server finds matching key → server sends challenge encrypted with public key → client decrypts with private key → authenticated.
   - ### SSH Config File: ~/.ssh/config for defining hosts, users, identity files, ProxyJump. Example config file in CodeBlock showing a bastion + internal server setup.
   - ### SSH Tunneling: Local port forwarding (`ssh -L`): access a remote service through a local port. Remote port forwarding (`ssh -R`): expose a local service to the remote network. ASCII diagram for each. `<Callout type="deep-dive">` for dynamic port forwarding (SOCKS proxy with `ssh -D`).
   - ### Agent Forwarding: `ssh -A` passes your key to the next hop so you can SSH from a bastion to an internal server without copying your private key. `<Callout type="warning">` for "Never use agent forwarding to servers you don't trust — the remote server can use your forwarded key to authenticate as you."
   - ### Host Key Verification: What known_hosts does, why "REMOTE HOST IDENTIFICATION HAS CHANGED" is a security warning (potential MITM attack), when it's safe to ignore (container recreation).

3. **## Hands-On Exercise** — ExerciseCard with scenario: "You're setting up secure access to a new server. You need to generate keys, configure key-based login, and set up an SSH config for convenient access." Difficulty: Intermediate. Tasks: Generate an SSH key pair with ssh-keygen, copy the public key to the sshd server's authorized_keys, connect with key-based auth (no password), create an ~/.ssh/config entry for the server, test the connection using the config name.    VerificationChecklist: key pair generated, key-based login works, config file created with host entry.

title: "Firewalls: Controlling Network Traffic"
description: "How netfilter processes packets, writing iptables rules, using ufw as a frontend, and understanding security group concepts"

```

Section structure:
1. **## Overview** — "Why This Matters" scenario: "Your new service is deployed and running, but external users can't reach it. The application logs show no incoming requests at all. A firewall rule is silently dropping the traffic before it ever reaches your application — you need to know how to find and fix it."

2. **## How It Works** — Deep mechanism:
   - ### Netfilter: The Linux Firewall Framework: Netfilter is the kernel-level packet filtering framework. iptables is the userspace tool to configure it. Packets flow through chains: PREROUTING → INPUT/FORWARD/OUTPUT → POSTROUTING. ASCII diagram showing the packet flow through netfilter chains with decision points.
   - ### iptables Rules: Structure of a rule: table → chain → match criteria → target (ACCEPT, DROP, REJECT, LOG). Example rules in CodeBlock: allow SSH (`iptables -A INPUT -p tcp --dport 22 -j ACCEPT`), drop all other incoming (`iptables -A INPUT -j DROP`), allow established connections (`iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT`). Explain rule ordering matters — first match wins.
   - ### iptables Tables: filter (default — accept/drop), nat (address translation), mangle (packet modification). Focus on filter table for this lesson.
   - ### ufw: The Friendly Frontend: `ufw enable`, `ufw allow 22/tcp`, `ufw deny 80/tcp`, `ufw status verbose`. Explain that ufw generates iptables rules under the hood. `<Callout type="tip">` for "ufw is what you'll use day-to-day; iptables is what you need to understand when debugging why ufw rules aren't working as expected."
   - ### Security Groups (Cloud Context): Briefly explain that cloud security groups (AWS, GCP, Azure) are stateful firewalls applied to network interfaces — same concept as iptables but managed through a cloud API. `<Callout type="tip">` for "When you see 'Security Group' in AWS, think 'stateful iptables rules attached to a network interface.' The mental model is the same."
   - `<Callout type="warning">` for "Always allow SSH (port 22) BEFORE setting a default DROP policy — otherwise you lock yourself out of the server. This is the most common firewall mistake."

3. **## Hands-On Exercise** — ExerciseCard with scenario: "You're hardening a web server that should only accept HTTP (port 80), HTTPS (port 443), and SSH (port 22) traffic. All other incoming traffic must be dropped." Difficulty: Intermediate. Tasks: Inspect current iptables rules (`iptables -L -n`), write rules to allow SSH/HTTP/HTTPS, set default DROP policy for INPUT chain, verify that allowed ports are reachable and others are blocked using nc from a second container, then do the same with ufw (reset iptables first). TerminalBlock for lab launch:
   ```
   docker compose -f docker/networking/06-firewalls/compose.yml up -d
   docker compose -f docker/networking/06-firewalls/compose.yml exec firewall-host bash
   ```
   VerificationChecklist: iptables rules block unauthorized ports, SSH/HTTP/HTTPS allowed, ufw status shows expected rules.

4. **## Quick Reference** — QuickReference: iptables command syntax, common rules (allow SSH, allow HTTP, default DROP), ufw commands, netfilter chain flow.

description: "A systematic approach to diagnosing network issues using ping, traceroute, tcpdump, ss, and curl — with a deliberately broken lab to practice on"

tags: ["troubleshooting", "ping", "traceroute", "tcpdump", "ss", "netstat", "diagnostics"]
```

Section structure:
1. **## Overview** — "Why This Matters" scenario: "It's 2 AM and the monitoring alert says 'service unreachable.' You have 5 minutes to figure out if it's a DNS problem, a routing issue, a firewall rule, or the service itself being down. This lesson gives you the systematic approach to find the answer fast."

2. **## How It Works** — Deep mechanism:
   - ### The Troubleshooting Methodology: Work bottom-up through the TCP/IP stack. Layer 1-2: Can I reach the host? (ping). Layer 3: Is the path correct? (traceroute, ip route). Layer 4: Is the port open? (ss, nc). Layer 7: Is the application responding? (curl). ASCII diagram showing the diagnostic flow chart.
   - ### ping: Tests basic connectivity (ICMP echo). Interpreting output: RTT (round-trip time), TTL, packet loss percentage. What "Destination Host Unreachable" vs "Request timed out" tells you.
   - ### traceroute: Shows the path packets take hop by hop. Each hop is a router. Interpreting output: * means a router didn't respond (may be filtered, not necessarily broken). Identifying where in the path latency increases or packets are lost.
   - ### ss (Socket Statistics): Replaces netstat. `ss -tlnp` shows listening TCP sockets with process info. `ss -s` for summary statistics. How to check if a service is actually listening on the expected port.
   - ### tcpdump: Packet capture and analysis. ALWAYS use filters (per RESEARCH.md pitfall 6): `tcpdump -i eth0 port 80 -A -c 5` (5 packets on port 80 with ASCII payload). `tcpdump -i eth0 host webserver -n` (filter by host). Explain reading tcpdump output: timestamp, source > dest, flags (S for SYN, . for ACK, P for PSH), seq/ack numbers. `<Callout type="warning">` for "Never run bare `tcpdump` without filters on a busy network — it will flood your terminal. Always specify at least a port or host filter."
   - ### curl for Application Debugging: `curl -v` shows full request/response including headers. `curl -w '%{http_code}\n'` for just the status code. `curl --connect-timeout 5` to limit waiting time. Common patterns: curl works but browser doesn't (DNS cache), curl to localhost works but curl to hostname doesn't (DNS), curl returns connection refused (service not listening).
   - ### Systematic Checklist: Present a numbered checklist that combines all tools: 1. ping the target (is it reachable?), 2. traceroute if ping fails (where does it break?), 3. ss -tlnp on the target (is the service listening?), 4. curl the service (does the application respond?), 5. tcpdump if everything looks right but it still doesn't work (what's actually on the wire?).

3. **## Hands-On Exercise** — ExerciseCard with scenario: "A three-service web application (client, app server, database) is broken. Users report they can't access the application. Your job is to systematically diagnose the fault using the tools from this lesson. The lab has an intentional misconfiguration — find it." Difficulty: Intermediate. Tasks: Start the deliberately broken lab, use ping to test connectivity between containers, use ss to check if services are listening on expected ports, use curl to test HTTP responses, use tcpdump to capture traffic and identify the fault, write your diagnosis to /tmp/diagnosis.txt. TerminalBlock for lab launch:
   ```
   docker compose -f docker/networking/07-troubleshooting/compose.yml up -d
   docker compose -f docker/networking/07-troubleshooting/compose.yml exec client bash
   ```
   NOTE: The exercise does NOT tell the learner what the fault is. They must discover it. The verify.sh checks that they wrote a diagnosis file, not that they fixed a specific thing.
   VerificationChecklist: used diagnostic tools, identified the fault, wrote diagnosis to /tmp/diagnosis.txt.

4. **## Quick Reference** — QuickReference: Troubleshooting checklist (bottom-up through TCP/IP stack), ping/traceroute/ss/tcpdump/curl command reference with common flags, diagnostic decision tree.
