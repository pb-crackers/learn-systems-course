**File 1: `docs/design/challenge-content-policy.md`**

Write a content policy document for challenge-mode exercises. Include these exact sections:

1. **Goal-Only Format**:
   - Challenge exercises display ONLY the `challengePrompt` paragraph — the overall goal the learner must achieve
   - The numbered step list (`steps` array) is NOT rendered when mode is `'compose'`
   - The step data still exists in the MDX (for guided fallback) but is hidden from view
   - No procedural language in the challengePrompt: do not say "First do X, then do Y, finally do Z"
   - The prompt describes WHAT to accomplish, not HOW: "Deploy a containerized web application accessible on port 8080" not "Run docker build, then docker run with port mapping"

2. **Reference Sheet Rules**:
   - Each challenge exercise includes a `ChallengeReferenceSheet` (wrapping `QuickReference`) at the bottom
   - Maximum 15 items per reference sheet — this is a hard cap, not a guideline
   - Items are organized by `ReferenceSection` groups (e.g., "Container Management", "Network Commands")
   - NO sequential ordering language: items must not imply an order of operations (no "First", "Then", "After", "Before")
   - Items do NOT name the solution — they list relevant commands the learner might need, including some they will not use (distractors)
   - Each item uses the `ReferenceItem` shape: `{ command: string, description: string, example?: string }`
   - Example good item: `{ command: "docker run", description: "Create and start a new container from an image", example: "docker run -d -p 80:80 nginx" }`
   - Example bad item: `{ command: "Step 1: docker build", description: "First, build the image" }` (sequential, names the solution)

3. **Verification Standards**:
   - Every challenge exercise must have a `VerificationChecklist` with minimum 3 items
   - Each verification item's `hint` field must contain a runnable command and its expected output
   - Example good hint: "Run `docker ps --format '{{.Ports}}'` — should show `0.0.0.0:8080->80/tcp`"
   - Example bad hint: "Check that the container is running" (vague, no command, no expected output)
   - Verification items test observable outcomes, not process steps
   - At least one verification item should test a negative case or edge condition

**File 2: `docs/design/foundation-safety-net.md`**

Write the Foundation safety-net specification. Include these exact sections:

1. **Rule**: Foundation exercises ALWAYS render in `'guided'` mode. This is a hard override at the ExerciseCard level, not a preference default. No UI toggle, no mode prop, no learner preference can change a Foundation exercise away from guided mode.

2. **Rationale**: Foundation learners are encountering CLI commands for the first time. They need:
   - Full command text visible and copyable
   - Per-flag annotations explaining every token
   - Numbered step-by-step instructions
   - Removing any of these creates a knowledge gap at the foundational level where learners have no prior experience to fill it

3. **Implementation Contract**: In ExerciseCard mode resolution, when `difficulty === 'Foundation'`, the resolved mode is ALWAYS `'guided'` — skip the mode prop check, skip the preference check, return `'guided'` immediately.

4. **Toggle Behavior**: The DifficultyToggle component is NOT rendered on Foundation lessons. Even if a learner navigates to a Foundation lesson after setting a preference on a Challenge lesson, the Foundation lesson ignores that preference entirely.

5. **Intermediate Behavior**: Intermediate exercises use `'recall'` mode by default. Unlike Foundation, Intermediate exercises DO respect the learner's preferredMode if set (a learner on a Challenge lesson can toggle to guided, and that preference carries to Intermediate lessons too). However, the DifficultyToggle is NOT rendered on Intermediate lessons — the toggle only appears on Challenge lessons.
