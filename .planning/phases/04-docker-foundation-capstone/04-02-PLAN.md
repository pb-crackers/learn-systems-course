       - ## Overview: "Your database container restarted and all the data is gone" scenario. Data persistence is the first production concern after learning to run containers.
       - ## How It Works:
         a) **Container Filesystem Ephemerality**: When a container is removed, its writable layer is gone. Demo with `docker run`, write a file, `docker rm`, run again — file is gone. Connect to overlay FS from DOC-01.
         b) **Bind Mounts**: Mount a host directory into the container. `-v /host/path:/container/path`. Use case: development (edit code on host, see changes in container). Callout type="warning" about bind mount security — container has full access to host path.
         c) **Named Volumes**: `docker volume create`, `docker volume ls`, `docker volume inspect`. `-v mydata:/app/data`. Docker manages the storage location. Use case: production data persistence.
         d) **Bind Mount vs Named Volume**: Comparison table (portability, backup, performance, use case). Named volumes for production data, bind mounts for development workflows.
         e) **tmpfs Mounts**: Brief mention for sensitive data that should never hit disk.
       - ## Hands-On Exercise: ExerciseCard difficulty="Intermediate", scenario about persisting application state across container restarts:
         Steps: (1) Run progressive app without volume, write data via curl, remove and restart — data gone, (2) Create named volume and mount to /app/data, write data, remove and restart — data persists, (3) Use bind mount for development: mount local docker/app/ directory, edit app.js on host, see changes reflected, (4) Inspect volume with docker volume inspect, (5) Clean up volumes
         VerificationChecklist: data lost without volume, data persists with named volume, bind mount reflects host changes, volume visible in docker volume ls

         b) **Default Bridge vs Custom Bridge**: Default bridge uses IP addresses for communication. Custom bridge provides DNS resolution between containers by name. This is the critical distinction. ASCII diagram showing two containers on a custom bridge with DNS arrows.
         c) **Container DNS Resolution**: Docker's embedded DNS server at 127.0.0.11. Container names become hostnames. Show `docker exec app ping api` resolving by name.          d) **Port Publishing**: `-p host:container` mapping. Explain that EXPOSE alone does nothing — `-p` is required. Show `-p 8080:3000` mapping. Callout type="tip" about `-P` for random port assignment.
         e) **Network Isolation**: Containers on different networks cannot communicate. Demo creating two networks, placing containers on different ones, showing ping fails.
       - ## Hands-On Exercise: ExerciseCard difficulty="Intermediate", scenario about connecting frontend and API services:
         Steps: (1) Create custom network: `docker network create mynet`, (2) Run progressive app on the network: `docker run -d --name api --network mynet myapp:v1`, (3) Run a second container (nginx or alpine) on same network, curl the api by name, (4) Demo that default bridge doesn't support name resolution, (5) Inspect network with `docker network inspect mynet`, (6) Show container-to-container DNS working, (7) Clean up
         VerificationChecklist: custom network created, containers resolve each other by name, default bridge fails name resolution, network inspect shows both containers

       - Frontmatter: moduleSlug: "03-docker", lessonSlug: "06-docker-compose", order: 6, difficulty: "Intermediate", estimatedMinutes: 30, prerequisites: ["03-docker/04-docker-volumes", "03-docker/05-docker-networking"], module: "Docker & Containerization"
       - ## Overview: "Running 5 docker run commands with the right flags every time is a recipe for errors" scenario. Compose captures the entire multi-service configuration declaratively.
       - ## How It Works:
         a) **What is Docker Compose**: Declarative YAML for multi-container apps. `compose.yml` (NOT docker-compose.yml — Compose v2 preference). `docker compose up`, `docker compose down`, `docker compose ps`.
         b) **Service Definition**: services block with image/build, ports, environment/env_file, volumes, networks. Walk through building the progressive app as a compose service.
         c) **depends_on and Healthchecks**: `depends_on` only controls start ORDER, not readiness. `depends_on: { api: { condition: service_healthy } }` with HEALTHCHECK. Show the difference — container started vs container healthy.
         d) **Environment Variables**: `environment:` inline, `env_file:`, variable substitution `${VAR:-default}`. Callout type="warning" about .env file NOT being the same as env_file.
         e) **Named Volumes in Compose**: `volumes:` top-level key declaring named volumes. Per-service `volumes:` mounting them. Data persistence across `docker compose down` (volumes survive) vs `docker compose down -v` (volumes removed).
         f) **Profiles**: `profiles: ["debug"]` for optional services. `docker compose --profile debug up`.
       - ## Hands-On Exercise: ExerciseCard difficulty="Intermediate", scenario about defining a multi-service stack:
         Steps: (1) Create a compose.yml with the progressive app as the api service, (2) Add an nginx service that reverse-proxies to the api, (3) Add a healthcheck to the api service, (4) Use depends_on with condition: service_healthy, (5) Add a named volume for persistent data, (6) Test with docker compose up -d, verify with docker compose ps, (7) Test persistence: docker compose down then docker compose up — data survives, (8) Clean up with docker compose down -v
         VerificationChecklist: compose.yml has 2+ services, healthcheck defined, depends_on with condition works, named volume persists data, docker compose ps shows all services healthy
