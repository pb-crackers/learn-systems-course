**01-iac-concepts.mdx (IAC-01)**
Frontmatter: title "Infrastructure as Code: Why Declaring Infrastructure Beats Scripting It", module "Infrastructure as Code", moduleSlug "06-iac", lessonSlug "01-iac-concepts", order 1, difficulty "Foundation", estimatedMinutes 25, prerequisites ["05-cicd/01-cicd-concepts"], tags ["iac", "infrastructure-as-code", "declarative", "state", "drift", "opentofu", "terraform"].

Overview: "Why This Matters" scenario — your team manually configured 12 servers via SSH and shell scripts. Three months later nobody remembers which server has which config. One goes down, rebuilding takes a full day of guesswork. IaC solves this.

How It Works section: Explain imperative vs declarative — "Remember manually creating Docker containers with docker run? That is imperative. HCL declares what you want and OpenTofu figures out how to get there." Explain the IaC workflow: write HCL -> tofu plan (preview) -> tofu apply (execute) -> state file records what was created. Explain state management: terraform.tfstate is a JSON file that maps your HCL resources to real-world infrastructure IDs. Explain drift detection: if someone manually changes infrastructure outside IaC, tofu plan detects the difference. Explain the plan/apply cycle as a safety net — you always see what will happen before it happens. Brief history: Terraform (HashiCorp, 2014) -> BSL license change (2023) -> OpenTofu fork (MPL-2.0, community-maintained). Note: "In job postings you will see 'Terraform' — OpenTofu is a 1:1 compatible open-source fork. Everything you learn here works with both tools."

Callout type="deep-dive" bridging to CI/CD: "In the CI/CD module, you learned that pipelines automate build/test/deploy. IaC is the 'deploy' piece done right — instead of a pipeline running ssh commands, it runs tofu apply against a state file."

Callout type="tip": "OpenTofu uses the tofu CLI command. Terraform uses terraform. The syntax, files, and workflow are identical — only the binary name differs."

Exercise: ExerciseCard difficulty="Foundation", scenario "Your team has been creating Docker containers manually. Compare the imperative approach (docker run commands in a shell script) with the declarative approach (HCL resource definitions)." Steps: examine a shell script that creates 3 containers, then read the equivalent HCL, identify what happens when you need to change a port number in each approach. VerificationChecklist: can explain why declarative is safer for changes, can describe what the state file tracks, can explain what drift means.

**03-terraform-state.mdx (IAC-03)**
Frontmatter: title "Terraform State: How OpenTofu Tracks Your Infrastructure", module "Infrastructure as Code", moduleSlug "06-iac", lessonSlug "03-terraform-state", order 3, difficulty "Intermediate", estimatedMinutes 30, prerequisites ["06-iac/02-hcl-basics"], tags ["state", "terraform-state", "tfstate", "remote-backends", "state-locking", "drift", "tofu-state"].

How It Works section: Explain what the state file IS — a JSON document that maps each HCL resource block to a real-world infrastructure object ID. Show a simplified terraform.tfstate excerpt with the resources array, provider reference, and attribute values. Explain state as the "single source of truth" — without state, OpenTofu cannot know what it previously created. Explain the refresh cycle: on every plan/apply, OpenTofu reads state, queries actual infrastructure, computes the diff. Explain drift detection in detail: manually stop a container created by OpenTofu, then run tofu plan — the plan output shows "must be replaced" because the actual state differs from the desired state. Explain tofu state subcommands: list (show all resources), show (inspect one resource), rm (remove from state without destroying), mv (rename in state). Explain remote backends concept: why local state files are dangerous for teams (no locking, no shared access, lost if laptop dies). Describe S3+DynamoDB backend pattern (AWS), GCS backend (GCP), and azurerm backend (Azure) at conceptual level — the learner understands WHY remote backends exist without needing a cloud account. Explain state locking: prevents two people running tofu apply simultaneously, which would corrupt state.

Callout type="deep-dive": "Open terraform.tfstate in your editor after running tofu apply. You will see a JSON object with a resources array. Each entry has a type (docker_container), a name (web), and an instances array with the actual Docker container ID. This is the bridge between your HCL code and reality."

Callout type="warning": "Never manually edit terraform.tfstate. Use tofu state commands to make changes. A corrupted state file means OpenTofu loses track of your infrastructure — it might try to create duplicates or fail to destroy resources."

Callout type="tip" about .gitignore: "Add terraform.tfstate and terraform.tfstate.backup to .gitignore. State files can contain sensitive data (resource IDs, IP addresses). In production, use a remote backend instead of committing state to git."

Exercise: ExerciseCard difficulty="Intermediate", scenario "Your colleague manually stopped a container that OpenTofu manages. Detect the drift, understand what the state file says, and restore the desired state." Steps: navigate to docker/iac/02-state/, run tofu init and tofu apply, inspect terraform.tfstate (cat or jq), manually stop the container with docker stop, run tofu plan to see drift detected, run tofu apply to restore, use tofu state list and tofu state show to inspect state entries, run tofu destroy to clean up. VerificationChecklist: can read terraform.tfstate JSON structure, tofu plan detects manually stopped container, tofu apply restores the container, tofu state list shows all managed resources.

**04-modules.mdx (IAC-04)**
Frontmatter: title "Terraform Modules: Reusable Infrastructure Components", module "Infrastructure as Code", moduleSlug "06-iac", lessonSlug "04-modules", order 4, difficulty "Intermediate", estimatedMinutes 25, prerequisites ["06-iac/03-terraform-state"], tags ["modules", "reusability", "composition", "registry", "root-module", "child-module", "opentofu"].

How It Works section: Explain root module vs child module — every OpenTofu working directory is a root module; a child module is a directory of .tf files called via the module block. Explain the module block syntax: module "name" { source = "./path", plus variable assignments }. Explain how modules enable reusability: define a web-container module once, call it multiple times with different parameters to create frontend, api, staging, production containers. Explain module inputs (variables.tf in child module) and outputs (outputs.tf in child module accessed via module.name.output_name). Explain module composition: calling multiple modules from a root module to build a complete infrastructure. Explain the OpenTofu registry: public modules with source = "registry.opentofu.org/..." and version constraints. Note that most production codebases use private modules from git repos (source = "git::https://github.com/...").

Callout type="deep-dive": "Modules are to OpenTofu what functions are to programming — they encapsulate complexity behind a clean interface. The module's variables.tf is the function signature, main.tf is the implementation, and outputs.tf is the return value."

Callout type="tip": "Start with local modules (source = \"./modules/name\") before using registry modules. Local modules let you see exactly what is inside, which is better for learning. Registry modules are convenient but hide the implementation."
