**Lesson 6 — 06-shell-fundamentals.mdx (LNX-06):**
Frontmatter: title "Shell Fundamentals", description "Bash, environment variables, PATH, pipes, redirects — your primary interface to Linux", module "Linux Fundamentals", moduleSlug "01-linux-fundamentals", lessonSlug "06-shell-fundamentals", order 6, difficulty "Intermediate", estimatedMinutes 22, prerequisites ["01-linux-fundamentals/05-processes"], tags ["bash", "shell", "environment", "PATH", "pipes", "redirects"].

Content (15-25 min reading target):
- **Overview**: "Why This Matters" — "A CI/CD pipeline failed because an environment variable wasn't set in the deployment container. Understanding how the shell manages your environment, finds commands, and connects processes through pipes is foundational to every DevOps task you'll ever do."
- **How It Works**:
  - What is a shell: program that reads commands and calls fork/exec (tie back to lesson 5). bash vs sh vs zsh — why bash is the standard for scripting
  - Shell startup: login shell vs non-login shell, /etc/profile, ~/.bashrc, ~/.bash_profile — which file runs when and why it matters
  - Environment variables: export vs local variables, env command, how child processes inherit environment. KEY variables: HOME, USER, SHELL, LANG
  - PATH: ordered list of directories, how the shell finds commands (`which`, `type`), why `/usr/local/bin` comes before `/usr/bin`, how to add to PATH. Use `<Callout type="deep-dive">` for PATH lookup algorithm
  - Command types: built-in (cd, echo, export) vs external (/usr/bin/ls), why `cd` must be a built-in (it changes the shell's own working directory)
  - Pipes: `|` connects stdout of left to stdin of right. Pipeline as a chain of processes running concurrently (not sequentially!). Use `<Callout type="deep-dive">` for pipe buffer and blocking behavior
  - Redirects: `>` (overwrite stdout to file), `>>` (append), `<` (stdin from file), `2>` (stderr), `2>&1` (merge stderr into stdout), `/dev/null` (discard). Use `<Callout type="warning">` about `>` truncating files before the command runs
  - Command substitution: `$(command)` vs backticks, when to use each
  - Quoting: single quotes (literal), double quotes (variable expansion), no quotes (word splitting + globbing). Use `<Callout type="tip">` for "when in doubt, double-quote your variables"
  - Exit codes: 0 = success, non-zero = failure, `$?`, `&&` (run if previous succeeded), `||` (run if previous failed)
- **Hands-On Exercise**: ExerciseCard with difficulty "Intermediate", scenario about debugging a deployment where commands aren't found because PATH is misconfigured. Steps: examine environment with env/printenv, manipulate PATH, create a pipe chain (ps aux | grep | awk), redirect command output to files, use exit codes with && and ||. Docker lab via TerminalBlock. VerificationChecklist: "You can display the current PATH", "You created a 3-stage pipeline", "You redirected both stdout and stderr to a file".
- **Quick Reference**: echo, export, env, which, type, pipe operators, redirect operators, $?, &&, ||

**Lesson 7 — 07-shell-scripting.mdx (LNX-07):**
Frontmatter: title "Shell Scripting", description "Variables, conditionals, loops, functions, error handling — automating tasks with bash", module "Linux Fundamentals", moduleSlug "01-linux-fundamentals", lessonSlug "07-shell-scripting", order 7, difficulty "Intermediate", estimatedMinutes 25, prerequisites ["01-linux-fundamentals/06-shell-fundamentals"], tags ["bash", "scripting", "automation", "functions", "error-handling"].

Content (15-25 min reading target):
- **Overview**: "Why This Matters" — "You need to provision 50 identical servers. Doing it manually means 50 chances for human error. A well-written shell script does it identically every time, handles errors gracefully, and documents exactly what happened."
- **How It Works**:
  - Script basics: shebang line (`#!/usr/bin/env bash`), making scripts executable, why `#!/usr/bin/env bash` is better than `#!/bin/bash` (portability)
  - Variables: assignment (no spaces around =), referencing ($VAR vs ${VAR}), readonly, local (inside functions). Use `<Callout type="warning">` about spaces around = being a syntax error
  - Conditionals: if/elif/else/fi, `[` vs `[[` (test command vs bash keyword), `-f` (file exists), `-d` (directory), `-z` (empty string), `-n` (non-empty), string comparison, numeric comparison (-eq, -lt, -gt)
  - Loops: for (iterating over lists, C-style), while (reading lines, condition loops), until. `break` and `continue`. Use `<Callout type="tip">` for `while IFS= read -r line` pattern for safe line-by-line file reading
  - Functions: declaration, local variables, return values (exit codes, not strings — use stdout for string output), calling functions
  - Error handling: `set -euo pipefail` — what each flag does: `-e` (exit on error), `-u` (unset variables are errors), `-o pipefail` (pipe failure = overall failure). `trap` for cleanup. Use `<Callout type="deep-dive">` for trap ERR pattern with stack trace
  - Script arguments: $1, $2, $#, $@, "$@" (quoted!), shift. Parsing with getopts or manual case statement
  - Debugging: `set -x` (trace), `bash -n script.sh` (syntax check without running)
  - Common patterns: logging function, usage/help function, argument validation, temp file cleanup with trap
- **Hands-On Exercise**: ExerciseCard with difficulty "Intermediate", scenario about writing a deployment script that backs up a config, validates inputs, and reports success/failure. Steps: write a script with shebang, add argument parsing, add conditionals for input validation, add a function for backup, add set -euo pipefail and trap for cleanup, test the script. Docker lab. VerificationChecklist: "Your script starts with #!/usr/bin/env bash", "Your script uses set -euo pipefail", "Your script accepts at least one argument and validates it", "Your script exits 0 on success and non-zero on failure".
- **Quick Reference**: shebang, set flags, test operators, loop syntax, function syntax, trap, $? $# $@ special variables

**Lesson 8 — 08-text-processing.mdx (LNX-08):**
Frontmatter: title "Text Processing", description "grep, sed, awk, sort, uniq, cut, xargs — transforming data on the command line", module "Linux Fundamentals", moduleSlug "01-linux-fundamentals", lessonSlug "08-text-processing", order 8, difficulty "Intermediate", estimatedMinutes 25, prerequisites ["01-linux-fundamentals/07-shell-scripting"], tags ["grep", "sed", "awk", "sort", "uniq", "cut", "xargs", "regex", "text"].

Content (15-25 min reading target):
- **Overview**: "Why This Matters" — "A web server is returning intermittent 500 errors. The access log has 2 million lines. You need to find all 500-status requests in the last hour, extract the URLs, count which endpoints are failing most, and get an answer in seconds — not hours of scrolling."
- **How It Works**:
  - The Unix philosophy applied: small tools that do one thing well, connected by pipes. Text as the universal interface.
  - grep: pattern matching in files/streams. Basic patterns, -i (case insensitive), -r (recursive), -n (line numbers), -c (count), -v (invert), -E (extended regex). Regular expression basics: `.` (any char), `*` (zero or more), `+` (one or more), `^` (start), `$` (end), `[abc]` (character class), `\d` equivalent. Use `<Callout type="deep-dive">` for regex engine behavior (greedy vs non-greedy)
  - sed: stream editor. `sed 's/old/new/'` (substitute first), `sed 's/old/new/g'` (global), `sed -i` (in-place edit — **dangerous**, use with caution), `sed -n 'Xp'` (print specific line), address ranges (`10,20s/...`). Use `<Callout type="warning">` about sed -i with no backup destroying the original file
  - awk: pattern-action language. `awk '{print $1}'` (first field), field separator (`-F:`), pattern matching (`/pattern/{action}`), built-in variables (NR = line number, NF = number of fields, $0 = entire line). Arithmetic and string operations. Use `<Callout type="deep-dive">` for awk as a mini programming language (BEGIN/END blocks, associative arrays)
  - sort: alphabetic vs numeric (-n), reverse (-r), field-based (-k), unique (-u). Stable sort.
  - uniq: only removes ADJACENT duplicates (must sort first!). -c (count), -d (only duplicates), -u (only unique). Use `<Callout type="warning">` about the "must sort first" gotcha
  - cut: extract columns by delimiter (-d) and field (-f). `cut -d: -f1 /etc/passwd` to get usernames.
  - xargs: convert stdin to arguments. `find . -name "*.log" | xargs rm`. -I {} for placeholder, -P for parallel execution. Use `<Callout type="tip">` for `xargs -I {} sh -c 'echo processing {}'` pattern
  - Pipeline composition: building multi-stage pipelines. Example: `grep "ERROR" app.log | awk '{print $5}' | sort | uniq -c | sort -rn | head -10` — find top 10 most common error messages. Walk through each stage.
- **Hands-On Exercise**: ExerciseCard with difficulty "Intermediate", scenario about analyzing a web server access log to find the most common error URLs and client IPs. Steps: use grep to filter status codes, use awk to extract fields, use sort | uniq -c | sort -rn for frequency analysis, use sed to transform a config file, use cut on /etc/passwd, build a multi-stage pipeline. Docker lab with a pre-populated sample access log in the setup script. VerificationChecklist: "You extracted the top 5 error URLs from the log", "You used sed to replace a value in a config file", "You built a pipeline with 4+ stages".
- **Quick Reference**: grep flags, sed substitution syntax, awk field access, sort flags, uniq flags, cut syntax, xargs patterns

**Lesson 9 — 09-package-management.mdx (LNX-09):**
Frontmatter: title "Package Management", description "apt, repositories, dependencies — installing and managing software on Linux", module "Linux Fundamentals", moduleSlug "01-linux-fundamentals", lessonSlug "09-package-management", order 9, difficulty "Intermediate", estimatedMinutes 20, prerequisites ["01-linux-fundamentals/06-shell-fundamentals"], tags ["apt", "dpkg", "packages", "repositories", "dependencies"].

Note: LNX-09 depends on shell fundamentals (LNX-06), NOT scripting — per RESEARCH.md. The learner needs to run commands, not write scripts, to understand package management.

Content (15-25 min reading target):
- **Overview**: "Why This Matters" — "A Docker build failed because apt-get install couldn't find a package. Understanding how repositories, package versions, and dependencies work tells you whether to add a repository, pin a version, or find an alternative package."
- **How It Works**:
  - What is a package: archive containing binaries, libraries, config files, and metadata (dependencies, version, maintainer). .deb format for Debian/Ubuntu.
  - Package managers: two layers — dpkg (low-level, installs single .deb files, no dependency resolution) and apt (high-level, resolves dependencies, fetches from repositories). Analogy: dpkg is like manually installing one file; apt is like an app store that handles everything.
  - Repositories: /etc/apt/sources.list and /etc/apt/sources.list.d/. Format: `deb http://archive.ubuntu.com/ubuntu jammy main restricted`. Components: deb (binary) vs deb-src (source), URL, codename (jammy), sections (main, restricted, universe, multiverse — what each contains and the licensing difference)
  - apt-get update: fetches the package INDEX (metadata), not the packages themselves. Use `<Callout type="warning">` about always running apt-get update before apt-get install in Dockerfiles (stale index = package not found)
  - apt-get install: dependency resolution algorithm (simplified), how it decides which version to install, -y flag for non-interactive
  - apt-cache: search for packages (apt-cache search), show package details (apt-cache show), show dependencies (apt-cache depends)
  - dpkg: list installed packages (dpkg -l), query a package (dpkg -s), list files in a package (dpkg -L), install a .deb directly (dpkg -i)
  - Version pinning: apt-get install package=version, /etc/apt/preferences for priority pinning. Why this matters in production (reproducible builds).
  - apt-get remove vs purge: remove keeps config files, purge deletes everything. autoremove for orphaned dependencies.
  - Use `<Callout type="deep-dive">` for how the dependency resolver works (satisfying constraints)
  - Use `<Callout type="tip">` for `apt-get install --no-install-recommends` to keep Docker images small
  - Forward reference: "In the Docker module (Module 3), you'll see why `apt-get update && apt-get install -y` must be in the same RUN command — Docker layer caching makes a separate update step stale."
- **Hands-On Exercise**: ExerciseCard with difficulty "Intermediate", scenario about setting up a development server where you need to install specific package versions and explore what's available. Steps: run apt-get update, search for packages, install a package and examine what files it installed (dpkg -L), check dependencies of a package, try installing without update first (see the error), explore sources.list. Docker lab. VerificationChecklist: "You installed tree package using apt-get", "You can list files installed by the tree package using dpkg -L", "You can show the dependencies of the curl package".
- **Quick Reference**: apt-get (update, install, remove, purge, autoremove), apt-cache (search, show, depends), dpkg (-l, -s, -L, -i), /etc/apt/sources.list
