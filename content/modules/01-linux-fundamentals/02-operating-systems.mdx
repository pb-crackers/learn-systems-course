## Overview

A container crashed with "permission denied" while calling `openat` — a system call to open a file. Was this a bug in the application code? A missing capability in the container security profile? A Linux security module blocking the call? You cannot diagnose this without understanding the boundary between user code and the kernel, and how that boundary is crossed via system calls.

This lesson explains what the Linux kernel does, why the userspace/kernel split exists, how system calls bridge that gap, and how the kernel exposes its internal state through `/proc` and `/sys`.

## How It Works

### What the Kernel Does

The **kernel** is the core of the operating system. It is the first program loaded after the bootloader and it runs for the entire lifetime of the system. It is responsible for:

**Hardware abstraction**: The kernel provides a uniform interface to hardware. Your application doesn't need to know whether storage is on an NVMe SSD, a SATA HDD, or a network-attached filesystem — it just calls `read()` and `write()`. The kernel translates those calls into the appropriate device-specific commands.

**Process management**: The kernel creates, schedules, and terminates processes. It decides which process runs on which CPU core for how long (the scheduler). It enforces isolation between processes — one process cannot read another's memory without explicit sharing.

**Memory management**: The kernel manages virtual memory. It maps each process's virtual address space to physical RAM pages, handles page faults (when a process accesses memory not yet in RAM), and manages the swap area.

**Filesystem access**: The kernel presents all storage — local disks, NFS mounts, tmpfs, procfs — through a unified filesystem interface (the VFS, covered in the next lesson).

**Device drivers**: Code inside the kernel (or loaded as modules) that speaks the protocol of specific hardware devices — a specific network card, a specific GPU, a specific USB controller.

### Kernel Space vs User Space

This is the most important architectural division in Unix-like systems.

**Ring 0 (kernel space)**: Code running in ring 0 has unrestricted access to all hardware, all memory, all I/O ports. The kernel runs here. A bug in kernel-space code can crash the entire system.

**Ring 3 (user space)**: All user programs — your shell, your web server, your container processes — run in ring 3. Ring 3 code cannot directly access hardware, cannot access other processes' memory, and cannot execute privileged instructions. This is enforced by the CPU hardware, not just by convention.

**Why this separation matters**: A buggy user-space program can only corrupt its own memory and crash itself — it cannot corrupt the kernel or other processes. This isolation is fundamental to system stability and security.

<Callout type="tip" title="What 'Segmentation Fault' Means">
A segmentation fault occurs when a program in user space tries to access a memory address it doesn't have permission to access. The CPU raises a hardware exception, the kernel catches it and sends `SIGSEGV` to the process, the process terminates. This is the protection model working correctly — preventing one bad program from corrupting system memory.
</Callout>

### System Calls: Crossing the Boundary

User-space code cannot directly access hardware. To do anything useful — read a file, send a network packet, create a new process — it must ask the kernel. This request is a **system call**.

The system call interface is the contract between user programs and the kernel. It is stable across kernel versions: programs compiled years ago continue to work because the system call ABI has not changed.

**Common system calls:**

| Syscall | What it does |
|---------|-------------|
| `open(path, flags)` | Open a file, returns a file descriptor |
| `read(fd, buf, count)` | Read bytes from a file descriptor |
| `write(fd, buf, count)` | Write bytes to a file descriptor |
| `fork()` | Create a copy of the current process |
| `exec(path, argv)` | Replace the current process image with a new program |
| `exit(status)` | Terminate the current process |
| `mmap(addr, len, ...)` | Map a file or memory region into the process's address space |
| `socket(domain, type, ...)` | Create a network socket |
| `clone(flags, ...)` | Create a new thread or process with fine-grained control (used by containers) |

**How a system call works mechanically:**

1. User code calls a C library function (e.g., `fopen()` from libc)
2. libc sets up the syscall number and arguments in registers
3. libc executes the `syscall` instruction (on x86-64)
4. The CPU switches from ring 3 to ring 0 — a privilege level transition
5. The kernel handles the request, performs the operation
6. The CPU switches back to ring 3, with the return value in a register
7. libc returns the result to the application

<Callout type="deep-dive" title="Tracing System Calls with strace">
`strace` is a Linux tool that intercepts all system calls made by a process and prints them to stderr. It's invaluable for debugging.

Example: watching what `ls /tmp` actually does at the syscall level:

```bash
strace ls /tmp 2>&1 | head -30
```

You'll see calls like `execve`, `openat`, `getdents64` (reading directory entries), and `write` (writing the output). Every file operation, every network connection, every process creation — all visible through strace.

The syscall table for x86-64 Linux has ~350 entries, documented in `man 2 <syscall-name>`. Each syscall number corresponds to a kernel function. You can find the full table in the kernel source at `arch/x86/entry/syscalls/syscall_64.tbl`.
</Callout>

<Callout type="warning" title="strace Has Overhead — Never Use on Production Processes">
`strace` works by attaching to a process via `ptrace()` and intercepting every system call. This causes a context switch overhead on every syscall, which can slow a process by 10-100x. Never attach strace to a production service without understanding this impact. Use it in staging, or briefly on a single process instance in production if you absolutely must.
</Callout>

### The Linux Boot Process

Understanding the boot sequence tells you where things can go wrong and where to look for problems:

1. **BIOS/UEFI**: The firmware runs POST (Power-On Self Test), finds bootable devices
2. **Bootloader (GRUB)**: Loaded from the boot sector; loads the kernel image into RAM, passes kernel parameters
3. **Kernel initialization**: The kernel decompresses itself, initializes hardware, mounts the root filesystem
4. **PID 1 (init/systemd)**: The first user-space process. On modern Linux, this is `systemd`. PID 1 is special: it cannot be killed, it adopts orphaned processes, and it is responsible for starting all other services
5. **User-space services**: systemd starts network services, login daemons, and all configured units

<Callout type="tip" title="PID 1 in Containers">
In a Docker container, the first process started (from `CMD` or `ENTRYPOINT`) becomes PID 1. This matters because PID 1 is responsible for reaping zombie processes. If your container's entrypoint is a shell script and it doesn't handle SIGTERM correctly, your container may take 30 seconds to stop (the Docker timeout before SIGKILL). This is why `tini` or proper signal handling matters in containers.
</Callout>

### /proc and /sys: Windows Into the Kernel

The `/proc` filesystem is a **virtual filesystem** — it has no data on disk. The kernel generates its contents on-the-fly when you read from it. It is the primary interface for inspecting kernel state.

**Key /proc paths:**

| Path | What it contains |
|------|-----------------|
| `/proc/cpuinfo` | CPU details (as seen in Lesson 1) |
| `/proc/meminfo` | Memory statistics |
| `/proc/loadavg` | 1, 5, 15-minute load averages + runnable/total processes |
| `/proc/uptime` | System uptime in seconds |
| `/proc/version` | Kernel version string |
| `/proc/mounts` | Currently mounted filesystems |
| `/proc/[pid]/` | Per-process directory (covered in the Processes lesson) |
| `/proc/net/` | Network interface and socket statistics |

**`/sys`** is similar to `/proc` but organized around hardware devices and kernel subsystems. It follows a strict hierarchy: `/sys/bus/`, `/sys/class/`, `/sys/devices/`. You can read hardware parameters and, as root, write to certain entries to change kernel settings at runtime.
