## Overview

A deploy failed because the application couldn't write to its log directory. The fix was one `chmod` command, but understanding WHY it failed requires knowing how Linux decides who can do what to every file. The permission check happens in the kernel — it's not optional and it's not configurable by user-space code.

This lesson explains the Linux permission model from first principles: how the kernel identifies processes by UID (not username), what each permission bit means for files versus directories, how to read and set permissions in both symbolic and numeric formats, and what setuid/setgid/sticky bits actually do at the kernel level.

## How It Works

### Users and Groups: UIDs and GIDs

The Linux kernel identifies users and groups by numbers, not names. Every process has:
- A **UID** (User ID): the numeric identity of the user running the process
- A **GID** (Group ID): the primary group of that user
- A set of **supplementary group IDs**: additional groups the user belongs to

When you log in as `alice`, the login process looks up `alice` in `/etc/passwd`, reads her UID (say, 1001), and sets the UID of all processes you start to 1001. The username `alice` is a human-readable label. The kernel only ever compares numbers.

**`/etc/passwd`** stores user accounts:
```
root:x:0:0:root:/root:/bin/bash
alice:x:1001:1001:Alice Smith:/home/alice:/bin/bash
```
Fields: username:password-placeholder:UID:GID:comment:home:shell

**`/etc/group`** stores group membership:
```
sudo:x:27:alice,bob
developers:x:1002:alice,carol
```
Fields: groupname:password:GID:member-list

### The Permission Model

Every file and directory in Linux has three permission sets:
- **Owner** (u): the user who owns the file
- **Group** (g): the group that owns the file
- **Other** (o): everyone else

Each permission set has three bits:
- **Read (r = 4)**: Can read the file's contents
- **Write (w = 2)**: Can modify the file's contents
- **Execute (x = 1)**: Can run the file as a program

The kernel evaluates permissions in order: owner first, then group, then other. If the process's UID matches the file's owner UID, the owner permissions apply — group and other are ignored, even if they would grant more access.

**Permissions mean different things for files vs directories:**

| Permission | On a file | On a directory |
|------------|-----------|----------------|
| **read (r)** | Can read file contents | Can list directory contents (`ls`) |
| **write (w)** | Can modify file contents | Can create, delete, rename files inside |
| **execute (x)** | Can run as a program | Can `cd` into the directory and access files by name |

<Callout type="tip" title="Missing Execute on a Directory">
The most common permission error: "Permission denied" when accessing a file inside a directory, even though you have read permission on the file itself. The issue is that `x` on the directory is required to traverse it. Without directory execute, you cannot access anything inside — not even if you know the exact filename. Run `ls -la` on the parent directory to check its `x` bit.
</Callout>

### Reading Permission Strings

The `ls -la` output shows permissions as a 10-character string:

```
-rwxr-xr--
^   ^   ^
|   |   other permissions (r--)
|   group permissions (r-x)
owner permissions (rwx)

^ first character: file type
  - = regular file
  d = directory
  l = symbolic link
  b = block device
  c = character device
```

**Examples:**

| String | Owner | Group | Other | Meaning |
|--------|-------|-------|-------|---------|
| `-rw-r--r--` | rw- | r-- | r-- | Config file: owner edits, everyone reads |
| `-rwxr-xr-x` | rwx | r-x | r-x | Program: everyone can run it |
| `-rw-------` | rw- | --- | --- | Private file: owner only |
| `drwxr-xr-x` | rwx | r-x | r-x | Typical directory |
| `drwx------` | rwx | --- | --- | Private directory: owner only |

**Numeric (octal) representation**: Each permission set is represented as a sum of its bit values (r=4, w=2, x=1):

| Octal | Binary | Permission |
|-------|--------|-----------|
| 7 | 111 | rwx |
| 6 | 110 | rw- |
| 5 | 101 | r-x |
| 4 | 100 | r-- |
| 0 | 000 | --- |

So `chmod 755` means: owner=7(rwx), group=5(r-x), other=5(r-x).

### chmod: Changing Permissions

**Symbolic mode** — human-readable, modifies specific bits:
```bash
chmod u+x script.sh      # add execute for owner
chmod g-w file.txt       # remove write for group
chmod o=r file.txt       # set other to read-only (overwrite)
chmod a+r file.txt       # add read for all (a = ugo)
chmod u+x,g-w file.txt   # multiple changes in one command
```

**Numeric mode** — precise, sets all three sets at once:
```bash
chmod 755 script.sh    # rwxr-xr-x — typical executable
chmod 644 config.txt   # rw-r--r-- — typical config file
chmod 600 secret.key   # rw------- — private key file
chmod 777 shared/      # rwxrwxrwx — world-writable (avoid)
```

<Callout type="warning" title="chmod 777 Is a Security Antipattern">
Setting permissions to 777 (world-writable) is almost never correct. It means any process running as any user on the system can modify the file. For a web application's upload directory, this means any compromised process (a hacked PHP script, a container escape) can overwrite any file. The minimum permissions that solve a problem are always the right choice. Start with 644 or 755 and add only what is needed.
</Callout>

### chown and chgrp

`chown` changes the owner and/or group of a file:

```bash
chown alice file.txt             # change owner to alice
chown alice:developers file.txt  # change owner and group
chown :developers file.txt       # change group only (colon prefix)
chown -R alice:devs /app/        # recursive — all files in directory
```

`chgrp` changes only the group:
```bash
chgrp developers project/
chgrp -R www-data /var/www/html/
```

**Why only root can chown**: If unprivileged users could change file ownership, a user could give a setuid binary to root, escalating privileges. The kernel enforces that only UID 0 (root) or a process with `CAP_CHOWN` capability can change file ownership.

### Special Permission Bits

Beyond the standard rwx bits, there are three special bits that modify execution behavior:

**setuid (SUID, 4000)**: When set on an executable, it runs with the **owner's UID** rather than the caller's UID. `/usr/bin/passwd` is the canonical example — it needs to write to `/etc/shadow` (owned by root, mode 640) so it runs as root even when called by a normal user. Shown as `s` in the owner execute position: `-rwsr-xr-x`.

**setgid (SGID, 2000)**: When set on an executable, it runs with the **owner's GID**. When set on a **directory**, new files created inside inherit the directory's group (instead of the creator's primary GID) — useful for shared project directories. Shown as `s` in the group execute position: `-rwxr-sr-x` (executable) or `drwxr-sr-x` (directory).

**Sticky bit (1000)**: When set on a directory, only the file's **owner** (or root) can delete or rename files inside — other users cannot delete each other's files even if they have write permission on the directory. `/tmp` always has the sticky bit set. Shown as `t` in the other execute position: `drwxrwxrwt`.

<Callout type="deep-dive" title="Security Implications of SUID">
SUID binaries are high-value attack targets. When a SUID binary has a vulnerability (buffer overflow, command injection), an attacker can exploit it to run arbitrary code as the owner (often root). This is why the number of SUID binaries on a system should be minimized and audited regularly.

```bash
# Find all SUID binaries on the system
find / -perm -4000 -type f 2>/dev/null
```

Legitimate SUID binaries include: `passwd`, `sudo`, `ping`, `mount`, `su`. Any SUID binary you don't recognize warrants investigation — it may be a backdoor or a misconfiguration.

In containers, SUID binaries are usually stripped or capabilities are used instead (CAP_NET_RAW for ping, etc.).
</Callout>

### umask: Default Permission Mask

When a new file or directory is created, its initial permissions are determined by the **umask** — a bitmask that removes permissions from the default:

- Default for files: `0666` (rw-rw-rw-)
- Default for directories: `0777` (rwxrwxrwx)
- The umask bits are **subtracted**: `permissions = default & ~umask`

Common umask values:

| umask | New file | New dir | Effect |
|-------|----------|---------|--------|
| `022` | 644 | 755 | Group and other cannot write — the standard default |
| `027` | 640 | 750 | Group can read, other has nothing |
| `077` | 600 | 700 | Owner only |

```bash
umask          # show current umask
umask 027      # set umask for current shell session
```

The umask is set per-process and inherited by child processes. It is configured system-wide in `/etc/profile` or `/etc/login.defs`.
