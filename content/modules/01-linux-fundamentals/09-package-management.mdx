## Overview

A Docker build failed because `apt-get install` couldn't find a package. Was the package renamed? Is the repository missing? Is the index stale? Understanding how repositories, package versions, and dependency resolution work tells you whether to add a PPA, pin a version, run `apt-get update` first, or look for an alternative package. The difference between a 10-second fix and an hour of confusion is understanding the two-layer architecture of Linux package management.

## How It Works

### What is a Package

A package is an archive containing:
- **Binaries**: the compiled executables (e.g., `/usr/bin/curl`)
- **Libraries**: shared `.so` files the binaries depend on
- **Configuration files**: defaults in `/etc/` or `/usr/share/`
- **Metadata**: name, version, description, maintainer, and — critically — the **dependency list** (other packages this one requires)

On Debian-based systems (Ubuntu, Debian, Kali), packages use the `.deb` format. Each `.deb` is essentially a tar archive with a standardized metadata structure.

### Two-Layer Architecture: dpkg and apt

Debian-based Linux uses two layers of package management that solve different problems:

**dpkg (low level):** Installs, removes, and queries individual `.deb` files on disk. It does NOT fetch packages from the internet and does NOT resolve dependencies. If you give it a `.deb` with unsatisfied dependencies, it will fail and tell you what's missing.

**apt (high level):** Knows about repositories (catalogs of packages on the internet), resolves dependencies automatically, and calls `dpkg` to do the actual installation. Think of `dpkg` as manually placing a file; `apt` is an app store that figures out everything you need first.

```
You type: apt-get install nginx
         ↓
apt:     fetch package index → resolve dependencies → download .deb files
         ↓
dpkg:    install each .deb file onto the filesystem
```

### Repositories: Where Packages Come From

`apt` fetches packages from **repositories** — servers that host curated collections of packages. The repository list lives at `/etc/apt/sources.list` and in files under `/etc/apt/sources.list.d/`.

A single repository entry looks like:

```
deb http://archive.ubuntu.com/ubuntu jammy main restricted universe
```

Breaking this apart:

| Component | Value | Meaning |
|-----------|-------|---------|
| `deb` | `deb` | Binary packages (use `deb-src` for source code) |
| URL | `http://archive.ubuntu.com/ubuntu` | The repository server |
| Codename | `jammy` | Ubuntu 22.04 LTS release name |
| Sections | `main restricted universe` | Which package categories to include |

**Repository sections:**

| Section | Contents | License status |
|---------|----------|----------------|
| `main` | Canonical-supported open-source software | Free and open-source |
| `restricted` | Drivers and firmware with restricted licenses | Not fully open-source |
| `universe` | Community-maintained open-source packages | Free, but unsupported |
| `multiverse` | Software with non-free licenses (codecs, etc.) | Non-free |

### apt-get update: Fetching the Index

Running `apt-get install` without first running `apt-get update` is one of the most common causes of "package not found" errors. Here's why:

`apt-get update` downloads the **package index** — a metadata file from each repository listing every available package, version, and download URL. It does NOT install anything. It just updates apt's local catalog of what's available.

<Callout type="warning" title="Always Run apt-get update Before apt-get install in Dockerfiles">
Docker caches each RUN layer. If you have:

```dockerfile
RUN apt-get update
RUN apt-get install -y curl
```

Docker will cache the first layer and never re-run `apt-get update` in subsequent builds. Over time, the cached index becomes stale and packages "disappear" (because the index no longer matches the repository). The correct pattern is:

```dockerfile
RUN apt-get update && apt-get install -y curl
```

A single RUN layer means the index is always fresh when the install runs. This is the official Docker best practice.
</Callout>

### apt-get install: Dependency Resolution

When you run `apt-get install nginx`, apt performs these steps:

1. Looks up `nginx` in the local package index (from the last `apt-get update`)
2. Reads nginx's dependency list from the metadata
3. Recursively resolves all dependencies (nginx depends on libssl, which depends on libc, etc.)
4. Downloads all required `.deb` files that aren't already installed
5. Calls `dpkg` to install each `.deb` in dependency order

The `-y` flag skips the "Do you want to continue? [Y/n]" prompt — essential for scripts and Dockerfiles:

```bash
apt-get install -y nginx curl git
```

<Callout type="tip" title="--no-install-recommends Keeps Docker Images Small">
By default, `apt-get install` also installs "recommended" packages — related software that isn't strictly required. In a Docker container, these are often unnecessary. Adding `--no-install-recommends` can significantly reduce image size:

```bash
apt-get install -y --no-install-recommends nginx
```

For minimal containers (alpine-based Dockerfiles, or slim variants like `ubuntu:22.04`), this flag is especially important.
</Callout>

### Exploring Packages: apt-cache

Before installing, you often want to search for the right package name or understand what it contains:

```bash
# Search for packages by keyword
apt-cache search nginx

# Show detailed info about a package
apt-cache show nginx

# List what packages a package depends on
apt-cache depends nginx

# List packages that depend on this package
apt-cache rdepends libc6

# List available versions of a package
apt-cache policy nginx
```

`apt-cache` reads from the local index (built by `apt-get update`). It never makes network requests.

### dpkg: Low-Level Package Queries

Once packages are installed, `dpkg` is your tool for inspecting them:

```bash
# List all installed packages (long list — pipe through less or grep)
dpkg -l
dpkg -l | grep nginx           # find specific package

# Check if a specific package is installed
dpkg -s nginx

# List all files installed by a package
dpkg -L nginx

# Find which package owns a specific file
dpkg -S /usr/bin/curl

# Install a .deb file directly (no dependency resolution)
dpkg -i mypackage.deb
```

`dpkg -L` is particularly useful when you need to know where a package put its binaries, config files, or library files.

### Version Pinning

In production environments, reproducibility matters. Installing the latest version of a package every time can cause unexpected behavior changes. Version pinning ensures every deployment gets exactly the same package version:

```bash
# Install a specific version
apt-get install -y nginx=1.18.0-0ubuntu1.4

# List available versions to find the exact version string
apt-cache policy nginx
```

For more complex pinning (like "always prefer packages from a specific repository"), use `/etc/apt/preferences`. This is advanced but important for maintaining stable server configurations.

### Removing Packages

```bash
# Remove a package (keeps config files in /etc/)
apt-get remove nginx

# Remove a package AND its config files (purge)
apt-get purge nginx

# Remove orphaned dependencies (packages nothing else depends on)
apt-get autoremove

# Combined: remove package + config + clean up orphans
apt-get purge nginx && apt-get autoremove
```

The difference between `remove` and `purge` matters when re-installing: `purge` gives you a completely clean slate; `remove` preserves your configuration so reinstallation picks up where you left off.

<Callout type="deep-dive" title="How the Dependency Resolver Works">
When `apt-get install package` runs, apt uses a constraint-satisfaction algorithm (in the `apt` library, derived from the original "boolean satisfiability" approach) to find a consistent set of package versions that satisfies all dependencies simultaneously.

The algorithm:

1. Start with the requested package and its stated dependencies
2. For each dependency, find candidate versions in the index
3. Recursively expand: each candidate may introduce new dependencies
4. Check for conflicts: two packages cannot both be installed if they conflict (e.g., `apache2` and `nginx` both want port 80 — but they don't actually conflict at the package level, only at runtime)
5. Apply version constraints: if package A needs `libssl >= 1.1.1` and package B needs `libssl < 1.1.0`, there is no satisfying solution — apt reports a conflict

When apt says "The following packages have unmet dependencies," it means the constraint solver found no valid assignment. Common causes: stale index (run `apt-get update`), conflicting requirements, or a package only available in a repository not in `sources.list`.
</Callout>

**Forward reference:** In the Docker module (Module 3), you'll see why `apt-get update && apt-get install -y` must be in the same RUN command. Docker layer caching makes a separate `apt-get update` step permanently stale after the first build — the install layer sees the old (cached) index and fails to find new packages.

  <ScenarioQuestion
    question="You tried apt-get install tree without running apt-get update first and it failed. In a Dockerfile, why must apt-get update and apt-get install be in the same RUN instruction instead of separate RUN layers?"
    answer="Docker caches each RUN layer as an image layer. If apt-get update and apt-get install are in separate RUN instructions, Docker caches the apt-get update result as its own layer. On subsequent builds, Docker sees that the apt-get update layer has not changed and skips re-running it — the local package index is never refreshed. Over time (days to weeks), the cached index becomes stale: packages get new versions, old versions are removed from repositories, and new package names appear. When apt-get install runs against the stale index, it either cannot find the package or installs an outdated version. Combining them in a single RUN instruction means the index is always fresh when the install runs, because both commands execute together or not at all — there is no cached intermediate layer to skip."
