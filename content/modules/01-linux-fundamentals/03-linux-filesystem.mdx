## Overview

A Docker container's disk filled up, but `df` showed plenty of space available. The container's overlay filesystem was full while the host filesystem had room to spare — two separate filesystems mounted at different points in the same directory tree. Understanding mount points, inodes, and the Filesystem Hierarchy Standard (FHS) would have caught this in minutes instead of hours.

This lesson explains how Linux organizes files: the directory hierarchy and what each directory is for, how inodes store file metadata, how hard links and soft links differ mechanically, how filesystems attach to the tree via mount points, and why "everything is a file" is not just a slogan — it is the fundamental abstraction that makes Unix tools composable.

## How It Works

### The Filesystem Hierarchy Standard

Every file on a Linux system lives under `/` — the root directory. Unlike Windows (where each drive letter has its own tree), Linux has a single unified tree. External drives, network filesystems, virtual filesystems — everything is attached somewhere under `/`.

The FHS defines what each top-level directory contains and why:

| Directory | Purpose |
|-----------|---------|
| `/bin` | Essential user binaries (`ls`, `cp`, `bash`) — must be available before `/usr` is mounted |
| `/sbin` | System administration binaries (`fdisk`, `iptables`) — historically root-only |
| `/etc` | System-wide configuration files (text, editable by admins) |
| `/home` | User home directories (`/home/alice`, `/home/bob`) |
| `/root` | Home directory of the root user |
| `/var` | Variable data that changes during operation: logs (`/var/log`), spool files, databases |
| `/tmp` | Temporary files — cleared on reboot (on most distros). World-writable with sticky bit. |
| `/proc` | Virtual filesystem: kernel state exposed as files (no disk data) |
| `/sys` | Virtual filesystem: hardware and driver state |
| `/dev` | Device files: block devices, character devices |
| `/usr` | User programs, libraries, documentation — the "secondary hierarchy" |
| `/usr/bin` | Most user-space programs (`vim`, `python3`, `git`) |
| `/usr/lib` | Shared libraries for programs in `/usr/bin` |
| `/lib` | Shared libraries needed for binaries in `/bin` and `/sbin` |
| `/opt` | Optional software installed outside the package manager |
| `/mnt` | Conventional mount point for manually mounted temporary filesystems |
| `/media` | Mount points for removable media (USB drives, CDs) |
| `/boot` | Kernel images, initrd, bootloader files |

<Callout type="tip" title="Why /bin and /usr/bin Both Exist">
Historically, `/usr` was a separate filesystem mounted later in the boot process. Essential tools (`ls`, `sh`, `mount`) needed to be available before `/usr` was mounted, so they lived in `/bin`. Modern systems (Ubuntu 20.04+, Fedora 17+) have merged them: `/bin` is now a symlink to `/usr/bin`. But the distinction matters when examining older systems or rescue scenarios.
</Callout>

### Inodes: The Metadata Objects

When you create a file, the filesystem allocates two things: **the data blocks** (the actual file contents) and an **inode** (index node) that describes the file.

An inode stores:
- File type (regular file, directory, symlink, block device, char device, pipe, socket)
- Permissions (mode bits)
- Owner UID and group GID
- File size in bytes
- Timestamps: atime (last access), mtime (last modification), ctime (last status change)
- Link count (number of directory entries pointing to this inode)
- Pointers to data blocks

**What the inode does NOT store**: the filename. Filenames are stored in **directory entries** (dentries) — a directory is just a special file that maps filenames to inode numbers. This is the key insight behind hard links.

<Callout type="deep-dive" title="Hard Links and the Inode Model">
A **hard link** is just a directory entry that points to the same inode as another directory entry. Both filenames reference the same underlying inode — the same permissions, the same data, the same timestamps. You cannot tell which is the "original" because there is no original; they are equals.

```bash
ln /etc/passwd /tmp/passwd-hardlink
ls -i /etc/passwd /tmp/passwd-hardlink
# Both show the same inode number
stat /etc/passwd | grep Links
# Links: 2  (the inode now has 2 directory entries pointing to it)
```

When you delete a file with `rm`, you are removing a directory entry. The inode and its data blocks are only freed when the link count reaches zero. This is why a program that has a file open can continue to read it after `rm` deletes the filename — the inode persists until all file descriptors are closed AND the link count is zero.

**Hard link limitations**: You cannot hard link across different filesystems (because inode numbers are only unique within one filesystem). You cannot hard link directories (this would create cycles in the filesystem tree, breaking tools that traverse it). For cross-filesystem or directory links, use soft links.
</Callout>

### Soft Links (Symbolic Links)

A **symbolic link** (symlink) is a special file that contains a path string. When the kernel resolves a symlink, it follows the path stored inside it. Unlike a hard link, a symlink:

- Can span filesystems
- Can point to directories
- Has its own inode (the symlink file itself is a filesystem object)
- Can be broken (dangling symlink) if the target is deleted

### Mount Points: Attaching Filesystems to the Tree

Linux presents all filesystems — regardless of device or type — through a single directory tree. The mechanism is **mounting**: attaching a filesystem to a directory, called the **mount point**.

When a filesystem is mounted at `/data`, files inside that filesystem appear under `/data/`. The mount point directory itself (on the parent filesystem) is hidden while the filesystem is mounted.

<Callout type="warning" title="Running Out of Inodes">
A filesystem has a fixed number of inodes allocated when it is formatted. You can run out of inodes before running out of disk space — this happens when you create millions of small files (log files, package manager caches, Node.js node_modules).

When inodes are exhausted, you cannot create new files even with available disk space. Check inode usage with `df -i`. The error you will see is "No space left on device" — misleading because disk space is not the issue.
</Callout>

### Everything Is a File

In Linux, the "everything is a file" philosophy means that a wide range of system objects are accessed through the same `open/read/write/close` interface:

| Object | Type indicator in `ls -la` | Example |
|--------|---------------------------|---------|
| Regular file | `-` | `/etc/passwd` |
| Directory | `d` | `/home/` |
| Symbolic link | `l` | `/bin -> /usr/bin` |
| Block device | `b` | `/dev/sda` (hard disk) |
| Character device | `c` | `/dev/null`, `/dev/urandom` |
| Named pipe (FIFO) | `p` | Created with `mkfifo` |
| Socket | `s` | `/var/run/docker.sock` |

**Why this matters**: The same tools — `cat`, `echo`, `dd`, `cp` — work on all of these. You can `cat /dev/urandom` (read random bytes), `dd if=/dev/sda of=backup.img` (copy a raw disk to a file), or `echo "data" > /dev/ttyS0` (write to a serial port). The interface is uniform regardless of the underlying mechanism.

### VFS: The Virtual Filesystem Switch

Different filesystem types — ext4, XFS, btrfs, FAT32, NFS, tmpfs, procfs — all have different on-disk (or in-memory) layouts. The **VFS** (Virtual Filesystem Switch) is the kernel layer that presents a uniform interface to all of them.

When your program calls `open("/home/alice/file.txt", O_RDONLY)`, the kernel:
1. Resolves the path `/home/alice/file.txt` through the VFS
2. Determines which filesystem is mounted at `/home`
3. Dispatches to that filesystem's `open` implementation
4. Returns a file descriptor

The VFS defines interfaces (as C function pointers: `inode_operations`, `file_operations`, `super_operations`) that each filesystem driver implements. This is why you can `cat` a file on ext4, a file on NFS, and a virtual file in `/proc` with the same command.
