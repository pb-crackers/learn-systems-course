## Overview

The exercises so far have put everything in one place: a single working directory with `main.tf`, `variables.tf`, and `outputs.tf`. That works for a single container. It does not work when you need to provision a frontend, an API server, a database, and a cache — each with its own configuration. Without modules, you would copy-paste the same resource blocks repeatedly, changing only a few values. Any change to the common pattern means updating every copy.

Modules are OpenTofu's answer to this problem. A module is a directory of `.tf` files with a defined interface — inputs (variables) and outputs. You define it once, call it multiple times with different inputs, and get different infrastructure each time. This lesson explains the mechanism: what root and child modules are, how variable passing works, how outputs compose, and when to use registry modules vs local ones.

## How It Works

### Root Module vs Child Module

Every OpenTofu working directory is a **root module** — the set of `.tf` files in the directory where you run `tofu plan` and `tofu apply`. The root module is your entry point: it defines the terraform block, providers, and calls to other modules.

A **child module** is a directory of `.tf` files that another module calls via a `module` block. The child module does not have a `terraform {}` or `provider {}` block — it inherits the providers from its caller. It has its own `variables.tf` (the input interface) and `outputs.tf` (the output interface).

```
docker/iac/03-modules/          ← Root module (where you run tofu apply)
├── main.tf                     ← Calls child modules
├── terraform.tfstate           ← State is always in the root module
└── modules/
    └── web-container/          ← Child module
        ├── main.tf             ← The implementation
        ├── variables.tf        ← The input interface
        └── outputs.tf          ← The output interface
```

State always lives in the root module. The root module's `terraform.tfstate` tracks all resources created by all modules it calls.

### The module{} Block: Calling a Child Module

You call a child module with a `module` block in the root module's `main.tf`:

```hcl
module "frontend" {
  source        = "./modules/web-container"
  name          = "learn-frontend"
  image_name    = "nginx:1.25"
  internal_port = 80
  external_port = 8081
}
```

The block structure: `module "LABEL" { source = "PATH", variable_assignments... }`.

- **LABEL:** A local name for this module call. Used to reference outputs later.
- **source:** Where to find the module. A local path (`./modules/web-container`), a git URL, or a registry address.
- **Variable assignments:** Every input variable declared in the child module's `variables.tf` can be assigned here. Omitting a variable with a default uses that default. Omitting a required variable (no default) causes an error.

Call the same module a second time with different inputs:

```hcl
module "api" {
  source        = "./modules/web-container"
  name          = "learn-api"
  image_name    = "nginx:1.25"
  internal_port = 80
  external_port = 8082
}
```

OpenTofu runs both module calls. Each creates its own `docker_image` and `docker_container` resources. With two module calls, `tofu plan` shows 4 resources to create (2 images + 2 containers).

### Child Module Structure

The child module looks like a regular OpenTofu configuration, minus the `terraform {}` and `provider {}` blocks:

```hcl
# modules/web-container/variables.tf
variable "name" {
  type        = string
  description = "Name for the Docker container"
}

variable "image_name" {
  type        = string
  description = "Docker image name and tag to run"
}

variable "internal_port" {
  type        = number
  description = "Port exposed inside the container"
  default     = 80
}

variable "external_port" {
  type        = number
  description = "Host port to map to the internal port"
}
```

```hcl
# modules/web-container/main.tf
resource "docker_image" "image" {
  name         = var.image_name
  keep_locally = true
}

resource "docker_container" "container" {
  name  = var.name
  image = docker_image.image.image_id

  ports {
    internal = var.internal_port
    external = var.external_port
  }
}
```

```hcl
# modules/web-container/outputs.tf
output "container_id" {
  description = "ID of the created Docker container"
  value       = docker_container.container.id
}

output "container_name" {
  description = "Name of the Docker container"
  value       = docker_container.container.name
}
```

The child module's variables are the function parameters. The child module's outputs are the return values.

<Callout type="deep-dive" title="Modules Are Functions for Infrastructure">
Modules are to OpenTofu what functions are to programming. The module's `variables.tf` is the function signature — it defines what inputs the module accepts. The `main.tf` is the implementation — it creates the infrastructure. The `outputs.tf` is the return value — it exposes results to the caller.

Just like a function can be called multiple times with different arguments, a module can be called multiple times (via separate `module` blocks) with different variable values to produce different infrastructure. And just like a well-designed function hides complexity behind a clean interface, a well-designed module hides resource creation details behind a simple set of inputs.
</Callout>

### Accessing Module Outputs

In the root module, you access a child module's outputs via `module.LABEL.output_name`:

```hcl
# In root main.tf — referencing the frontend module's output
output "frontend_container_id" {
  value = module.frontend.container_id
}

output "api_container_id" {
  value = module.api.container_id
}
```

You can also pass one module's output as input to another module — this is module composition:

```hcl
module "network" {
  source = "./modules/docker-network"
  name   = "learn-net"
}

module "frontend" {
  source      = "./modules/web-container"
  name        = "learn-frontend"
  network_id  = module.network.network_id   # output of network module
  external_port = 8081
}
```

OpenTofu understands dependencies between module calls and creates them in the correct order.

### The terraform{} Block in Root vs Child Modules

The root module has the `terraform {}` block with `required_providers`. Child modules do **not** declare providers — they inherit them from the root module. This is intentional: the root module owns the provider configuration, and child modules use whatever providers the root declares.

If a child module needs a provider the root hasn't declared, OpenTofu will error at `tofu init`. The fix is always to add the provider declaration to the root module's `terraform {}` block.

### Registry Modules vs Local Modules

Local modules (`source = "./modules/name"`) are defined in your repository. They are transparent — you can read every `.tf` file to understand exactly what they create.

Registry modules (`source = "registry.opentofu.org/namespace/module-name/provider"`) are published by the community or vendors. They work the same way but the implementation lives on the registry server, downloaded during `tofu init`.

```hcl
# Example: a hypothetical registry module
module "vpc" {
  source  = "registry.opentofu.org/terraform-aws-modules/vpc/aws"
  version = "5.5.2"

  name = "my-vpc"
  cidr = "10.0.0.0/16"
}
```

Version constraints for registry modules follow the same syntax as provider versions: `"5.5.2"` for exact, `"~> 5.5"` for patch-only updates, `">= 5.0.0, < 6.0.0"` for a range.

In production codebases, private modules are often stored in git repositories and referenced with git source URLs:

```hcl
module "vpc" {
  source = "git::https://github.com/my-org/terraform-modules.git//vpc?ref=v2.1.0"
}
```

The `//vpc` selects the `vpc` subdirectory within the repository. The `?ref=v2.1.0` pins to a git tag.

<Callout type="tip" title="Start with Local Modules">
Start with local modules (`source = "./modules/name"`) before using registry modules. Local modules let you see exactly what is inside — every resource, every variable, every output. Registry modules are convenient for production use but hide the implementation, which makes them harder to debug and learn from.

Once you understand how a module category works (VPC modules, ECS modules, etc.), using registry modules is appropriate. But for learning — always start local.
</Callout>

### Running tofu init with Modules

When your configuration includes `module` blocks, `tofu init` initializes both providers and modules:

```bash
tofu init

# Output:
# Initializing the backend...
# Initializing modules...
# - frontend in modules/web-container
# - api in modules/web-container
#
# Initializing provider plugins...
# - Installing kreuzwerker/docker v3.9.0...
```

OpenTofu registers each module call (frontend + api both using `web-container`), then downloads providers. A `.terraform/modules/` directory is created alongside `.terraform/providers/`.
