## Overview

HCL (HashiCorp Configuration Language) is the language you write OpenTofu configurations in. It looks like a cross between JSON and a programming language — structured enough for machines to parse, readable enough for humans to review in pull requests. This lesson teaches you how each piece of HCL works: where providers come from, what a resource block means, how variables flow through configurations, and how outputs expose values to callers.

By the end you will write a complete configuration: three files (`main.tf`, `variables.tf`, `outputs.tf`) that provision a real Docker container running nginx.

## How It Works

### File Structure: OpenTofu Reads All .tf Files in a Directory

OpenTofu does not have a single entry point like `index.js`. When you run `tofu plan` in a directory, **OpenTofu reads every `.tf` file in that directory and merges them into a single configuration**. The split into `main.tf`, `variables.tf`, and `outputs.tf` is a convention — you could put everything in one file, but separating concerns makes configurations easier to navigate.

```
my-infra/
├── main.tf          # Provider config, resource declarations
├── variables.tf     # Input variable declarations
└── outputs.tf       # Output value declarations
```

OpenTofu processes all three files together. A variable declared in `variables.tf` is available in `main.tf`. An output in `outputs.tf` can reference resources in `main.tf`.

### The terraform{} Block: Required Providers

Every configuration starts with a `terraform {}` block that declares which providers it needs:

```hcl
terraform {
  required_providers {
    docker = {
      source  = "registry.opentofu.org/kreuzwerker/docker"
      version = "3.9.0"
    }
  }
}
```

**Source:** Tells OpenTofu where to download the provider. The format is `registry/namespace/type`. `registry.opentofu.org` is the OpenTofu provider registry — use this instead of `registry.terraform.io` for the Docker provider (see warning callout below).

**Version:** Pins the provider to an exact version. `3.9.0` means exactly that version, not newer. This matters because `v4.0.0-beta2` of the Docker provider was released with breaking changes — a loose constraint like `~> 3.0` should be safe under SemVer, but explicit pinning removes all ambiguity and is the right practice for learners and production configs alike.

**Note on `terraform {}` vs `opentofu {}`:** OpenTofu 1.x accepts both `terraform {}` and `opentofu {}` as the root block name. Use `terraform {}` — it works in both tools and is what you will see in every codebase and job posting.

### The provider{} Block: Configuring the Provider

After declaring required providers, you configure the provider with a `provider` block:

```hcl
provider "docker" {}
```

This minimal form uses all defaults — the Docker provider connects to the Docker daemon at `/var/run/docker.sock`. For most Docker Desktop installations on macOS and Linux, this is correct and no further configuration is needed.

If you use a different Docker runtime, you may need to specify the socket path (see warning callout below).

### Resource Blocks: Declaring Infrastructure

A resource block declares a piece of infrastructure you want to exist:

```hcl
resource "docker_image" "nginx" {
  name         = "nginx:1.25"
  keep_locally = true
}
```

The block structure: `resource "TYPE" "NAME" { arguments }`.

- **TYPE:** The resource type, provided by the Docker provider. `docker_image` manages Docker images, `docker_container` manages containers.
- **NAME:** A local label you choose. Used to reference this resource elsewhere in the configuration.
- **Arguments:** The configuration for this specific resource. Different resource types have different required and optional arguments.

The `keep_locally = true` argument tells the Docker provider not to delete the image when you run `tofu destroy`. Without this, running `tofu destroy` would delete the local nginx image, which you probably want to keep.

A container resource that references the image:

```hcl
resource "docker_container" "web" {
  name  = var.container_name
  image = docker_image.nginx.image_id

  ports {
    internal = 80
    external = var.external_port
  }

  labels {
    label = "managed-by"
    value = "opentofu"
  }
}
```

Notice `docker_image.nginx.image_id` — this is a **resource reference**. It refers to the `image_id` attribute of the `docker_image` resource named `nginx`. OpenTofu uses these references to determine dependency order: it creates the image resource before the container because the container depends on the image's ID.

### Variable Blocks: Parameterizing Configurations

Hard-coding values in `main.tf` makes configurations inflexible — you would need a different configuration for every environment. Variables let you parameterize values:

```hcl
variable "container_name" {
  type        = string
  description = "Name for the Docker container"
  default     = "learn-iac-web"
}

variable "external_port" {
  type        = number
  description = "Host port to map container port 80 to"
  default     = 8080

  validation {
    condition     = var.external_port >= 1024 && var.external_port <= 65535
    error_message = "Port must be in the range 1024-65535 (non-privileged ports)."
  }
}
```

The `validation` block runs before `tofu apply` and rejects invalid inputs with the `error_message`. If you try to use port 80 (a privileged port requiring root), the validation catches it before any infrastructure is touched.

You reference variables with `var.name` syntax: `var.container_name`, `var.external_port`.

**Setting variable values** (in priority order, highest first):
1. `-var` flag: `tofu apply -var="external_port=9090"`
2. `.tfvars` file: `terraform.tfvars` in the working directory is loaded automatically
3. Environment variable: `TF_VAR_external_port=9090`
4. Default value in the `variable` block

### Output Blocks: Exposing Values

Outputs expose values from your configuration — useful for seeing what was created, and essential for module composition (covered in lesson 4):

```hcl
output "container_id" {
  description = "ID of the created Docker container"
  value       = docker_container.web.id
}

output "container_name" {
  description = "Name of the created Docker container"
  value       = docker_container.web.name
}
```

After `tofu apply`, outputs are printed to the terminal. You can also query them explicitly: `tofu output container_name`.

### The Complete Workflow

Every OpenTofu workflow follows the same sequence:

```bash
tofu init      # Download providers declared in required_providers
tofu fmt       # Format .tf files to canonical style (modifies files in place)
tofu validate  # Check syntax and schema validity (does not access infrastructure)
tofu plan      # Preview changes (safe, read-only, shows diff)
tofu apply     # Execute the plan (creates/modifies/destroys infrastructure)
tofu destroy   # Destroy all managed resources
```

`tofu init` must run before any other command in a new directory — it downloads the provider binaries into a `.terraform/` directory. This directory should be added to `.gitignore` (the provider binaries are large and platform-specific).

<Callout type="warning" title="Docker Socket Path for Non-Docker-Desktop Users">
The Docker provider connects to your Docker daemon via `/var/run/docker.sock`. If you use Rancher Desktop or Podman Desktop instead of Docker Desktop, the socket path may differ. Configure the provider explicitly:

```hcl
provider "docker" {
  host = "unix:///Users/yourusername/.rd/docker.sock"
}
```

For Rancher Desktop: `~/.rd/docker.sock`. For OrbStack: `/var/run/docker.sock` (same as Docker Desktop). If `tofu init` succeeds but `tofu plan` fails with "Cannot connect to the Docker daemon," this is the issue.
</Callout>

<Callout type="warning" title="Use registry.opentofu.org, Not registry.terraform.io">
Use `registry.opentofu.org/kreuzwerker/docker` as the source in your `required_providers` block — not `registry.terraform.io/kreuzwerker/docker`. The Terraform registry has had GPG key verification issues with the kreuzwerker Docker provider since v3.7.0. The OpenTofu registry resolves these cleanly. This is the correct source for OpenTofu configurations.
</Callout>

### Complete Example: main.tf, variables.tf, outputs.tf

Here is the complete configuration from the exercise directory `docker/iac/01-basics/`:

```hcl
# docker/iac/01-basics/main.tf
terraform {
  required_providers {
    docker = {
      source  = "registry.opentofu.org/kreuzwerker/docker"
      version = "3.9.0"
    }
  }
}

provider "docker" {}

resource "docker_image" "nginx" {
  name         = "nginx:1.25"
  keep_locally = true
}

resource "docker_container" "web" {
  name  = var.container_name
  image = docker_image.nginx.image_id

  ports {
    internal = 80
    external = var.external_port
  }

  labels {
    label = "managed-by"
    value = "opentofu"
  }
}
```

```hcl
# docker/iac/01-basics/variables.tf
variable "container_name" {
  type        = string
  description = "Name for the Docker container"
  default     = "learn-iac-web"
}

variable "external_port" {
  type        = number
  description = "Host port to map container port 80 to"
  default     = 8080

  validation {
    condition     = var.external_port >= 1024 && var.external_port <= 65535
    error_message = "Port must be in the range 1024-65535 (non-privileged ports)."
  }
}
```

```hcl
# docker/iac/01-basics/outputs.tf
output "container_id" {

}

```
