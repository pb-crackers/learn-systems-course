## Overview

Your team manually configured 12 servers via SSH and shell scripts. Three months later nobody remembers which server has which configuration — different software versions, different environment variables, different firewall rules. One server goes down. Rebuilding it takes a full day of guesswork, cross-referencing chat logs and runbooks that were last updated in 2022.

Infrastructure as Code (IaC) solves this problem by treating infrastructure configuration the same way developers treat software: written in files, stored in version control, applied automatically, and auditable at any time. If a server goes down, you rebuild it in minutes from the same code that built the original.

This lesson explains the mechanism before the commands. Once you understand why IaC works the way it does, the commands are obvious.

## How It Works

### Imperative vs Declarative: The Core Distinction

Remember manually creating Docker containers with `docker run`? That is the **imperative** approach — you issue a sequence of commands and the system executes them in order:

```bash
# Imperative: you tell the system HOW to get there
docker run -d --name web nginx:1.25
docker run -d --name db postgres:15
docker network connect my-net web
docker network connect my-net db
```

The problem with imperative scripts: they are sequential instructions. If the first command already ran and you run the script again, you get an error ("container already exists"). If you want to change the port, you have to figure out whether to `docker stop`, `docker rm`, and re-run, or `docker update`. The script tells you what to do, not what the end state should be.

**Declarative IaC** flips this. You describe what you want to exist — the desired end state — and OpenTofu figures out how to get there:

```hcl
# Declarative: you tell the system WHAT you want
resource "docker_container" "web" {
  name  = "web"
  image = docker_image.nginx.image_id
}

resource "docker_container" "db" {
  name  = "db"
  image = docker_image.postgres.image_id
}
```

OpenTofu reads this, compares it to what currently exists, and applies the minimum set of changes to reach the desired state. Run it twice and the second run does nothing — already there. Change the port and reapply — OpenTofu figures out whether to update in place or recreate.

### The IaC Workflow: Write, Plan, Apply

Every OpenTofu workflow follows this cycle:

```
Write HCL → tofu plan (preview changes) → tofu apply (execute changes)
```

1. **Write HCL:** You declare the infrastructure you want in `.tf` files.
2. **`tofu plan`:** OpenTofu reads your HCL, reads the current state, and computes the diff. It shows you exactly what will be created, modified, or destroyed — before touching anything.
3. **`tofu apply`:** You confirm the plan. OpenTofu executes the changes and updates the state file.

The `tofu plan` step is the safety net. You always see what will happen before it happens. This is impossible with shell scripts — a script starts executing immediately with no preview.

### The State File: OpenTofu's Source of Truth

When you run `tofu apply`, OpenTofu does not just create infrastructure — it records what it created in a file called `terraform.tfstate`. This JSON file is OpenTofu's memory of the world.

The state file maps each HCL resource block to a real-world infrastructure object:

```json
{
  "resources": [
    {
      "type": "docker_container",
      "name": "web",
      "instances": [
        {
          "attributes": {
            "id": "a1b2c3d4e5f6...",
            "name": "learn-iac-web",
            "image": "sha256:abc123...",
            "ports": [{"external": 8080, "internal": 80}]
          }
        }
      ]
    }
  ]
}
```

Without this file, OpenTofu cannot know what it previously created. It cannot distinguish "this container does not exist yet" from "this container was created by someone else." The state file is what makes declarative IaC work at all.

### Drift Detection: When Reality Diverges from Desired State

Drift happens when someone modifies infrastructure outside of OpenTofu — manually stopping a container, changing a firewall rule in the cloud console, or running `docker rm` on a managed container.

On every `tofu plan` run, OpenTofu:
1. Reads the current state file (what it thinks exists)
2. Queries the actual infrastructure (what actually exists)
3. Computes the diff between your HCL (what you want) and reality (what's there)

If someone manually stopped a container that OpenTofu manages, the next `tofu plan` shows:

```
  # docker_container.web must be replaced
-/+ resource "docker_container" "web" {
      ~ id     = "a1b2c3..." -> (known after apply)
        name   = "learn-iac-web"
    }

Plan: 1 to add, 0 to change, 1 to destroy.
```

OpenTofu detected that the running container (tracked in state) no longer matches reality (container is stopped). Running `tofu apply` restores the desired state. This is drift detection in action.

### A Brief History: Terraform → OpenTofu

**Terraform** was created by HashiCorp in 2014 and became the dominant IaC tool. In August 2023, HashiCorp changed Terraform's license from the open-source Mozilla Public License to the Business Source License (BSL), which restricts commercial use.

The community responded by forking Terraform as **OpenTofu** under the Linux Foundation in September 2023. OpenTofu uses the original Mozilla Public License (MPL-2.0) and maintains 1:1 CLI and HCL compatibility with Terraform.

**For your job search:** Job postings say "Terraform." OpenTofu is what you use in this course. Everything you learn here — HCL syntax, the plan/apply cycle, state management, modules — works identically with both tools. The only difference is the binary name: `tofu` instead of `terraform`.

<Callout type="deep-dive" title="IaC and CI/CD: The Deployment Piece Done Right">
In the CI/CD module, you learned that pipelines automate the build/test/deploy lifecycle. The deploy step is where IaC fits: instead of a pipeline running `ssh` commands and hoping the server is in the right state, the pipeline runs `tofu apply` against a state file.

This means deployments are reproducible — the same HCL produces the same infrastructure every time. It means infrastructure is auditable — changes go through pull requests and review. And it means rollback is a real option — `git revert` + `tofu apply` restores the previous state.

IaC is the "deploy" piece of CI/CD done correctly.
</Callout>

<Callout type="tip" title="CLI Command Names">
OpenTofu uses the `tofu` CLI command. Terraform uses `terraform`. The syntax, file structure, and workflow are identical — only the binary name differs. When you read job postings or tutorials that show `terraform plan`, mentally substitute `tofu plan`.
</Callout>
