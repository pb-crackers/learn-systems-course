## Overview

In the Docker volumes lesson, you learned that containers have ephemeral writable layers — data written inside a container disappears when the container is removed. Named volumes solved that for local development. Cloud storage solves it at infrastructure scale, across multiple machines and availability zones.

Cloud storage comes in three fundamentally different types: block, object, and file. Choosing wrong is one of the most expensive architectural mistakes in cloud infrastructure — the wrong choice either limits your scale, increases your cost by 10x, or breaks your application in subtle ways. Understanding the model of each type prevents that.

## How It Works

### Block Storage = Docker Named Volumes

In the Docker volumes lesson, you created named volumes with `docker volume create` and mounted them at container paths. Cloud block storage works exactly the same way — a virtual disk that attaches to a VM or container and persists when the instance stops.

```bash
# Docker: create a named volume and attach it to a container
docker volume create dbdata
docker run -v dbdata:/var/lib/postgresql/data postgres:16

# Cloud equivalent (conceptual):
# Create an EBS volume in AWS (a virtual 100GB disk)
# Attach it to your EC2 instance at /dev/xvdf
# Format it: mkfs.ext4 /dev/xvdf
# Mount it: mount /dev/xvdf /var/lib/postgresql/data
# Now PostgreSQL writes to the volume, not the ephemeral instance disk
```

Block storage is formatted with a filesystem (ext4, xfs, NTFS) exactly like a physical hard drive. You run the same filesystem commands you learned in the sysadmin module — `df -h`, `mount`, `lsblk`, `mkfs`. When an EC2 instance stops, the EBS volume persists — just like a named Docker volume survives `docker rm`.

Key limitation: a block volume attaches to **one instance at a time**. You cannot mount the same EBS volume to two EC2 instances simultaneously. This mirrors the Docker volume behavior — you can share a named volume across containers on the same host, but not across different Docker hosts.

| Provider | Block Storage Service | Equivalent |
|----------|----------------------|------------|
| AWS | EBS (Elastic Block Store) | docker volume with local driver |
| GCP | Persistent Disk | docker volume with local driver |
| Azure | Azure Disk Storage | docker volume with local driver |

<Callout type="deep-dive" title="Cloud VMs Use Block Storage for Their Root Disk">
Remember Docker's overlay filesystem and storage driver from the container internals lesson? Cloud VMs use a similar layered approach. The root volume of an EC2 instance (the disk that contains the OS) is an EBS volume backed by a snapshot. Additional volumes are attached as block devices — `/dev/xvdf`, `/dev/xvdg`, etc. The `df` and `mount` commands you learned in sysadmin work identically on cloud VMs. When you run `df -h` on an EC2 instance, you see EBS volumes listed like physical disks.
</Callout>

### Object Storage = No Direct Docker Equivalent

This is where cloud genuinely adds something new. Object storage stores files as key-value pairs accessible via HTTP URLs — no filesystem, no mount points, no directories, no `ls` command, no `chmod`. Just a flat namespace of objects with globally unique identifiers.

```bash
# No Docker equivalent — this is cloud-native

# Store a file (AWS S3 using the AWS CLI):
aws s3 cp report.pdf s3://my-company-reports/2026/march/report.pdf

# Retrieve the file directly via HTTP (if bucket is public):
curl -O https://my-company-reports.s3.amazonaws.com/2026/march/report.pdf

# The "path" (2026/march/) is just part of the key — there are no real directories.
# S3 simulates directory structure by treating "/" as a delimiter in the key name.
```

The HTTP interface is the defining feature. Any client with an internet connection and the right credentials can retrieve an object from its URL. You do not need to SSH into a machine. You do not need to mount a drive. The object is just a URL.

This matters for two reasons:

1. **Scale.** Object storage scales to billions of files and petabytes with no performance degradation. A block volume tops out at the size of a single disk. An NFS share tops out at the performance of a single file server. S3 serves 11 nines (99.999999999%) durability — AWS redundantly stores each object across multiple AZs automatically.

2. **Access model.** A block volume requires an attached instance. An object in S3 is accessible from anywhere — your API, a browser, a CDN, a Lambda function, a developer's laptop — all via HTTP.

| Provider | Object Storage Service | Pricing Model |
|----------|----------------------|--------------|
| AWS | S3 (Simple Storage Service) | Per GB stored + per request |
| GCP | Cloud Storage | Per GB stored + per operation |
| Azure | Azure Blob Storage | Per GB stored + per operation |

<Callout type="tip" title="The Most Common Architectural Mistake">
Using block storage (an EBS volume or a named Docker volume) to store files that should be in object storage. If your files are accessed via HTTP and do not need filesystem operations (`ls`, `chmod`, `grep`), use object storage. It scales infinitely and costs significantly less per GB. Block storage is for databases and applications that need a local filesystem. Object storage is for anything users or services retrieve over HTTP — user uploads, static assets, reports, backups, logs.
</Callout>

### File Storage = Network-Mounted Filesystem

File storage bridges block and object. It is a managed NFS server — multiple instances mount the same filesystem simultaneously, and they all see each other's changes in real-time. No copy-on-write, no HTTP — just a networked filesystem.

```bash
# Mount an NFS share (the local equivalent of cloud file storage)
# From the networking module, NFS works the same way as cloud file storage

sudo mount -t nfs 10.0.0.5:/shared /mnt/shared
# After this mount, any file written to /mnt/shared by this machine
# is immediately visible to all other machines that have mounted the same NFS share.

# Cloud file storage: same model, provider manages the NFS server
# AWS EFS: mount target is at a DNS name like fs-abc12345.efs.us-east-1.amazonaws.com
# sudo mount -t nfs4 fs-abc12345.efs.us-east-1.amazonaws.com:/ /mnt/efs
```

The key differentiation from block storage: multiple instances can mount the same file storage volume simultaneously. This makes it useful for applications that need shared configuration, shared uploads visible to all app servers, or legacy applications that expect a filesystem interface but need to run on multiple machines.

| Provider | File Storage Service | Protocol |
|----------|---------------------|---------|
| AWS | EFS (Elastic File System) | NFS v4 |
| GCP | Filestore | NFS v3/v4 |
| Azure | Azure Files | SMB / NFS |

### The Storage Decision Framework

| Workload | Storage Type | Why |
|----------|-------------|-----|
| Database (PostgreSQL, MySQL, MongoDB) | Block | Needs filesystem IOPS, one writer, random access patterns |
| User file uploads | Object | Accessed via HTTP, arbitrary scale, no filesystem ops needed |
| Static website assets (images, CSS, JS) | Object | Served directly to browsers via HTTP |
| Application logs (archive) | Object | Written once, read occasionally, HTTP retrieval works |
| Shared config files for multiple app servers | File (NFS) | Multiple instances need simultaneous read access |
| Container/VM root disk | Block | OS needs a filesystem to boot from |
| ML training datasets | Object | Large files, batch access, HTTP from compute cluster |

    answer="PostgreSQL requires a filesystem with random read/write access and consistent low-latency IOPS — the ability to read or write any 8KB database page in microseconds in any order. Object storage has no filesystem at all: there is no concept of seeking to a byte offset within an object, no way to update a portion of a file without replacing the entire object, and HTTP round-trip latency for each read would be hundreds of milliseconds instead of microseconds. PostgreSQL also requires POSIX filesystem semantics for its write-ahead log and lock files, which S3 does not provide. Block storage presents as a virtual disk with a filesystem (ext4 or xfs), which PostgreSQL interacts with using the same kernel filesystem calls it would use on a physical hard drive.
