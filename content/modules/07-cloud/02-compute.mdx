## Overview

Cloud providers offer three tiers of compute. Each tier abstracts away more of the infrastructure beneath it — and each maps to something you already know from the Docker module. Understanding the tiers means knowing when to use each: choosing wrong costs you either control you needed or complexity you did not.

## How It Works

### Tier 1: Virtual Machines

In the Docker module, you learned that containers use Linux namespaces and cgroups to isolate processes while sharing the host kernel. A virtual machine goes one layer deeper — it virtualizes the entire hardware, including CPU, memory, network interfaces, and storage controllers. Each VM runs its own kernel.

```
Physical Host (AWS, GCP, or Azure data center machine)
┌─────────────────────────────────────────────────────────┐
│  Hypervisor (KVM, Xen, Hyper-V)                         │
│  ┌─────────────────────┐  ┌─────────────────────┐       │
│  │  VM 1 (your EC2)    │  │  VM 2 (another user) │       │
│  │  ┌───────────────┐  │  │  ┌───────────────┐  │       │
│  │  │ Linux kernel  │  │  │  │ Linux kernel  │  │       │
│  │  │ your OS       │  │  │  │ their OS      │  │       │
│  │  │ your Docker   │  │  │  │ their app     │  │       │
│  │  └───────────────┘  │  │  └───────────────┘  │       │
│  └─────────────────────┘  └─────────────────────┘       │
└─────────────────────────────────────────────────────────┘
```

You SSH into a VM exactly like you SSH into your local Docker host — because it IS a full machine. The difference is that AWS, GCP, or Azure owns the physical hardware. You get the machine via API call.

| Provider | VM Service | CLI to Launch |
|----------|-----------|---------------|
| AWS | EC2 (Elastic Compute Cloud) | `aws ec2 run-instances` |
| GCP | Compute Engine | `gcloud compute instances create` |
| Azure | Azure VMs | `az vm create` |

**When VMs are the right choice:** You need a specific OS version, a custom kernel module, GPU access, or an application that does not run in containers. VMs give you the most control — and the most responsibility. You manage OS patches, Docker installation, and everything above the hardware.

<Callout type="deep-dive" title="VMs vs Containers: The Kernel Boundary">
Containers share the host kernel — `docker run` starts an isolated process using namespaces and cgroups, but the container's `ps` output shares PIDs with the host (just hidden by the PID namespace). A VM has its own kernel. This means:

1. A container image must be built for the host kernel's architecture (ARM64 vs AMD64). A VM can run any OS.
2. A kernel-level bug affects all containers on the host. A VM is isolated at the hypervisor level — a kernel bug in VM 1 does not affect VM 2.
3. VMs start in seconds to minutes. Containers start in milliseconds. For burst traffic that needs thousands of new compute units in seconds, containers win.
</Callout>

### Tier 2: Managed Container Services

You know how to run `docker compose up` to start multiple services on your local machine. Managed container services do the same thing — but across multiple machines, with automatic load balancing and scaling.

```dockerfile
# Your local Dockerfile — unchanged when moving to cloud
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
```

You build this image exactly as you do locally: `docker build -t myapp:v1.0 .`. Then you push it to a container registry (Amazon ECR, Google Artifact Registry, Azure Container Registry — all are registry services identical to Docker Hub). The managed service pulls the image and runs it.

The provider handles:
- Running your image across multiple machines
- Health checks (equivalent to Docker `HEALTHCHECK` directive, but cloud-managed)
- Load balancing traffic across healthy instances
- Replacing failed instances automatically
- Scaling up replicas when CPU or request count rises

You do not write `docker run` commands. You write a service definition — either in HCL with OpenTofu, or in the provider's console/CLI — and the platform handles the `docker run` equivalent on each machine.

| Provider | Managed Container Service | Model |
|----------|--------------------------|-------|
| AWS | ECS (Elastic Container Service) + Fargate | Fargate = serverless compute for containers (no EC2 to manage) |
| GCP | Cloud Run | Fully serverless container platform |
| Azure | Container Apps | Serverless containers with Kubernetes underneath |

**When managed containers are the right choice:** Your app already runs in Docker and you want cloud deployment without managing VMs or Kubernetes. This is the fastest path from a working Dockerfile to cloud production. If you know Docker (and you do), managed container services are the on-ramp.

<Callout type="tip" title="Start Here">
If you know Docker, managed container services are the fastest path to cloud deployment. VMs are for when you need full OS control. Serverless is for event-driven glue code. For most web applications and APIs that already have a Dockerfile, managed containers are the right choice.
</Callout>

### Tier 3: Serverless Functions

Serverless has no Docker equivalent. You write a single function — not an application, not a container, just a function — upload it, and the cloud runs it only when triggered. You pay per invocation (each function call), not per hour (like VMs or managed containers).

```javascript
// AWS Lambda function — a single function, not an application
// No HTTP server, no express, no port binding
// The platform handles all of that
exports.handler = async (event) => {
  const body = JSON.parse(event.body);
  const result = processData(body.input);

  return {
    statusCode: 200,
    body: JSON.stringify({ result })
  };
};
```

The cloud provider manages:
- The runtime (Node.js, Python, Go, Java — you choose at deploy time)
- Scaling from zero to thousands of concurrent invocations in milliseconds
- The HTTP wrapper (API Gateway in front of Lambda, for example)
- The server that actually runs the code (you never see it)

| Provider | Serverless Service | Max Execution Time |
|----------|-------------------|-------------------|
| AWS | Lambda | 15 minutes |
| GCP | Cloud Functions | 60 minutes |
| Azure | Azure Functions | 60 minutes (Consumption plan) |

**When serverless is the right choice:** Event-driven processing (resize an image when uploaded to S3), low-frequency tasks (run a report at midnight), or background jobs triggered by queue messages. Serverless is a poor fit for steady-traffic web APIs — a server that gets 100 requests/second has no "zero instances" state to scale from.

**When serverless is the wrong choice:** Applications with constant traffic (a serverless function that runs 24/7 costs more than a container), long-running connections (WebSockets), or applications that need to hold state in memory between requests.

### The Compute Decision Framework

| Question | Answer | Right Choice |
|----------|--------|-------------|
| Do you need a specific OS or kernel module? | Yes | VM (EC2, Compute Engine, Azure VM) |
| Does your app already have a Dockerfile? | Yes | Managed containers (ECS, Cloud Run, Container Apps) |
| Is your workload event-driven, spiky, or infrequent? | Yes | Serverless (Lambda, Cloud Functions, Azure Functions) |
| Does your app run 24/7 under steady traffic? | Yes | Managed containers (more cost-effective than serverless at scale) |
| Do you need root access to the OS? | Yes | VM |
