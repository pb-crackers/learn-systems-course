## Overview

You have metrics in Prometheus. You can query them with PromQL in the Prometheus UI. But a Prometheus query box is not a monitoring system — it is a calculator. You have to manually type a query every time you want to see something. There is no history, no side-by-side comparison, no alert visualization.

Grafana fills this gap. Grafana is a visualization layer: it queries your data sources (Prometheus, Loki, databases, APIs) and renders the results as panels arranged in dashboards. A dashboard is a collection of panels, each answering one question about your system. The sum of those panels tells a story — the health of your application at a glance.

## How It Works

### Grafana's Role in the Stack

Grafana does not store data. It is a query-and-render system: it sends queries to data sources and renders the responses as visualizations. This is a critical design distinction.

```
Data flow:
  Application → /metrics endpoint → Prometheus scrapes → TSDB stores
  Grafana → sends PromQL query to Prometheus → Prometheus returns data → Grafana renders panel
```

Grafana is stateless with respect to metrics. If you delete Grafana, your metrics are still in Prometheus. If you delete Prometheus, Grafana has nothing to render. Grafana is the view layer; Prometheus is the data layer.

This also means Grafana can query multiple data sources in the same dashboard. A single dashboard can have panels querying Prometheus for metrics and panels querying Loki for logs — different data sources, same visualization surface.

### Datasource Provisioning

The most important Grafana configuration concept for this course is provisioning — configuring Grafana automatically via YAML files instead of through the UI.

Grafana reads YAML files from `/etc/grafana/provisioning/datasources/` at startup. Each file defines one or more datasources. No manual UI configuration is required — on every `docker compose up`, Grafana starts with the datasource already configured.

The Prometheus datasource configuration:

```yaml
# grafana/provisioning/datasources/prometheus.yml
apiVersion: 1

datasources:
  - name: Prometheus
    type: prometheus
    access: proxy
    url: http://prometheus:9090
    isDefault: true
    editable: true
    jsonData:
      httpMethod: POST
    version: 1
```

Key details:
- `access: proxy` means Grafana's backend queries Prometheus on your behalf — your browser does not reach Prometheus directly
- `url: http://prometheus:9090` uses the Docker service name, not `localhost` — inside the Docker network, Grafana reaches Prometheus by service name
- `isDefault: true` makes this the pre-selected datasource when creating new panels
- `httpMethod: POST` is required for Prometheus v3 — older Grafana versions use GET by default, which does not work with long PromQL queries

Why provision over manual configuration: provisioned datasources appear automatically on every fresh stack start. Manual configuration requires clicking through the Grafana UI each time. In a team environment, provisioning ensures every developer has identical Grafana configuration — it is reproducible and version-controlled.

### Dashboard Concepts

A **dashboard** is a JSON document containing a collection of panels, layout positions, variables, and refresh settings. Grafana renders this JSON as an interactive grid of visualizations.

A **panel** is one visualization. Each panel has:
- A PromQL query (or multiple queries)
- A visualization type (time series, stat, gauge, table, heatmap)
- A title, display options, and optional alert configuration

Panel types and when to use them:

| Panel Type | Best For | Example PromQL |
|-----------|---------|----------------|
| **Time Series** | Metrics that change over time — request rate, memory, error rate | `rate(http_requests_total[5m])` |
| **Stat** | Single current value — uptime, current error rate, total users | `up{job="monitored-app"}` |
| **Gauge** | Value with defined min/max and thresholds — CPU usage (0-100%) | `container_cpu_usage_seconds_total` |
| **Table** | Tabular data — top 10 slowest routes, all active alerts | `topk(10, rate(http_requests_total[5m]))` |
| **Heatmap** | Distribution over time — maps directly to histogram bucket data | `rate(http_request_duration_seconds_bucket[5m])` |

### Dashboard Provisioning

Like datasources, dashboards can be provisioned via files. Grafana reads a provider YAML that tells it where to find dashboard JSON files:

```yaml
# grafana/provisioning/dashboards/dashboards.yml
apiVersion: 1

providers:
  - name: default
    type: file
    disableDeletion: false
    updateIntervalSeconds: 30
    options:
      path: /etc/grafana/provisioning/dashboards
      foldersFromFilesStructure: true
```

Grafana scans the `path` directory every `updateIntervalSeconds` and loads any `.json` files it finds as dashboards. The `updateIntervalSeconds: 30` means changes to dashboard JSON files appear in Grafana within 30 seconds without restarting.

Dashboard JSON is complex (100-500 lines per dashboard). The exercise for this lesson uses a provided starter dashboard JSON that includes the four golden signals. Focus your effort on understanding the PromQL queries inside each panel and the panel type choices — not the JSON structure itself.

### Anonymous Access Configuration

The monitoring stack runs Grafana with anonymous access enabled:

```yaml
environment:
  - GF_AUTH_ANONYMOUS_ENABLED=true
  - GF_AUTH_ANONYMOUS_ORG_ROLE=Admin
  - GF_AUTH_DISABLE_LOGIN_FORM=true
```

With these settings, anyone who opens Grafana in their browser is immediately logged in as an admin — no login screen, no credentials. This removes the authentication barrier for local learning environments.

<Callout type="warning" title="Anonymous Admin Access is Development-Only">
`GF_AUTH_ANONYMOUS_ENABLED=true` with `GF_AUTH_ANONYMOUS_ORG_ROLE=Admin` grants every visitor full administrative access to Grafana. In production, Grafana must use authentication: OIDC (single sign-on), LDAP, or at minimum built-in username/password credentials.

The anonymous admin setting exists in this course's Docker stack to eliminate the login barrier for learning labs. If you ever copy these configuration files for a real deployment, remove these three environment variables and configure proper authentication.
</Callout>

<Callout type="tip" title="Port Mapping: App at 3000, Grafana at localhost:3001">
Grafana internally runs on port 3000. The monitoring stack's `compose.yml` maps Grafana's container port to host port 3001 (`"3001:3000"`). This is because the monitored Node.js app also runs on port 3000.

- Open the app at: `http://localhost:3000`
- Open Grafana at: `http://localhost:3001`
- Open Prometheus at: `http://localhost:9090`
- Open Alertmanager at: `http://localhost:9093`

Inside the Docker network, Grafana talks to Prometheus at `http://prometheus:9090` (service name, not localhost). The port 3001 mapping only affects access from your host machine — inside the Docker network, Prometheus is still at port 9090 by its service name.
</Callout>

### The Four Golden Signals

Google's Site Reliability Engineering book defines four signals that, together, tell you whether a service is healthy:

1. **Latency** — how long requests take (P95 or P99 is more meaningful than average)
2. **Traffic** — how many requests the service handles per second
3. **Errors** — the rate of requests that fail (5xx errors, timeouts)
4. **Saturation** — how full the service's resources are (memory usage relative to limit, CPU utilization)

A dashboard with panels for all four golden signals gives you immediate insight into service health. If all four look normal, the service is working. If any one spikes, you know where to investigate.
