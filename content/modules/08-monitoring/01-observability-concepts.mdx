## Overview

Your Node.js application is slow. Users are complaining. You SSH in and run `top` — CPU is fine, memory is fine. The load average is 0.3 on a 4-core machine. Nothing is wrong, according to the machine.

But something is wrong. The machine is fine. The application is not.

A database query is taking 3 seconds. A dependency is timing out. A third-party API is returning slowly. `top` cannot tell you any of this — it sees CPU and memory, not request latency, query times, or service call durations.

This is the gap between system monitoring and observability.

## How It Works

In the System Administration module, you used `top`, `htop`, `vmstat`, and `iostat`. Those tools monitor the machine — CPU cycles, memory pages, disk sectors. Observability goes further: it monitors the application running on the machine. It answers not just "is the server stressed?" but "is my application working correctly, and if not, why?"

The three pillars of observability provide three complementary types of answers.

### Pillar 1: Metrics

**Metrics are numeric measurements collected over time.** Every metric is a number — a count, a size, a duration — associated with a timestamp and optional labels.

There are three metric types:

| Type | Behavior | Examples | PromQL Function |
|------|----------|----------|-----------------|
| **Counter** | Only goes up (resets on restart) | Total requests, total errors, bytes sent | `rate()` over a window |
| **Gauge** | Goes up and down | Current memory, active connections, queue depth | Read the raw value |
| **Histogram** | Distribution across buckets | Request latency, response sizes | `histogram_quantile()` |

The analogy: a car's dashboard. The speedometer (a gauge) shows your current speed — it goes up and down. The odometer (a counter) only ever increases. Neither tells you about the quality of the roads you are driving on — that requires traces.

**Metrics answer "how much" and "how fast."**

A metric can tell you: "There were 2,400 requests in the last minute, and 48 of them returned a 500 error." That is enough to fire an alert. It is not enough to tell you why 48 requests failed.

**Tool:** Prometheus (covered in the next lesson).

### Pillar 2: Logs

**Logs are timestamped text records of discrete events.** A log line says something happened: "2026-03-15T14:23:01Z ERROR Query timeout after 3002ms on /api/orders."

Logs come in two forms:

- **Structured logs** (JSON): `{"level":"error","message":"Query timeout","duration_ms":3002,"route":"/api/orders"}` — machine-parseable, easier to filter and aggregate
- **Unstructured logs** (plain text): `[ERROR] 14:23:01 Query timeout after 3002ms` — human-readable but harder to parse programmatically

The analogy: a flight recorder's black box. Every event is captured in full detail. When something goes wrong, you replay the recording to understand exactly what happened. The downside: at scale, you generate billions of log lines per day. Searching through them is expensive.

**Logs answer "what happened."**

A log can tell you: "The query that timed out was `SELECT * FROM orders WHERE user_id = 12345 ORDER BY created_at` — it ran for 3 seconds and exceeded the 2-second timeout." Now you know not just that something failed, but exactly what failed and what the application was doing.

**Tool:** Loki, ELK stack (covered in Lesson 4).

### Pillar 3: Traces

**Traces follow a single request across every service it touches.** A trace is composed of spans — each span represents one operation (an API call, a database query, a cache lookup). Spans have parent-child relationships: the root span is the incoming request; child spans are the operations it triggered.

The analogy: tracking a package through shipping. The package leaves the warehouse (root span), gets scanned at each facility (child spans), and arrives at your door. Each scan is a span: timestamp, duration, location. If the package is late, you look at the trace to find which facility held it.

**Traces answer "where did the time go."**

A trace can tell you: "The /api/orders request took 3.2 seconds. 0.1s was spent in the API handler. 2.9s was spent in a database query. 0.2s was in a Redis cache check that missed." You now know exactly which service in the call chain is the bottleneck.

**Tools:** Jaeger, Zipkin (these are v2 topics — distributed tracing across services is covered in advanced coursework. For now, understand the concept: traces are how you debug latency in distributed systems.)

<Callout type="deep-dive" title="Monitoring vs Observability">
These two words are often used interchangeably, but they describe different capabilities.

**Monitoring** tells you WHEN something is wrong. An alert fires: "P95 latency exceeded 500ms for the last 2 minutes." You know there is a problem.

**Observability** tells you WHY it is wrong. You open the trace for one of those slow requests and see: "Database service added 450ms. The slow query is a full table scan on the orders table — no index on user_id."

Monitoring requires only metrics and alerts. Observability requires all three pillars working together: metrics to detect the anomaly, logs to see the specific errors, traces to pinpoint which service in the chain is slow.

You need both. Monitoring without observability gives you alerts with no way to diagnose them. Observability without monitoring means you only notice problems when users complain.
</Callout>

<Callout type="tip" title="Bridge from Phase 5: System vs Application Monitoring">
You already know process-level monitoring — `top` shows CPU and memory per process. `vmstat` shows system-wide I/O wait. `iostat` shows disk saturation.

Prometheus extends this to application-level metrics: how many HTTP requests per second, what percentage return a 500 error, how long does the P95 request take. Same concept (numeric measurements over time), different granularity (application behavior instead of kernel resources).

The two levels complement each other. If Prometheus shows high request latency AND `vmstat` shows high `%wa`, you know the application is slow because the disk is slow. If Prometheus shows high latency but `top` shows idle CPU and low I/O wait, the bottleneck is elsewhere — a downstream service, a slow query, a network hop.
</Callout>

### When to Use Which Pillar

The three pillars are not competing alternatives — they are a debugging sequence:

```
1. Start with metrics (cheap, always collected, good for alerting)
   → Alert: P95 latency > 500ms for 2 minutes

2. Drill into logs when the metric anomaly appears (expensive to search, but detailed)
   → Filter logs by time window and service
   → Find: "database query timeout" errors, frequency, affected routes

3. Use traces when the request crosses multiple services (most expensive, most precise)
   → Open trace for a slow request
   → Identify: database service span takes 450ms of the 500ms total
```

Metrics are the cheapest data type to collect and store — they are just numbers. Alert on metrics. Logs are expensive at scale because they are text — do not query logs unless metrics pointed you there. Traces are the most expensive to collect because they require instrumentation across every service — use them to diagnose latency problems in distributed systems.
