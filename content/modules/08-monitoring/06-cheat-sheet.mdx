## Overview

All monitoring concepts, PromQL patterns, Grafana configurations, and incident response templates from this module in one place. Use this as a reference when you are writing queries, building dashboards, or responding to incidents.

## Incident Response

<QuickReference
  sections={[
    {
      title: "Incident Response (Lesson 5)",
      items: [
        { command: "Alert Lifecycle", description: "1. Metric exceeds threshold (HighRequestLatency expr evaluates true). 2. PENDING: alert fires on each evaluation but has not held for the 'for' duration yet. 3. FIRING: 'for' duration elapsed — alert sent to Alertmanager. 4. Alertmanager routes, groups, and deduplicates. 5. Notification sent to receiver (webhook, Slack, PagerDuty). 6. RESOLVED when metric drops below threshold.", example: `up{job="app"}==0 for 1m → PENDING (0-60s) → FIRING → Alertmanager → webhook` },
        { command: "Runbook format", description: "Symptoms: what the alert looks like, what users report. Diagnosis Steps: which PromQL queries to run, what to look for in logs, which services to check. Resolution Actions: ordered steps to fix the most likely causes. Escalation: who to page if resolution steps fail.", example: "HighMemoryUsage runbook:\nSymptoms: process_resident_memory_bytes growing\nDiagnosis: rate(nodejs_heap_used_bytes[5m]) > 0\nResolution: restart service; check for leak in code" },
        { command: "Postmortem format", description: "Summary: 1-2 sentences on what happened. Timeline: timestamped sequence from first symptom to resolution. Root Cause: the actual technical failure (not 'human error'). Action Items: specific, owned, dated tasks to prevent recurrence.", example: "Summary: DB connection pool exhausted causing 503s for 12 minutes.\nRoot Cause: Deployment with missing pool size config.\nAction Items: Add pool size validation to CD pipeline." },
        { command: "On-call rule", description: "You own the incident until it is resolved or escalated to someone who accepts it. Silence without resolution is not on-call handoff. 'I paged someone' is not resolution — confirm they responded and are active.", example: "Incident at 2AM: you are paged, you investigate, you resolve or you wake your escalation contact and confirm they are active." },
        { command: "MTTR vs MTTD", description: "MTTD (Mean Time To Detect): time from failure start to first alert firing. Minimize with: lower alert thresholds, higher scrape frequency. MTTR (Mean Time To Recover): time from detection to full recovery. Minimize with: runbooks, feature flags, rollback automation.", example: "MTTD 2min (alert fires fast) + MTTR 15min (runbook followed) = 17min total incident" },
