## Overview

An alert fires at 2 AM. P95 latency on `/api/orders` exceeded 500ms for two minutes. The on-call engineer wakes up, acknowledges the page, and opens a laptop. What happens next determines whether this is a 15-minute incident or a 4-hour outage.

The difference between engineers who resolve incidents quickly and those who flail is almost never technical knowledge — it is process. Do you know which dashboard to open first? Do you have a runbook that tells you the diagnosis steps? Do you know when to escalate?

This lesson covers the operational side of monitoring: alert lifecycles, Alertmanager configuration, runbook patterns, postmortem formats, and on-call basics. These are the practices that turn Prometheus metrics and Grafana dashboards into an actionable incident response system.

## How It Works

### The Alert Lifecycle

Understanding the flow from metric threshold to notification is essential for debugging "why didn't I get paged?":

```
1. Prometheus evaluates alert rule every evaluation_interval (default: 15s)
   → expr: histogram_quantile(0.95, ...) > 0.5

2. Expression evaluates to true
   → Alert enters PENDING state

3. Alert remains PENDING for the 'for' duration (e.g., for: 2m)
   → Prevents noise from short-lived spikes

4. After 'for' duration: alert becomes FIRING
   → Prometheus sends alert to Alertmanager endpoint

5. Alertmanager receives the firing alert
   → Groups alerts by 'group_by' labels
   → Waits 'group_wait' for additional alerts in the same group
   → Routes to matching receiver

6. Receiver sends notification (Slack, PagerDuty, OpsGenie, email, webhook)

7. Condition resolves: metric drops below threshold
   → Prometheus sends RESOLVED notification to Alertmanager
   → Alertmanager sends resolved notification if 'send_resolved: true'
```

The PENDING → FIRING distinction is important: an alert in PENDING state is "this condition is true right now but hasn't been sustained long enough to page someone." A spike that lasts 90 seconds will go PENDING and then resolve before it ever FIRES — the on-call engineer is not notified.

### Alertmanager Configuration

Alertmanager handles the notification side: receiving firing alerts from Prometheus, grouping related alerts, deduplicating repeated notifications, and routing to the appropriate notification channel.

```yaml
# alertmanager/alertmanager.yml
global:
  resolve_timeout: 5m

route:
  group_by: ['alertname', 'severity']
  group_wait: 10s
  group_interval: 10s
  repeat_interval: 1h
  receiver: 'webhook-log'

receivers:
  - name: 'webhook-log'
    webhook_configs:
      - url: 'http://app:3000/alert-webhook'
        send_resolved: true
```

Route configuration:
- `group_by: ['alertname', 'severity']` — alerts with the same `alertname` and `severity` labels are grouped into a single notification. Without grouping, a single bad deployment could generate 200 individual alert notifications.
- `group_wait: 10s` — wait 10 seconds after the first alert in a group before sending the notification. This allows time for related alerts to arrive and be included in the same notification.
- `group_interval: 10s` — after the initial notification, how long to wait before sending another notification for new alerts in the same group.
- `repeat_interval: 1h` — if an alert is still firing, re-notify after 1 hour. Prevents silent alerts that stay FIRING for hours without reminding the on-call engineer.

For the learning lab, the webhook receiver points to the monitored app itself (`http://app:3000/alert-webhook`). When an alert fires, Alertmanager POSTs the alert JSON to the app, and the app logs it. This creates a visible notification path without requiring Slack credentials or a PagerDuty account. In production, you would configure `slack_configs` or `pagerduty_configs` receivers instead.

### Runbooks

A runbook is a structured procedure for diagnosing and resolving a specific alert. Good runbooks save time during incidents by eliminating the cognitive load of remembering steps while under pressure.

**Runbook structure:**

```markdown
# Runbook: High Request Latency

## Symptoms
- Alert: HighRequestLatency firing
- P95 latency > 500ms for 2+ minutes on one or more routes

## Diagnosis Steps
1. Check the Grafana dashboard — is latency high on ALL routes or just one?
   - All routes high → likely a resource saturation problem (CPU, memory, connection pool)
   - One route high → likely a specific query or dependency issue for that route

2. Check database metrics
   - Is average query time elevated in the database metrics?
   - Are there lock waits or connection pool exhaustion?

3. Check container resource usage
   - Is CPU saturated? (PromQL: rate(container_cpu_usage_seconds_total[5m]))
   - Is memory near the limit? (PromQL: process_resident_memory_bytes)

4. Check recent deployments
   - When did latency start? Correlate with the deployment timeline in Grafana.
   - Did a code change 10 minutes ago coincide with the latency spike?

## Resolution Actions
- **Database query issue:** Check the slow query log, add an index, or optimize the query
- **Resource saturation:** Scale horizontally (add more instances) or increase container limits
- **Code regression:** Rollback to the previous version using the deployment tool

## Escalation
- If not resolved in 30 minutes: escalate to senior on-call engineer
- If database issue: escalate to database administrator
- Escalation contact: [on-call schedule link]
```

The key structural elements of a runbook:
1. **Symptoms** — what does the alert look like, what does the user experience look like
2. **Diagnosis steps** — a decision tree that narrows down the root cause
3. **Resolution actions** — concrete actions for each root cause
4. **Escalation** — who to call and when, with time bounds

### Postmortems

A postmortem (also called a post-incident review or incident retrospective) is a structured analysis written after an incident resolves. Its purpose is to prevent the same incident from recurring — not to assign blame.

**Blameless postmortem template:**

```markdown
# Postmortem: API Latency Spike — 2026-03-15

## Summary
API P95 latency exceeded 2s for 45 minutes, affecting approximately 1,200 users
who experienced slow or failed checkout requests.

## Timeline
- 14:00 — Deploy v2.3.1 with new ORM query for the /api/orders endpoint
- 14:15 — HighRequestLatency alert fires (P95 > 500ms for 2+ minutes)
- 14:17 — On-call engineer receives PagerDuty notification, acknowledges
- 14:20 — Engineer opens Grafana, confirms latency spike on /api/orders route
- 14:35 — Root cause identified: N+1 query in /api/orders endpoint
- 14:45 — Hotfix deployed (eager loading replaces lazy loading), latency recovers
- 14:50 — Alert resolves, resolved notification sent

## Root Cause
The new /api/orders endpoint used ORM lazy loading. For a request to /api/orders
that returned 100 orders, the ORM executed 1 SQL query to fetch orders + 100
additional queries to fetch each order's items — 101 queries instead of 1.

Under normal load, 100 queries complete in 200ms. Under the load spike at 14:15,
the database connection pool was exhausted, and queries queued. P95 latency
climbed from 100ms to 2.1s.

## Action Items
- [ ] Add query count metric to all API endpoints to detect N+1 queries in staging
- [ ] Add load test step to CI pipeline that fails if P95 latency regresses > 20%
- [ ] Update code review checklist: always review ORM query behavior for collection endpoints
- [ ] Write runbook for HighRequestLatency alert (this incident is the template)
```

The postmortem's action items are the most valuable output. They are the system changes that prevent recurrence. Action items must be concrete and assignable — "be more careful" is not an action item.

<Callout type="deep-dive" title="Blameless Postmortems">
The most common mistake in incident retrospectives is assigning blame to individuals. "Engineer X deployed bad code" is not a root cause analysis — it is a symptom that the system failed to catch the bad code before it reached production.

Blameless postmortems focus on system failures, not human failures:

- "Our CI pipeline has no latency regression test" (system failure)
- "The code review process does not require ORM query review for collection endpoints" (system failure)
- "Staging load does not reflect production patterns, so the N+1 was not detected" (system failure)

Each of these system failures has a concrete fix. "Engineer X should be more careful" has no fix — it produces anxiety without improving the system.

The blameless culture principle: treat every incident as a system design problem. Given the same system, the same human, under the same conditions, they will make the same mistake again. Change the system so the mistake is either impossible or caught automatically.
</Callout>

<Callout type="tip" title="On-Call Basics">
On-call rotations ensure that one engineer is always responsible for responding to alerts. The key rules:

1. **Acknowledge means you own it.** When you acknowledge a PagerDuty page, you are committing to either resolving the incident or escalating it — not ignoring it after acknowledgment.

2. **Escalate on time, not on ego.** If you have been debugging for 30 minutes without progress, escalate. The escalation threshold in your runbook exists precisely to prevent one engineer from holding an incident alone for 4 hours.

3. **Document as you go.** Write a running incident timeline in a shared doc or incident channel. Future you writing the postmortem will thank present you.

4. **Tools:** PagerDuty and OpsGenie manage on-call rotations, escalation policies, and schedule overrides. Both can be configured as Alertmanager receivers.
</Callout>
