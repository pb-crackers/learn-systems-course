## Overview

You just joined a startup as their first DevOps engineer. They have a Node.js API that is
"working fine in production" — but users report intermittent slowness. The previous developer
left no monitoring, no dashboards, no alerting. Your mission: set up the full observability
pipeline and diagnose the hidden problem.

<Callout type="tip" title="The Course Finale">
This is the final project of the course. Everything you have learned — Linux fundamentals,
networking, Docker, CI/CD, IaC, and monitoring — comes together here. There are no
step-by-step instructions. You have a brief, starter code, and a verification script.
</Callout>

## What You Will Build

The capstone integrates all eight modules into a complete system:

- **Application layer:** Node.js API instrumented with prom-client (Docker, from Module 3)
- **CI/CD pipeline:** GitHub Actions workflow that lints and builds the app (from Module 5)
- **Infrastructure as Code:** OpenTofu config provisioning Prometheus via Docker provider (from Module 6)
- **Monitoring:** Prometheus scraping app metrics, Grafana dashboards, Alertmanager routing (from Module 8)
- **The Challenge:** An intentional failure hidden in the application that you must diagnose using your monitoring stack

### Architecture

```
[GitHub Actions CI]
       |
       v
  [Docker Build]
       |
       v
  [Docker Compose Stack]
  +-----------+    +------------+    +---------+
  |   App     |--->| Prometheus |--->| Grafana |
  | (Node.js) |    | (scrape)   |    | (dash)  |
  +-----------+    +------------+    +---------+
       |                |
       v                v
  [/metrics]      [Alert Rules]
                       |
                       v
                [Alertmanager]
                       |
                       v
                [Webhook -> App logs]
```

All four services run in a single Docker Compose network called `capstone`. Prometheus scrapes
the app's `/metrics` endpoint every 15 seconds. Alert rules evaluate on the same interval.
When an alert fires, Alertmanager routes it to a webhook receiver on the app itself — creating
a visible notification loop without requiring Slack or PagerDuty credentials.

What does this discrepancy tell you about the nature of the memory problem, and how does it point you toward the root cause in the application source code?"
    answer="The discrepancy is diagnostic. process_resident_memory_bytes includes all memory the OS has allocated to the process — including pages that have been freed by the JavaScript garbage collector but not yet returned to the OS. A flat resident memory reading can mask a JavaScript heap leak because the OS keeps those pages allocated as a performance optimization. The growing increase(nodejs_heap_used_bytes[5m]) tells you that the JavaScript heap itself is growing continuously without being reclaimed by garbage collection, which happens when live object references accumulate and cannot be freed. This points directly to a module-scope data structure — an array or object — that is being populated on a timer or per-request without ever being cleared. The garbage collector cannot free those objects because the module-scope reference keeps them alive indefinitely."

## Skills Integrated

<Callout type="deep-dive" title="Eight Modules in One Project">
Count the modules you are applying in this capstone:

1. **Linux Fundamentals** — reading logs, understanding processes, memory management concepts
2. **Networking** — HTTP endpoints, port mapping, DNS resolution between containers (app resolves to `app:3000` inside the Compose network)
3. **Docker** — Compose stack with 4 services, multi-stage Dockerfile, HEALTHCHECK, resource limits (`mem_limit: 256m`)
4. **System Administration** — service management, log analysis, resource monitoring
5. **CI/CD** — GitHub Actions pipeline for linting and building the container image
6. **Infrastructure as Code** — OpenTofu provisioning Docker resources using the kreuzwerker/docker provider
7. **Cloud Fundamentals** — the local Docker setup maps directly to what cloud engineers do with managed Kubernetes, CloudWatch, and Terraform
8. **Monitoring** — Prometheus metrics, PromQL queries, Grafana dashboards, Alertmanager routing, incident response and postmortem writing

This is what real-world DevOps looks like. Not one tool, not one module — the whole stack working together.
</Callout>
