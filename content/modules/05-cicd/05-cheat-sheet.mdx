## Your Pipeline Exercise

This is the complete CI/CD pipeline for the Node.js app you built in the Docker module. Copy this workflow into your own GitHub repository to see it run.

<CodeBlock language="yaml" title=".github/workflows/ci.yml — complete CI/CD pipeline for docker/app/">
{`name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v6
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint

  test:
    runs-on: ubuntu-latest
    needs: lint
    steps:
      - uses: actions/checkout@v6
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
      - run: npm ci
      - run: npm test

  build-push:
    runs-on: ubuntu-latest
    needs: test
    if: github.ref == 'refs/heads/main'
    permissions:
      contents: read
      packages: write
    steps:
      - uses: actions/checkout@v6
      - uses: docker/setup-buildx-action@v4
      - uses: docker/login-action@v4
        with:
          registry: ghcr.io
          username: \${{ github.actor }}
          password: \${{ secrets.GITHUB_TOKEN }}
      - uses: docker/build-push-action@v7
        with:
          context: .
          push: true
          tags: ghcr.io/\${{ github.repository }}:\${{ github.sha }}
          cache-from: type=gha
          cache-to: type=gha,mode=max`}
</CodeBlock>

<Callout type="tip" title="No secrets setup required for ghcr.io">
You do not need to set up any secrets for ghcr.io — the GITHUB_TOKEN is provided automatically by GitHub Actions with the permissions block in the workflow. The `packages: write` permission grants your workflow the right to push images to your repository's package registry without any manual secret creation.
</Callout>

## Deployment Strategies

<QuickReference
  sections={[
    {
      title: "Deployment Strategies (Lesson 4)",
      items: [
        { command: "Blue/Green deployment", description: "Run two identical environments (blue = current, green = new). Deploy to green, test it, then switch the load balancer. Instant rollback: flip back to blue. Cost: double the infrastructure while both run." },
        { command: "Blue/Green: instant switch", description: "The load balancer flip takes milliseconds — zero downtime. All traffic moves at once (no mixed versions serving users simultaneously)." },
        { command: "Blue/Green: rollback", description: "If green has a problem, flip the load balancer back to blue. The old version was never torn down — rollback is instant." },
        { command: "Rolling deployment", description: "Replace instances one at a time (or in small batches). Both v1 and v2 serve traffic simultaneously during the rollout. Lower cost than blue/green but more complex rollback." },
        { command: "Rolling: gradual", description: "Deploy to 1 of 5 instances → wait → deploy to 2 → wait → ... until all instances run the new version. Fails stop the rollout." },
        { command: "Canary deployment", description: "Send a small percentage of traffic (e.g., 5%) to the new version. Monitor error rates and latency. If metrics are good, increase the percentage gradually until 100%." },
        { command: "Canary: safest approach", description: "Real user traffic tests the new version before full rollout. A bug affecting 5% of users is caught before it affects everyone. Requires traffic-splitting infrastructure (load balancer rules or service mesh)." },
        { command: "Strategy comparison", description: "Blue/Green: fast + safe rollback, expensive. Rolling: lower cost, mixed versions during rollout. Canary: safest for catching production bugs, most infrastructure complexity." },
      ]
    }
