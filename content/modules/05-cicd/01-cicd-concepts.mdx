## Overview

Your team ships code by SSH-ing into a server, running `git pull`, crossing fingers, and hoping nothing breaks. It works — until it doesn't. A missing dependency on the server, a test that only runs on the developer's machine, a deployment at 11pm that takes the site down for 20 minutes. These are symptoms of a manual process that doesn't scale.

**CI/CD (Continuous Integration / Continuous Delivery or Deployment)** is the practice of automating the entire journey from code commit to running production software. Every push triggers a pipeline: build the code, run tests, and deploy — automatically, reproducibly, with full visibility into what happened and why.

This lesson explains the concepts. The next two lessons make them concrete with GitHub Actions and a real Docker pipeline.

## How It Works

### The Problem CI/CD Solves

Without automation, software delivery has predictable failure modes:

- **Integration hell:** Developers work in isolation for days or weeks, then merge everything at once. Conflicts and bugs multiply.
- **"Works on my machine":** The server has different software versions, environment variables, or dependencies than the developer's laptop.
- **Manual steps get skipped:** Under pressure, someone skips the test suite. A bug ships.
- **Infrequent deployments are scary:** If you deploy rarely, each deployment is a large, risky event. Teams avoid deploying, which makes each deployment even larger and riskier.

CI/CD solves this by making integration and deployment routine, automated, and small.

### Continuous Integration (CI)

**Continuous Integration** means every code push to the shared repository triggers an automated build and test run. The system validates the code immediately, before anyone else pulls it.

The key properties:
- **Triggered automatically:** No human remembers to run the tests — the system does it on every push.
- **Runs in a clean environment:** The build runs on a fresh machine (not anyone's laptop), with exactly the dependencies specified in the repository.
- **Fails fast and notifies:** If the build or tests fail, the developer is notified immediately — when the change is still fresh in their mind, not three days later.

CI catches integration bugs in minutes instead of days. When every developer integrates their changes multiple times per day against a known-good baseline, the scope of any conflict is tiny.

### Continuous Delivery vs Continuous Deployment

"CD" is used for two related but distinct practices:

**Continuous Delivery:** The pipeline builds and tests every commit, producing a deployment-ready **artifact** (a Docker image, a compiled binary, a release package). Deployment to production is a manual decision — but the artifact is always ready to deploy. The team can ship at any time with confidence.

**Continuous Deployment:** Goes one step further. If the build and tests pass, the artifact is automatically deployed to production. No human approval step. This requires high confidence in the test suite.

Most organizations practice Continuous Delivery. Continuous Deployment is common for SaaS products with strong test coverage and mature monitoring.

### Pipeline Stages

A pipeline is a sequence of stages. Each stage gates the next — if a stage fails, the pipeline stops and reports the failure. No broken code advances.

```
Push → [Build] → [Test] → [Deploy to Staging] → [Deploy to Prod]
         ↓ fail    ↓ fail
       [Notify]  [Notify]
```

**Build:** Compile the code, install dependencies, create the artifact. For a Node.js app with Docker, this means running `npm ci` and `docker build`. For a Go binary, `go build`. If the build fails, there is nothing to test.

**Test:** Run the automated test suite. Unit tests, integration tests, linting. If tests fail, the artifact is not promoted.

**Deploy to Staging:** Deploy the artifact to a staging environment — a copy of production, but not customer-facing. Run smoke tests against the live staging service.

**Deploy to Production:** Deploy the same artifact to production. Because it has already passed build, test, and staging, confidence is high.

### Artifacts: The Output of a Build

An **artifact** is the output of the Build stage — something concrete that can be stored, versioned, and deployed. Artifacts travel through pipeline stages rather than being rebuilt at each step. This guarantees that exactly what was tested is exactly what gets deployed.

Common artifact types:
| Type | Example | Storage |
|------|---------|---------|
| Container image | `ghcr.io/myorg/myapp:a1b2c3d` | Container registry (ghcr.io, Docker Hub) |
| Compiled binary | `myapp-linux-amd64` | Object storage (S3, GCS) |
| Package | `myapp-1.2.3.tar.gz` | Package registry (npm, PyPI) |
| Test report | `junit.xml` | Pipeline artifact storage |

The Docker image tag includes the commit SHA (`a1b2c3d`) — you can always trace a running container back to the exact commit that produced it.

<Callout type="deep-dive" title="Why sysadmin knowledge matters for CI/CD">
When a CI/CD pipeline "deploys," it is managing services on a machine — starting processes, checking health, handling failures. Everything you learned about systemd, logging, and monitoring applies directly.

A deployment step is essentially: stop the old service, pull the new artifact, start the new service, verify the health endpoint. That is systemd lifecycle management plus Docker image management. The pipeline automates what a sysadmin would do manually. When a deployment fails at 3am and wakes you up, you debug it using the same tools: `journalctl`, `systemctl status`, container logs, health check endpoints.

A pipeline is just an automated sysadmin.
</Callout>

### The Feedback Loop: Why Speed Matters

The value of CI/CD is proportional to how fast the feedback loop is. A pipeline that takes 45 minutes to run is better than no pipeline, but not much. A pipeline that reports results in 3 minutes changes how developers work.

When feedback is fast:
- Developers fix bugs while the change is still in working memory
- Broken builds block everyone, so the team fixes them immediately
- Frequent small deployments become normal — the "scary deployment" problem disappears

Pipeline speed is an engineering investment. Caching dependencies, running tests in parallel, and keeping build images small all exist to keep the feedback loop tight.
