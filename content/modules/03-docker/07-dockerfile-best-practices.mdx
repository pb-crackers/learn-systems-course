## Overview

Your Docker image is 1.1GB and runs as root — that is a production incident waiting to happen. A 1.1GB image takes minutes to pull in CI, costs money in registry storage, and the root user inside means a vulnerability in your app potentially gives an attacker root access to the host.

This lesson teaches the practices that close the gap between "it works on my machine" and "production-ready". You will refactor the progressive app's `Dockerfile.basic` into `Dockerfile.optimized` — taking it from 1.1GB to ~150MB, adding a non-root user, a health check, and a proper `.dockerignore`. Every change is explained mechanically, not just prescribed.

## How It Works

### Multi-Stage Builds

A single-stage Dockerfile installs everything needed to **build** the app and keeps it all in the final image — compilers, dev dependencies, build cache, source code. For a compiled language this is especially wasteful: the C compiler needed to build a binary has no place in the image that runs that binary.

Multi-stage builds introduce **named stages** with `AS <name>`. You can copy specific files from a stage into a later one. The intermediate stages are never in the final image.

```dockerfile
# Stage 1: Build — install and compile
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev
COPY . .
# (build step would go here for TypeScript, webpack, etc.)

# Stage 2: Runtime — only what's needed to run
FROM node:20-alpine
WORKDIR /app
COPY --from=builder /app .  # pull just the output from builder
CMD ["node", "app.js"]
```

The final image starts from a clean `node:20-alpine` base. Only the files explicitly `COPY --from=builder`'d make it in.

<Callout type="warning" title="Use --no-cache for Fair Size Comparisons">
Docker aggressively reuses cached layers from previous builds. When comparing image sizes, always build with `docker build --no-cache`. Without it, Docker may reuse cached layers from earlier builds and you will not get accurate timings or size readings. Also run `docker image prune` to remove dangling images from previous test builds before comparing.
</Callout>

### Layer Caching

Every `RUN`, `COPY`, and `ADD` instruction creates a new layer. Docker caches each layer keyed by the instruction and its inputs. If the inputs haven't changed since the last build, Docker reuses the cached layer — making subsequent builds fast.

**The rule:** Order instructions from least-frequently-changed to most-frequently-changed.

```dockerfile
# WRONG: cache-busting order
COPY . .              # invalidated on every code change
RUN npm install       # re-runs on every code change, even if no new packages

# RIGHT: cache-friendly order
COPY package*.json ./ # only changes when you add/remove packages
RUN npm ci --omit=dev # cached as long as package-lock.json hasn't changed
COPY . .              # invalidated by code changes — but npm install is already cached
```

With the correct order, changing `app.js` only invalidates the final `COPY . .` layer. The `npm ci` layer is reused from cache, saving 10-30 seconds per build.

### .dockerignore

When you run `docker build`, Docker sends the entire **build context** (the directory you specify) to the Docker daemon. If that directory contains `node_modules`, `.git`, test files, and local secrets, they are all transferred — even if your Dockerfile never copies them.

`.dockerignore` works exactly like `.gitignore`, but for the build context:

```
# docker/app/.dockerignore
node_modules        # don't send local node_modules — npm ci installs fresh ones inside
.git               # git history is not needed in the image
.env               # never send secrets to the build context
*.test.js          # test files don't belong in production images
coverage/          # test coverage reports
npm-debug.log
```

### Security — Non-Root USER

By default, Docker containers run as `root` (UID 0). This is convenient but dangerous. If an attacker exploits a vulnerability in your application and escapes the container namespace, they arrive on the host with root privileges.

The fix is straightforward: create a system user and switch to it before running the application.

```dockerfile
# Create a system group and user with no login shell
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

# Transfer ownership of the app directory to the new user
RUN chown -R appuser:appgroup /app

# All subsequent instructions (including CMD) run as appuser
USER appuser
```

Verify your running container is not root:

<Callout type="warning" title="Never Run Production Containers as Root">
Running containers as root is the single most common Docker security mistake. Many container breakout vulnerabilities have minimal impact when the container runs as a non-privileged user — the attacker's blast radius is limited to the container's assigned UID. For any production workload, always add a `USER` instruction.
</Callout>

### HEALTHCHECK Instruction

A Dockerfile `HEALTHCHECK` defines how Docker knows whether the container is actually working — not just that the process is running.

```dockerfile
# Check every 30 seconds; fail after 3 seconds; unhealthy after 3 failures
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1
```

With a `HEALTHCHECK`, `docker ps` shows `(healthy)` or `(unhealthy)` in the STATUS column:

### Minimal Base Images

The base image you choose accounts for the majority of the final image size. Use the smallest image that provides what you need:

| Image | Size | When to Use |
|-------|------|-------------|
| `node:20` | ~1.1GB | Never in production — full Debian with all build tools |
| `node:20-slim` | ~250MB | Debian-based, stripped of build tools — a middle ground |
| `node:20-alpine` | ~140MB | Alpine Linux — smallest option, suitable for most Node.js apps |
| `scratch` | 0B | Starting from nothing — for statically compiled binaries only |

Alpine uses the **musl libc** instead of glibc. Most Node.js apps work fine on Alpine. Some native npm packages that link against glibc may require debugging — if you encounter issues, `node:20-slim` is a safer alternative with minimal size increase.

<Callout type="deep-dive" title="Counting Layers with docker image inspect">
Multi-stage builds reduce both image size and layer count. Each `RUN`, `COPY`, and `ADD` instruction adds a layer. You can count them:

```bash
docker image inspect app:basic --format '{{len .RootFS.Layers}}'
# 8

docker image inspect app:optimized --format '{{len .RootFS.Layers}}'
# 6
```

Fewer layers means faster pulls and pushes. Fewer layers also means fewer cached intermediate states that an attacker could potentially inspect. The RESEARCH notes that multi-stage builds produce a cleaner layer graph — the builder stage layers never appear in the final image.
</Callout>

    answer="The non-root user is a meaningful security control, not cosmetic. Containers share the host's Linux kernel, and container breakout vulnerabilities — exploits that escape the container's namespace isolation — do occur. If the container runs as root (uid=0) and an attacker exploits a vulnerability in your application to execute arbitrary code, they arrive in the host kernel context as root. A non-root user (uid=100) limits the blast radius: the attacker's access on the host is constrained to what that unprivileged UID can do, which is typically nothing useful. Additionally, if your app is exploited but stays inside the container, running as non-root prevents the process from writing to system paths, installing backdoors in system directories, or reading sensitive files owned by root. Many container security scanners and Kubernetes admission controllers reject images that run as root, so it is also a practical deployment requirement."
