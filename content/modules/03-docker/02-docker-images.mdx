## Overview

You need to ship your app — but every developer's machine is different. One runs Node.js 18, another 20. One is on macOS ARM, another on Ubuntu x86. When you deploy, the server has its own environment. The classic "it works on my machine" problem.

Docker images solve this by packaging your application and its entire runtime environment into a single, portable, reproducible artifact. An image is a complete snapshot of a filesystem — OS libraries, runtime, dependencies, application code, and configuration — frozen at build time. When you run a container from an image, you always get exactly the same environment, regardless of the host machine.

This lesson explains how images are built from layers, how to write Dockerfiles, and how the `docker build` process works.

## How It Works

### Image Layers: Copy-on-Write Filesystems

A Docker image is not a single large file — it is a **stack of read-only layers**. Each layer stores only the filesystem changes from the previous layer. This design enables sharing: two images that share a common base (e.g., both start `FROM node:20`) share those layers on disk. Pulling a second `node:20`-based image only downloads the layers that differ.

```
Image Layer Stack (from bottom to top)
──────────────────────────────────────────────────────────────────────

  Layer 4: /app/app.js  (COPY . .)
           ID: sha256:d4e5f6...         ← changes: app source code only

  Layer 3: /app/node_modules/          (RUN npm install)
           ID: sha256:c3d4e5...         ← changes: ~150MB of dependencies

  Layer 2: /app/package.json           (COPY package.json .)
           ID: sha256:b2c3d4...         ← changes: one file

  Layer 1: /usr /bin /lib /etc ...     (FROM node:20)
           ID: sha256:a1b2c3...         ← changes: entire OS + Node.js runtime (~1.1GB)

  Each layer stores only its diff from the layer below.
  Layers are content-addressed by their SHA256 hash.
```

### Dockerfile Instructions: What Each One Does

A Dockerfile is a text file containing a sequence of instructions. Each instruction either creates a new layer (if it modifies the filesystem) or sets image metadata (no new layer).

**Instructions that create layers:**

| Instruction | What It Does | Layer Created |
|-------------|-------------|--------------|
| `FROM` | Sets the base image | Imports all base image layers |
| `RUN` | Executes a shell command | Stores filesystem changes from the command |
| `COPY` | Copies files from build context | Stores the copied files |
| `ADD` | Like COPY but also handles URLs and tar archives | Stores the added files |

**Instructions that set metadata (no new layer):**

| Instruction | What It Does |
|-------------|-------------|
| `WORKDIR` | Sets working directory for subsequent instructions |
| `ENV` | Sets environment variables in the container |
| `EXPOSE` | Documents which port the app uses |
| `CMD` | Default command to run when the container starts |
| `ENTRYPOINT` | Sets the executable; CMD provides default arguments |

<Callout type="warning" title="EXPOSE Does NOT Publish Ports">
`EXPOSE 3000` in a Dockerfile is **documentation only**. It tells Docker (and humans) that the app inside listens on port 3000, but it does not make the port accessible from the host. To publish a port, you use the `-p` flag at run time:

```
docker run -p 3000:3000 myapp:v1
         ↑host port ↑container port
```

This is one of the most common Docker confusions. You can run `docker run myapp:v1` without `-p` and the container starts fine — but nothing on the host can reach port 3000.
</Callout>

Here is the progressive app's `Dockerfile.basic` with each instruction explained:

```dockerfile filename="docker/app/Dockerfile.basic"
# FROM sets the base image — every subsequent instruction builds on top of this
FROM node:20

# WORKDIR creates the directory if it doesn't exist and sets it as the
# working directory for all subsequent COPY, RUN, CMD instructions
WORKDIR /app

# COPY package.json FIRST — before copying app source
# This is a layer caching strategy: if package.json hasn't changed,
# Docker reuses the cached RUN npm install layer
COPY package.json .

# RUN executes inside the build container — creates a layer with node_modules
RUN npm install

# COPY the rest of the source code
# This is a separate layer from package.json — so changing app.js only
# invalidates this layer and not the npm install layer above
COPY . .

# EXPOSE is documentation — it does not publish the port
EXPOSE 3000

# CMD is the default command — uses exec form (JSON array) to avoid
# spawning a shell, so node receives signals (SIGTERM) directly
CMD ["node", "app.js"]
```

**Layer cache invalidation rules:**
1. If an instruction's content changes, that layer and ALL subsequent layers are rebuilt
2. `COPY` and `ADD` compare file checksums — any change in the copied files busts the cache
3. `RUN` instructions only rebuild if their command string changes or a previous layer changed

This is why `COPY package.json .` comes before `COPY . .` — `package.json` changes rarely, so `npm install` is cached most of the time. If you copied everything first (`COPY . .`) and then ran `npm install`, every code change would re-run the full `npm install`.

**Build context** is the directory you pass to `docker build` (the last argument). Docker sends the entire build context to the build engine. Large build contexts slow down builds. Use `.dockerignore` to exclude `node_modules`, `.git`, and other large directories.

<Callout type="deep-dive" title="The .dockerignore File">
`docker/app/.dockerignore` prevents files from being included in the build context. This matters for two reasons:

1. **Speed**: Docker must read every file in the build context. A `node_modules` directory with 10,000 files takes significant time to scan even if you COPY it explicitly.

2. **Correctness**: If you don't exclude `node_modules` from the context, and you do `COPY . .`, you might inadvertently copy your LOCAL `node_modules` into the image instead of running a clean `npm install`. Your local modules may differ from what you get with a fresh install (platform-specific binaries, different package lock).

The `.dockerignore` syntax is identical to `.gitignore`. Always include at minimum: `node_modules`, `.git`, `.env`.
</Callout>

### Registries: Docker Hub and Tagging

Images are stored in and pulled from **registries**. Docker Hub (`docker.io`) is the default public registry. Images are identified by a naming convention:

```
registry/repository:tag
──────── ──────────────── ────
│        │               └─ version label (default: "latest")
│        └─ image name (e.g., "node", "ubuntu", "myapp")
└─ registry hostname (default: docker.io)

Examples:
  node:20                → docker.io/library/node:20
  ubuntu:22.04           → docker.io/library/ubuntu:22.04
  myapp:v1               → docker.io/<your-username>/myapp:v1
  registry.example.com/myapp:v1  → private registry
```
