## Overview

Your containers need to talk to each other — but you cannot hardcode IP addresses. Container IPs are assigned dynamically by Docker and change every time a container is recreated. If your API container is at `172.18.0.3` today, it might be `172.18.0.4` tomorrow. Hardcoding IPs breaks everything.

Docker networking solves this with a custom DNS system: containers on the same network can resolve each other by **service name**. When your web container needs to call the API, it connects to `http://api:3000` — and Docker's embedded DNS server resolves `api` to the current container's IP automatically.

This lesson bridges directly to the networking fundamentals from Phase 2. The same concepts — bridges, DNS resolution, IP routing — apply here. Docker just handles the plumbing.

## How It Works

### Docker Network Drivers

Docker supports several network drivers, each with a different isolation model:

| Driver | Description | Use Case |
|--------|-------------|----------|
| `bridge` | Software bridge on the host, containers get private IPs | Default for single-host container communication |
| `host` | Container shares the host's network namespace, no isolation | Performance-critical apps where network overhead matters |
| `none` | No network interface (except loopback) | Maximum isolation — container cannot reach anything |
| `overlay` | Multi-host networking for Docker Swarm clusters | Distributed deployments across multiple Docker hosts |

For single-host development and most production deployments, `bridge` is what you will use. The `host` driver is occasionally useful for performance-critical services but sacrifices isolation.

<Callout type="deep-dive" title="Bridge Networks Are Virtual Linux Switches">
In the networking module, you learned that a switch forwards Ethernet frames based on a MAC address table. When Docker creates a bridge network, it creates a **Linux bridge interface** on the host — effectively a virtual switch running in the kernel.

Run `ip link show` on your host after creating a Docker network and you will see entries like `docker0` (the default bridge) and `br-xxxxxxxx` (custom bridges). These ARE the virtual switches for your containers. Each container gets a virtual Ethernet pair (`veth`): one end lives in the container, the other connects to the bridge on the host. The bridge routes frames between containers exactly as a physical switch would route frames between hosts.
</Callout>

### Default Bridge vs Custom Bridge

Docker provides a default bridge network named `bridge` that all containers join unless you specify otherwise. **The default bridge has a critical limitation: containers cannot resolve each other by name.** They must use IP addresses.

Custom bridge networks enable **container name DNS resolution**. This single difference makes custom bridges necessary for multi-container applications.

```
Default Bridge Network (docker0)
┌──────────────────────────────────────────────┐
│  container: app1             container: app2  │
│  IP: 172.17.0.2              IP: 172.17.0.3   │
│  ─────────────────────────────────────────── │
│  No DNS: app2 cannot be resolved by name      │
│  Must use: curl http://172.17.0.3:3000        │
└──────────────────────────────────────────────┘

Custom Bridge Network (br-abc123)
┌──────────────────────────────────────────────┐
│  container: api              container: web   │
│  IP: 172.18.0.2              IP: 172.18.0.3   │
│  ─────────────────────────────────────────── │
│  DNS enabled: web resolves "api" → 172.18.0.2 │
│  Works: curl http://api:3000/health           │
└──────────────────────────────────────────────┘
```

### Container DNS Resolution

Docker embeds a DNS server inside each custom bridge network at the fixed IP `127.0.0.11`. When a container on a custom network performs a DNS lookup (e.g., `curl http://api:3000`), the lookup goes to `127.0.0.11` first. Docker's DNS responds with the current IP of the container named `api`.

In the DNS lesson from Phase 3, you learned about resolution chains — stub resolvers, recursive resolvers, authoritative servers. Docker's embedded DNS is a **local resolver** sitting at `127.0.0.11`. It is authoritative for container names on the same network and forwards everything else upstream.

### Port Publishing

Containers on a bridge network have private IPs not accessible from the host. To expose a container's port to the host (and the outside world), you must **publish** it with `-p`:

```bash
# Map host port 8080 to container port 3000
docker run -p 8080:3000 myimage

# Map to a specific host IP (useful when host has multiple interfaces)
docker run -p 127.0.0.1:8080:3000 myimage

# Assign a random host port
docker run -P myimage
```

`EXPOSE` in a Dockerfile is documentation only. It does not publish any port. The `-p` flag at `docker run` time is what actually opens the host port.

<Callout type="tip" title="Use -P for Quick Local Testing">
`docker run -P myimage` publishes all EXPOSEd ports to random high-numbered host ports. Run `docker ps` to see the assigned ports. Useful for quickly spinning up a container when you do not care which host port it lands on.
</Callout>

### Network Isolation

Containers on different networks cannot communicate, even if they are on the same Docker host. This is the primary security mechanism for multi-service applications — your database container should not be reachable by anyone except your API container.

```
Network Isolation
┌──────────────────────┐    ┌──────────────────────┐
│  frontend-net        │    │  backend-net          │
│  ┌──────┐ ┌──────┐   │    │  ┌──────┐ ┌──────┐   │
│  │ web  │ │ api  │   │    │  │ api  │ │ db   │   │
│  └──────┘ └──────┘   │    │  └──────┘ └──────┘   │
│  web can reach api   │    │  api can reach db     │
└──────────────────────┘    └──────────────────────┘
                  web cannot reach db directly
```

A container can be connected to **multiple networks simultaneously**. In the diagram above, the API container is on both `frontend-net` (to receive requests from `web`) and `backend-net` (to query `db`). This is the standard multi-tier architecture pattern in Docker Compose.
