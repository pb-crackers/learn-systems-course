## Overview

Your container crashed at 3am — what happened? Without understanding the container lifecycle, you are left staring at an empty terminal. With it, you know exactly where to look: `docker logs` shows what the process printed before dying, `docker inspect` shows the exit code and when it stopped, and `docker exec` lets you walk through a running container's environment before the crash repeats.

This lesson covers the full container lifecycle (created → running → stopped → removed), interactive debugging with `docker exec`, log inspection with `docker logs`, and resource limits via `--memory` and `--cpus`. The resource limits section connects back to the cgroups mechanism from the previous lesson.

## How It Works

### Container Lifecycle: Five States

Every container passes through defined states. Understanding these states tells you what commands are available and what is happening to the process.

```
Container Lifecycle
────────────────────────────────────────────────────────────────────

  ┌─────────┐  docker start     ┌─────────┐  docker pause   ┌────────┐
  │ Created │ ────────────────► │ Running │ ──────────────► │ Paused │
  └─────────┘                  └────┬────┘ ◄─────────────── └────────┘
       ▲                            │         docker unpause
       │  docker create             │ docker stop
       │                            │ (SIGTERM → SIGKILL after 10s)
       │                            ▼
  ┌────┴────┐  docker rm       ┌─────────┐
  │ Removed │ ◄──────────────  │ Stopped │
  └─────────┘                  └─────────┘
                                    │
                                    └─ docker start  (restart without recreate)
```

**Key transition details:**

- `docker create` prepares the container (sets up namespaces, overlay FS) but does not start the process
- `docker start` starts the process (the CMD or ENTRYPOINT from the image)
- `docker run` = `docker create` + `docker start` in one command
- `docker stop` sends **SIGTERM** to PID 1 inside the container, then waits 10 seconds; if the process hasn't exited, it sends **SIGKILL**
- `docker kill` sends SIGKILL immediately (no grace period)
- `docker pause` uses the `SIGSTOP` cgroup freezer — process is suspended, not terminated
- `docker rm` removes a stopped container — the writable overlay layer is discarded

<Callout type="tip" title="Graceful Shutdown: Why It Matters">
When your application handles SIGTERM correctly, it can finish in-flight requests, flush write buffers, and release database connections before exiting. This is critical for zero-downtime deployments.

For SIGTERM to reach your application, it must be PID 1 in the container. If you use `CMD ["npm", "start"]`, npm is PID 1 and it may not pass signals to the Node.js process it spawned. Use `CMD ["node", "app.js"]` — node becomes PID 1 and receives signals directly.

To measure the shutdown timing difference: `time docker stop <container>` with `CMD ["node", "app.js"]` vs `CMD ["npm", "start"]`. The npm version will always wait the full 10 second grace period before SIGKILL.
</Callout>

The `--rm` flag automatically removes the container when it exits — useful for one-off commands:

```bash
docker run --rm node:20 node -e "console.log('Node version: ' + process.version)"
```

### Interactive Access: docker exec

`docker exec` runs a new process inside a running container's namespaces. This is the primary debugging tool — get a shell inside a container to inspect its filesystem, environment variables, running processes, and network configuration.

<Callout type="warning" title="docker exec vs docker attach: Important Difference">
`docker exec` starts a NEW process inside the container's namespaces. When you exit the exec shell, the container continues running.

`docker attach` connects your terminal to the container's **PID 1** stdin/stdout. If you press Ctrl+C while attached, you send SIGINT to PID 1 — this KILLS the container's main process, stopping the container. Use `Ctrl+P Ctrl+Q` to detach without stopping.

For debugging, always prefer `docker exec -it <name> /bin/sh`. Reserve `docker attach` for the rare case where you specifically need to interact with PID 1's input stream.
</Callout>

You can also run non-interactive commands with `docker exec` (without `-it`):

```bash
# Check what files exist in /app without opening a shell
docker exec debug-app ls -la /app

# Read an environment variable
docker exec debug-app printenv NODE_ENV

# Run a health check command
docker exec debug-app node -e "require('./app.js')" 2>&1 | head -5
```

`docker inspect` returns full JSON metadata about a container. Key fields for debugging:

| Field | What It Tells You |
|-------|------------------|
| `.State.Status` | Current state: running, exited, paused |
| `.State.ExitCode` | Exit code of PID 1 (0 = clean, 137 = SIGKILL, 1 = error) |
| `.State.StartedAt` | When the container started |
| `.State.FinishedAt` | When it stopped (and why, combined with ExitCode) |
| `.NetworkSettings.IPAddress` | Container's IP on the Docker bridge network |
| `.Mounts` | Volumes or bind mounts attached to the container |
| `.HostConfig.Memory` | Memory limit in bytes (0 = unlimited) |

### Resource Limits: Connecting Back to cgroups

In the previous lesson (What Are Containers), you learned that cgroups are the kernel mechanism Docker uses for resource limits. The `--memory` and `--cpus` flags on `docker run` directly write values to the cgroup files you saw at `/sys/fs/cgroup/`.

<Callout type="deep-dive" title="What --memory and --cpus Write to cgroups">
When you run `docker run --memory=128m --cpus=0.5`, Docker:
1. Creates a cgroup at `/sys/fs/cgroup/system.slice/docker-<id>.scope/`
2. Writes `134217728` (128 * 1024 * 1024 bytes) to `memory.max`
3. Writes `50000 100000` to `cpu.max` (50,000 microseconds per 100,000 microsecond period = 50% of one CPU)

When the container process exceeds `memory.max`, the kernel OOM killer fires and terminates the process. The container stops with exit code 137 (SIGKILL). This is why memory-constrained containers can die suddenly without any application-level error message.
</Callout>
