## Overview

Your team lead says "just Docker it" — but what IS a container? The answer matters more than it seems. Teams that treat containers as magic black boxes hit confusing problems: processes that can't find their files, ports that appear open but aren't reachable, memory limits that kill processes without warning. Teams that understand what containers are debug these problems in minutes.

Here is the key fact: **a container is not a lightweight VM**. There is no guest OS, no hypervisor, no hardware emulation. A container is an isolated Linux process — or group of processes — using three kernel features you already learned about: namespaces (for isolation), cgroups (for resource limits), and overlay filesystems (for the image layer system). This lesson traces each feature to its Linux foundation.

## How It Works

### Linux Namespaces: Process-Level Isolation

A **namespace** wraps a global Linux resource so that processes inside the namespace see their own isolated instance of that resource. When Docker creates a container, it calls the `clone()` system call with namespace flags — the new process starts in fresh namespaces, isolated from everything outside.

Linux provides six namespaces Docker uses for containers:

| Namespace | Flag | What It Isolates |
|-----------|------|-----------------|
| PID | `CLONE_NEWPID` | Process IDs — container thinks its first process is PID 1 |
| NET | `CLONE_NEWNET` | Network interfaces, routing tables, iptables rules |
| MNT | `CLONE_NEWNS` | Filesystem mount points — container sees its own tree |
| UTS | `CLONE_NEWUTS` | Hostname and NIS domain name |
| IPC | `CLONE_NEWIPC` | System V IPC, POSIX message queues |
| USER | `CLONE_NEWUSER` | User and group IDs — map container root to non-root on host |

<Callout type="deep-dive" title="Remember PID Namespaces from Lesson 5?">
In Linux Fundamentals Lesson 5 (Processes), you learned that each process has a PID visible in `/proc`. When Docker creates a container, it calls `clone()` with the `CLONE_NEWPID` flag — the container gets its own PID namespace. The first process inside the container is assigned PID 1. From the host's perspective, the same process has a completely different PID (whatever the kernel assigned in the host's PID namespace).

This is why `docker inspect mybox --format "{{.State.Pid}}"` returns something like `12345` — that is the PID you will find in `/proc/12345/ns/` on the HOST. Inside the container, that same process believes it is PID 1.
</Callout>

You can see namespace links directly in `/proc`. Every namespace is represented as a symbolic link in `/proc/<PID>/ns/`:

The different inode numbers (e.g., `net:[4026532346]` vs `net:[4026531992]`) prove the container is using a different namespace instance. The container has its own network — its own interfaces, routing table, and iptables rules — completely separate from the host.

<Callout type="tip" title="Networking Module Bridge to Docker">
In the Networking module you learned about network interfaces — each device has its own interfaces and routing table. Each container gets its own NET namespace with its own virtual ethernet interface (typically named `eth0` inside the container). Docker creates a `veth` pair — one end inside the container namespace, one end attached to the `docker0` bridge on the host. When you run `ip link show` on the host after starting a container, you will see `docker0` and a `veth...` interface. That IS the container's network plumbing.
</Callout>

### Control Groups (cgroups): Resource Limits

Namespaces provide isolation — but an isolated process could still consume all available CPU or memory, starving other processes. **Control Groups (cgroups)** are the kernel mechanism that limits, accounts for, and isolates resource usage of process groups.

Docker uses cgroups to enforce resource limits set by `--memory` and `--cpus` flags. Internally, Docker writes values to files in the cgroup filesystem at `/sys/fs/cgroup/`.

**cgroup v2 hierarchy** (modern Linux — the default since kernel 5.x):

```
/sys/fs/cgroup/
├── system.slice/                    ← systemd services
├── user.slice/                      ← user sessions
└── docker/                          ← all Docker containers
    └── a1b2c3d4e5f6.../             ← per-container cgroup
        ├── memory.max               ← max memory in bytes
        ├── cpu.max                  ← CPU quota and period
        ├── io.max                   ← I/O limits per device
        └── pids.max                 ← max number of processes
```

Each file is a text file you can read (and, as root, write). Docker writes to these files when you use `--memory` or `--cpus` flags. When the container process exceeds `memory.max`, the kernel OOM killer terminates it — that is why containers can be killed without warning when they exceed their memory limit.

<Callout type="tip" title="cgroup v1 vs v2">
Older systems use cgroup v1, which has a different hierarchy (separate directories per subsystem under `/sys/fs/cgroup/memory/`, `/sys/fs/cgroup/cpu/`, etc.). Modern Linux distributions (Ubuntu 22.04+, Fedora 31+) use cgroup v2, which unifies all resource controllers under a single hierarchy. Docker 20.10+ supports both. The concepts are identical — only the file paths differ.
</Callout>

### Union/Overlay Filesystem: Image Layers

The third kernel feature containers rely on is the **overlay filesystem** (overlayfs). This is what makes Docker image layers possible.

<Callout type="deep-dive" title="Remember Filesystems from Lesson 3?">
In Linux Fundamentals Lesson 3 (Linux Filesystem), you learned about mount points — how a filesystem is attached at a directory, and how `/proc` and `/sys` are special virtual filesystems. The overlay filesystem is another special filesystem, but its purpose is to merge multiple directory trees into a single unified view.
</Callout>

The overlay filesystem has three components:

```
Overlay FS Structure
─────────────────────────────────────────────────────────────────────

  upper (writable layer)       ← container-specific changes
  ┌──────────────────────────────────────────────────────────────┐
  │  /app/new-file.txt   (created inside container)              │
  │  /etc/hosts          (modified — whiteout of lower layer)    │
  └──────────────────────────────────────────────────────────────┘
           ↕  merged view (what the container sees)
  lower (read-only layers)     ← image layers, stacked
  ┌──────────────────────────────────────────────────────────────┐
  │  Layer 3: /app/app.js, /app/package.json (COPY instruction)  │
  │  Layer 2: /app/node_modules/ (RUN npm install instruction)   │
  │  Layer 1: /usr, /bin, /etc ... (FROM node:20 base image)     │
  └──────────────────────────────────────────────────────────────┘
           ↕  workdir (internal — tracks changes to upper)
  work directory (internal)    ← kernel bookkeeping
```

**Copy-on-write**: When a container modifies a file that exists in a lower layer, the overlay filesystem copies that file to the upper layer first, then modifies the copy. The lower layers are never changed. This is why multiple containers can share the same image layers without interfering with each other.

When a container is deleted, its upper (writable) layer is discarded. The lower layers remain — they are the image, and they are shared by every container based on that image.

### Putting It Together: What `docker run` Actually Does

When you run `docker run -d ubuntu:22.04 sleep infinity`, here is the sequence:

```
docker run sequence
────────────────────────────────────────────────────────────────────

  1. Docker daemon prepares the overlay filesystem
     - Stacks image layers as lower (read-only) directories
     - Creates a new empty upper (writable) directory for this container

  2. Docker daemon calls clone() with namespace flags:
     CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWNS |
     CLONE_NEWUTS | CLONE_NEWIPC | CLONE_NEWUSER

  3. Inside the new namespaces, Docker:
     - Mounts the overlay filesystem as / (the container's root)
     - Sets up /proc, /sys, /dev virtual filesystems
     - Configures the network namespace (veth pair, IP address)
     - Sets hostname in the UTS namespace

  4. Docker writes resource limits to /sys/fs/cgroup/docker/<id>/:
     - memory.max, cpu.max (if --memory or --cpus were specified)

  5. Docker exec()'s the container command (sleep infinity)
     - This process becomes PID 1 in the container's PID namespace
     - From the host's view, it has a normal PID (e.g., 12345)
```

The container process is a regular Linux process. It has a PID on the host. It is visible in `ps` output from the host. The only difference from a non-containerized process is: it sees isolated namespaces, it is subject to cgroup limits, and its root filesystem is an overlay mount.
