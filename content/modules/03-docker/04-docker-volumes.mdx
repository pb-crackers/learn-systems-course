## Overview

Your database container restarted and all the data is gone. This is the most jarring lesson new Docker users learn the hard way — container filesystems are ephemeral. Everything written inside a container lives in the container's writable layer, and when the container is removed, that layer disappears with it.

This lesson explains the three ways Docker handles data: the temporary writable layer (why data disappears), bind mounts (for development workflows), and named volumes (for production persistence). You will also learn when to choose each strategy and how they connect to the overlay filesystem you encountered in the container internals lesson.

## How It Works

### Container Filesystem Ephemerality

When Docker creates a container, it builds a layered filesystem using the overlay FS driver — the same Linux kernel feature covered in the container internals lesson. The image layers are read-only. Docker adds a thin writable layer on top for the running container.

Any file you create, modify, or delete inside a container lives in this writable layer:

```
Overlay FS: Container Filesystem Stack
┌────────────────────────────────┐
│  Container Writable Layer      │  ← ephemeral: deleted with the container
│  (your /app/data, logs, etc.)  │
├────────────────────────────────┤
│  Image Layer 3 (COPY app.js)   │  ← read-only
├────────────────────────────────┤
│  Image Layer 2 (RUN npm ci)    │  ← read-only
├────────────────────────────────┤
│  Image Layer 1 (FROM node:20)  │  ← read-only
└────────────────────────────────┘
```

When you run `docker rm` (or `docker run --rm`), the writable layer is discarded. The next container created from the same image starts with a clean writable layer. This is intentional — it is what makes containers reproducible — but it means you cannot store state in the container itself.

### Bind Mounts

A bind mount connects a directory on the **host filesystem** directly into the container. The host path and the container path point to the same underlying storage — changes made from either side are immediately visible to the other.

```bash
# Syntax: -v /host/absolute/path:/container/path
docker run -v /home/user/myapp:/app myimage
```

This is what makes bind mounts ideal for **development workflows**: you edit source files on your host with your editor, and the running container immediately sees the changes without a rebuild.

<Callout type="warning" title="Bind Mounts Give the Container Full Host Path Access">
When you bind-mount a directory, the container process has the same access to that path as the user who started Docker. If you mount `/home/user` and the container runs as root, it can read and write everything in your home directory. Keep bind mounts scoped to project directories, never to sensitive paths like `/`, `/etc`, or your SSH keys directory.
</Callout>

### Named Volumes

Named volumes are Docker-managed storage objects. Instead of pointing to a host path, you give the volume a name and Docker handles where the data lives. On Linux, named volumes are typically stored under `/var/lib/docker/volumes/<name>/_data`.

```bash
# Create a named volume
docker volume create mydata

# Mount a named volume (Docker creates it automatically if it doesn't exist)
docker run -v mydata:/app/data myimage
```

The key distinction: the data in a named volume **outlives the container**. You can remove and recreate the container, and the same volume mounts with all the data intact.

### Bind Mount vs Named Volume: Choosing the Right Strategy

| Property | Bind Mount | Named Volume |
|----------|-----------|--------------|
| Storage location | Host path you specify | Docker-managed (e.g., `/var/lib/docker/volumes/`) |
| Portability | Not portable — path must exist on host | Portable — works on any Docker host |
| Development workflow | Excellent — edit on host, run in container | Poor — data not directly accessible from host IDE |
| Backup | Manual — backup the host path | Manual — backup volume or use `docker cp` |
| Performance (macOS/Windows) | Slower due to filesystem translation | Faster — less cross-boundary translation |
| Use case | Source code during development | Database files, uploads, persistent state |
| Security | Container has full access to host path | Scoped to Docker's storage — more isolated |

**Rule of thumb:**
- Development: bind mount your source code directory
- Production data (databases, file uploads, logs): named volumes

### tmpfs Mounts

A third option exists for sensitive data that should never touch disk. `tmpfs` mounts store data in the host's memory only — they disappear when the container stops and are never written to disk. Useful for secrets, session tokens, or temporary credentials.

```bash
docker run --tmpfs /app/secrets:rw,size=64m myimage
```

<Callout type="deep-dive" title="How Overlay FS Handles the Writable Layer">
Docker's overlay filesystem uses two key directories: a `lowerdir` (the read-only image layers) and an `upperdir` (the writable container layer). When a container writes to a file that exists in a lower layer, the overlay driver performs a **copy-on-write**: it copies the file from the lower layer into the upper layer, then modifies the copy. The original in the lower layer is untouched. When the container is removed, only the `upperdir` is deleted — the image layers (lowerdir) remain intact for future containers.

Named volumes and bind mounts **bypass this mechanism entirely**. Their data goes directly to the host filesystem, not through the overlay driver. That is why they persist across container deletions — the data was never in the writable layer to begin with.
</Callout>

That layer is the container's private scratchpad, and anything written there (files created or modified during the container's lifetime that are not in a volume) disappears. The named volume is a separate Docker object stored at /var/lib/docker/volumes/appstate/_data on the host. It is mounted into the container at /app/data using a bind at the filesystem level — the data goes directly to that host path, bypassing the overlay driver's writable layer entirely. Because the volume is a separate object with its own lifecycle, docker rm does not touch it. Only docker volume rm or docker compose down -v explicitly removes named volumes."
