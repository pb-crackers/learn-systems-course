## The Challenge

You are a new DevOps engineer. Your first task: take a three-tier application — web frontend,
API backend, and PostgreSQL database — and make it runnable anywhere with a single command.

The application source code lives in `docker/capstone/`. Read `docker/capstone/README.md` for the
full project brief including the API contract, architecture, and success criteria.

## Architecture

The stack has three services. Each communicates with the next tier; none of the backend services
are directly accessible from outside the Compose network.

```
Browser
   │
   │ HTTP :8080
   ▼
┌────────────────────────────────┐
│  web  (nginx)                  │  Serves index.html, reverse-proxies /api/* to api
│  host port 8080 → container 80 │
└──────────────┬─────────────────┘
               │ HTTP :3000  (internal network only)
               ▼
┌────────────────────────────────┐
│  api  (Node.js Express)        │  GET /health, GET /api/items, POST /api/items
│  port 3000 (not published)     │
└──────────────┬─────────────────┘
               │ TCP :5432   (internal network only)
               ▼
┌────────────────────────────────┐
│  db  (PostgreSQL 16)           │  Items table; data in a named volume
│  port 5432 (not published)     │
└────────────────────────────────┘
```

The web service is the only one reachable from the host. It serves the static frontend and
acts as a reverse proxy for API requests. The API talks to the database using PostgreSQL's
default port. The database stores data in a named volume so it persists across restarts.

## Getting Started

1. Read `docker/capstone/README.md` — it has the full architecture, API contract, and requirements.
2. Read `docker/capstone/api/app.js` and `docker/capstone/web/index.html` — understand what each service does before writing the Dockerfiles.
3. A sample nginx configuration is at `docker/capstone/web/nginx.conf` — it shows the reverse proxy pattern you need.
4. Write the files in this order: `api/Dockerfile` → `web/Dockerfile` → `compose.yml` → `deploy.sh`
5. Run `bash docker/capstone/verify.sh` when you think everything is working.

You do NOT need to modify `api/app.js`, `web/index.html`, or `web/nginx.conf`. Those are provided.

## What You Are Practicing

<Callout type="deep-dive" title="Skills from Every Module So Far">
This capstone deliberately spans all three foundation modules:

**Linux Fundamentals**
- Shell scripting in `deploy.sh` — conditionals, exit codes, set -e, health checks
- File permissions — `chmod +x deploy.sh` before it can run
- Process awareness — PID 1 in containers, signal handling with `CMD ["node", "app.js"]`

**Networking Foundations**
- Port publishing — host port 8080 mapped to container port 80
- DNS resolution — `api:3000` resolved by Docker's embedded DNS on a custom bridge network
- Reverse proxying — nginx forwards `/api/*` to the API service by service name
- Network isolation — api and db not published to host

**Docker and Containerization**
- Dockerfiles — multi-stage build for web, non-root user for api
- Docker Compose — service definitions, networks, volumes, depends_on
- Named volumes — data persistence across container restarts
- Dockerfile best practices — layer caching, .dockerignore, HEALTHCHECK
</Callout>
