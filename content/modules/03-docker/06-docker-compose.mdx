## Overview

Running five `docker run` commands with the right flags every time is a recipe for errors. One missed `--network`, one wrong environment variable, one forgotten `-v` flag — and the application behaves differently than it did yesterday, and nobody knows why.

Docker Compose solves this by capturing the entire multi-service configuration declaratively in a `compose.yml` file. The file is version-controlled alongside your application code. `docker compose up` starts everything; `docker compose down` stops and removes it. The configuration is the single source of truth.

This lesson builds on what you learned about volumes (lesson 4) and networking (lesson 5). Compose is not a new technology — it is a declarative interface over the same `docker run`, `docker network create`, and `docker volume create` commands you already know.

## How It Works

### What Is Docker Compose

Docker Compose reads a `compose.yml` file and translates it into Docker API calls. Everything in `compose.yml` maps directly to flags you have already used on the `docker run` command line.

<Callout type="tip" title="compose.yml vs docker-compose.yml">
The preferred filename for Compose v2 is `compose.yml` (no hyphen, no `docker-` prefix). Older projects use `docker-compose.yml` — Docker still recognizes it, but `compose.yml` is the current standard. This course uses `compose.yml` throughout.
</Callout>

### Service Definition

Each entry in the `services:` block is one container. The keys under a service correspond directly to `docker run` flags:

```yaml
# compose.yml
services:
  api:
    build: ./docker/app          # docker build ./docker/app
    image: myapp:v1              # or use an existing image instead of build
    ports:
      - "3000:3000"              # -p 3000:3000
    environment:
      - NODE_ENV=production      # -e NODE_ENV=production
      - PORT=3000
    volumes:
      - appdata:/app/data        # -v appdata:/app/data
    networks:
      - backend                  # --network backend
    restart: unless-stopped      # --restart unless-stopped

  nginx:
    image: nginx:alpine
    ports:
      - "80:80"
    networks:
      - backend

networks:
  backend:                       # docker network create <project>_backend

volumes:
  appdata:                       # docker volume create <project>_appdata
```

The `services:`, `networks:`, and `volumes:` blocks at the top level of `compose.yml` each manage a different Docker object type. Service-level `volumes:` and `networks:` keys reference the top-level declarations.

### depends_on and Healthchecks

`depends_on` controls the **start order** of services. Without it, Compose starts all services simultaneously — which can cause an API container to crash because its database is not yet accepting connections.

**Critical distinction:** `depends_on` with just a service name only waits for the container to **start**, not for it to be **ready**. The database process inside the container might still be initializing when the API tries to connect.

To wait for actual readiness, combine `depends_on` with a `condition` and a `healthcheck`:

```yaml
services:
  api:
    build: ./docker/app
    depends_on:
      db:
        condition: service_healthy   # waits until db is healthy
    ports:
      - "3000:3000"

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD: secret
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s       # run healthcheck every 10 seconds
      timeout: 5s         # consider failed after 5 seconds
      retries: 5          # mark unhealthy after 5 consecutive failures
      start_period: 30s   # grace period before first healthcheck
```

`docker compose ps` shows the health status. A service is `(healthy)` only after the healthcheck command returns exit code 0.

<Callout type="warning" title="depends_on Without condition Does Not Wait for Readiness">
`depends_on: db` (without a `condition`) only waits for the `db` container to START — the Docker daemon started the container process. It does not wait for PostgreSQL, MySQL, or any service inside the container to be ready to accept connections. Always use `condition: service_healthy` with a `healthcheck` when your service needs a dependency to be functionally ready before starting.
</Callout>

### Environment Variables

Compose offers three ways to provide environment variables to services:

**Inline `environment:` block** — values in `compose.yml` directly:
```yaml
environment:
  - NODE_ENV=production
  - DATABASE_URL=postgres://localhost/mydb
```

**`env_file:` reference** — read from a separate file (one `KEY=VALUE` per line):
```yaml
env_file:
  - .env.production
```

**Variable substitution** — reference shell environment or `.env` file in the project directory:
```yaml
environment:
  - DATABASE_URL=${DATABASE_URL:-postgres://localhost/mydb}
```

<Callout type="warning" title=".env File Is Not the Same as env_file">
Docker Compose automatically loads a file named `.env` in the same directory as `compose.yml` for **variable substitution** (the `${VAR}` syntax). The `env_file:` key in a service definition loads a file directly as environment variables **inside the container**. These are different mechanisms. A value in `.env` is used to expand `${VAR}` in `compose.yml` but is not automatically injected into the container unless you also reference it in the `environment:` block.
</Callout>

### Named Volumes in Compose

Named volumes in Compose require two declarations: the top-level `volumes:` block registers the volume, and each service's `volumes:` key mounts it.

```yaml
services:
  api:
    volumes:
      - appdata:/app/data    # mount the named volume

volumes:
  appdata:                   # declare it — Compose creates it on first up
```

The **data persistence behavior** across Compose lifecycle commands is critical to understand:

| Command | Containers | Networks | Named Volumes |
|---------|-----------|----------|---------------|
| `docker compose down` | Removed | Removed | **Preserved** |
| `docker compose down -v` | Removed | Removed | **Removed** |
| `docker compose stop` | Stopped (not removed) | Preserved | Preserved |
| `docker compose restart` | Restarted | Preserved | Preserved |

Data in named volumes survives `docker compose down` — you can bring the stack back up and pick up where you left off. Use `docker compose down -v` only when you want a completely clean slate.

### Profiles

Profiles let you group optional services that are not started by default. This is useful for debug utilities, admin tools, or test harnesses that you do not want running in normal operation.

```yaml
services:
  api:
    build: ./docker/app           # always started

  debug-shell:
    image: node:20-alpine
    profiles:
      - debug                     # only started when --profile debug is used
    command: sh
    stdin_open: true
    tty: true
```

```bash
# Normal start — debug-shell is excluded
docker compose up -d

# Start with debug profile — debug-shell is included
docker compose --profile debug up -d
```
