## Overview

Your new service is deployed and running, but external users can't reach it. The application logs show no incoming requests at all. A firewall rule is silently dropping the traffic before it ever reaches your application — you need to know how to find and fix it. Firewalls are the traffic control systems for your network: they examine packets and decide whether to allow or block them based on rules you define.

This lesson explains how the Linux kernel processes packets through netfilter, how iptables rules work, how ufw simplifies day-to-day firewall management, and how cloud security groups map to the same underlying concepts.

## How It Works

### Netfilter: The Linux Firewall Framework

**Netfilter** is the packet filtering framework built into the Linux kernel. `iptables` is the userspace command-line tool that configures netfilter rules. Every packet that enters or exits a Linux system passes through a series of netfilter **hooks** at specific points in the kernel's network stack.

Packets flow through **chains** — ordered lists of rules. When a packet matches a rule, the **target** (ACCEPT, DROP, REJECT, LOG) is applied and processing stops. If no rule matches, the chain's **default policy** applies.

```
Incoming packet
       │
       ▼
┌─────────────┐
│ PREROUTING  │  (before routing decision — used for DNAT/NAT)
│   chain     │
└──────┬──────┘
       │
       ▼ (routing decision)
      / \
     /   \
    ▼     ▼
┌───────┐  ┌─────────┐
│ INPUT │  │ FORWARD │  INPUT: destined for this machine
│ chain │  │  chain  │  FORWARD: passing through to another machine
└───┬───┘  └────┬────┘
    │            │
    ▼            ▼
┌──────────────────────┐
│      Application      │  Packets accepted by INPUT reach the app
└──────────────────────┘
    │
    ▼ (packets the application sends)
┌─────────┐
│ OUTPUT  │  (packets generated by this machine)
│  chain  │
└────┬────┘
     │
     ▼
┌──────────────┐
│ POSTROUTING  │  (after routing decision — used for SNAT/masquerade)
│    chain     │
└──────────────┘
       │
       ▼
Outgoing packet
```

For most service-hardening scenarios, you work with the **INPUT** chain — controlling what traffic can reach your machine — and the **OUTPUT** chain for what your machine can send. FORWARD is used on routers.

### iptables Rules

Each iptables rule has four parts: the **table**, the **chain**, **match criteria**, and a **target**.

```bash
# General form:
iptables -t <table> -A <chain> <match-criteria> -j <target>
#         ^ table   ^ append  ^ what to match   ^ action

# Most common targets:
# ACCEPT  — allow the packet through
# DROP    — silently discard the packet (sender gets no response)
# REJECT  — discard and send an error back to the sender
# LOG     — log the packet (continues to next rule — no termination)
```

**Essential rules for securing a server:**

```bash
# Allow already-established connections (MUST be first for TCP to work)
iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# Allow loopback interface (required for local services)
iptables -A INPUT -i lo -j ACCEPT

# Allow SSH so you don't lock yourself out
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Allow HTTP and HTTPS
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# Drop everything else (set default policy last, after the ALLOW rules)
iptables -P INPUT DROP

# View current rules with line numbers
iptables -L INPUT -n --line-numbers

# Delete a rule by line number
iptables -D INPUT 3

# Flush all rules (start over — dangerous if you have a DROP default policy)
iptables -F INPUT
```

<Callout type="warning" title="Always Allow SSH Before Setting DROP Policy">
Always add your SSH ACCEPT rule **before** setting the default policy to DROP. If you run `iptables -P INPUT DROP` first and your SSH rule isn't in place, you immediately lose all access to the server. Every SSH session — including the one you're currently using — is an incoming connection. The rule must exist before the default policy takes effect. This is the most common firewall mistake that results in emergency console access to a cloud instance.
</Callout>

**Rule ordering matters — first match wins:**

```bash
# This rule accepts all traffic from 10.0.0.1
iptables -A INPUT -s 10.0.0.1 -j ACCEPT

# This rule drops all traffic from 10.0.0.0/24
iptables -A INPUT -s 10.0.0.0/24 -j DROP

# Result: 10.0.0.1 is ACCEPTED (matches first rule before reaching second)
# All other 10.0.0.x hosts are DROPPED

# If the rules were in reverse order, 10.0.0.1 would also be DROPPED
# because the subnet rule matches first
```

### iptables Tables

iptables has multiple **tables** for different purposes. The default table (used when you omit `-t`) is the **filter** table:

| Table | Purpose | Chains |
|-------|---------|--------|
| **filter** (default) | Allow or block packets | INPUT, FORWARD, OUTPUT |
| **nat** | Network address translation (DNAT/SNAT) | PREROUTING, POSTROUTING |
| **mangle** | Modify packet headers (TTL, TOS) | All 5 chains |
| **raw** | Bypass connection tracking | PREROUTING, OUTPUT |

For most service-hardening and lab work, you only need the filter table.

### ufw: The Friendly Frontend

`ufw` (Uncomplicated Firewall) is a wrapper around iptables designed to simplify common tasks. It generates the correct iptables rules under the hood.

```bash
# Enable ufw (starts the firewall and sets up default rules)
ufw enable

# Check current status and rules
ufw status verbose

# Allow common services by port/protocol
ufw allow 22/tcp      # SSH
ufw allow 80/tcp      # HTTP
ufw allow 443/tcp     # HTTPS

# Allow by service name (ufw knows common port mappings)
ufw allow ssh
ufw allow http
ufw allow https

# Deny a port
ufw deny 23/tcp       # Block telnet

# Delete a rule
ufw delete allow 80/tcp

# Allow from a specific IP
ufw allow from 10.0.0.100 to any port 22

# Deny all incoming from a subnet
ufw deny from 192.168.100.0/24

# Reset to defaults (removes all rules)
ufw reset
```

<Callout type="tip" title="ufw for Day-to-Day, iptables for Debugging">
`ufw` is what you'll use day-to-day on Ubuntu servers — it has a clean syntax, tracks rules persistently across reboots, and handles edge cases correctly. `iptables` is what you need to understand when debugging why ufw rules aren't working as expected, or when working on systems that don't have ufw installed. When a ufw rule doesn't seem to be working, run `iptables -L -n -v` to see exactly what rules are in place and which ones are matching traffic.
</Callout>

### Security Groups (Cloud Context)

In AWS, GCP, and Azure, **security groups** are the cloud-native equivalent of iptables rules. They are stateful firewall rules attached to network interfaces (or VMs directly).

- **Stateful** means return traffic for an established connection is automatically allowed — you only need to write rules for the initial direction. iptables achieves this with the `ESTABLISHED,RELATED` state match.
- Security groups are applied per-interface in the cloud provider's virtual network fabric — before the packet even reaches the VM's OS.
- Rules are configured through a cloud API or console instead of a command line.

<Callout type="tip" title="Security Groups Are Stateful iptables">
When you see "Security Group" in AWS, think "stateful iptables rules attached to a network interface." The mental model is the same: you define which source IPs, protocols, and ports are allowed. The main differences are that cloud security groups are always stateful (no need for ESTABLISHED rules), are applied outside the OS (can't be bypassed by a compromised process inside the VM), and are managed through the cloud API rather than iptables commands.
</Callout>
