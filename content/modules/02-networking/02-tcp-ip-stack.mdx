## Overview

Your API is timing out вҖ” is it a DNS problem, a routing issue, or is TCP retransmission burning time? Understanding the TCP/IP stack tells you which layer to investigate. The stack is not just theory вҖ” every network tool you will use (curl, ss, tcpdump, ping) operates at a specific layer, and knowing which layer explains what the tool can and cannot see.

This lesson explains the four layers of the TCP/IP model, how data is encapsulated as it moves down the stack, how IP addressing and subnets determine whether two hosts are on the same network, and how TCP delivers data reliably while UDP delivers it fast.

## How It Works

Each layer adds its own header (and sometimes trailer) when sending data down the stack. This is **encapsulation**. When receiving, each layer strips its header and passes the payload up.

### IP Addressing and Subnets

An **IPv4 address** is a 32-bit number written as four decimal octets separated by dots: `192.168.1.100`. Each octet is 0-255 (`192` = `11000000` in binary).

**Subnet masks** define which portion of an IP address identifies the network and which identifies the host:

```
IP address:   192.168.1.100
Subnet mask:  255.255.255.0   (or /24 in CIDR notation)

Binary breakdown:
  Address:    11000000.10101000.00000001.01100100
  Mask:       11111111.11111111.11111111.00000000
              в”ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ network в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө в”ңв”Җhostв”Җв”Ө

Network:      192.168.1.0
Host range:   192.168.1.1 to 192.168.1.254
Broadcast:    192.168.1.255
```

**CIDR notation** (`/24`) counts the number of 1-bits in the subnet mask. Common values:

| CIDR | Subnet Mask     | Hosts | Typical Use |
|------|-----------------|-------|-------------|
| /8   | 255.0.0.0       | 16.7M | Large private networks (10.x.x.x) |
| /16  | 255.255.0.0     | 65K   | Enterprise networks (172.16.x.x) |
| /24  | 255.255.255.0   | 254   | Typical office/home subnet |
| /30  | 255.255.255.252 | 2     | Point-to-point router links |

**Same subnet or not?** To determine if two IPs are on the same subnet, apply the subnet mask to both addresses:

```
Is 192.168.1.100/24 on the same subnet as 192.168.1.200/24?
  192.168.1.100  AND  255.255.255.0  =  192.168.1.0  вң“ same network
  192.168.1.200  AND  255.255.255.0  =  192.168.1.0  вң“ same network
  вҶ’ YES, same subnet вҶ’ can communicate directly via layer 2

Is 192.168.1.100/24 on the same subnet as 10.0.0.1/24?
  192.168.1.100  AND  255.255.255.0  =  192.168.1.0
  10.0.0.1       AND  255.255.255.0  =  10.0.0.0
  вҶ’ NO, different subnets вҶ’ must go via router
```

**Private address ranges** (RFC 1918 вҖ” not routable on the public internet):

| Range | Size | Typical Use |
|-------|------|-------------|
| `10.0.0.0/8` | Largest block | Data centers and cloud VPCs |
| `172.16.0.0/12` | Medium block | Docker default bridge (`172.17.0.0/16`) |
| `192.168.0.0/16` | Smallest block | Home router subnets |

### TCP: The Reliable Protocol

**TCP (Transmission Control Protocol)** provides reliable, ordered, connection-oriented data delivery. Before any data is exchanged, TCP establishes a connection via the **three-way handshake**:

```
TCP Three-Way Handshake
                    Client                    Server
                      в”Ӯ                         в”Ӯ
          SYN в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–әв”Ӯ  seq=1000               в”Ӯ
          (I want      в”Ӯ  "I want to connect,    в”Ӯ
          to connect)  в”Ӯ   my seq starts at 1000"в”Ӯ
                       в”Ӯ                         в”Ӯ
                       в”Ӯв—„в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җ SYN-ACK        в”Ӯ  seq=5000, ack=1001
                       в”Ӯ  "OK, connect accepted, в”Ӯ
                       в”Ӯ   my seq starts at 5000,в”Ӯ
                       в”Ӯ   I got your byte 1000" в”Ӯ
                       в”Ӯ                         в”Ӯ
          ACK в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–ә  ack=5001               в”Ӯ
          (Confirmed)  в”Ӯ  "Got your byte 5000"   в”Ӯ
                       в”Ӯ                         в”Ӯ
          DATA в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв–ә  "GET / HTTP/1.1..."    в”Ӯ
                       в”Ӯ                         в”Ӯ
```

After the handshake, both sides have established **sequence numbers** вҖ” a 32-bit counter that tracks how many bytes have been sent. This enables:

- **Acknowledgments:** The receiver confirms each byte it receives. `ACK=5001` means "I've received everything up to byte 5000, send me byte 5001 next."
- **Retransmission:** If the sender doesn't receive an ACK within a timeout period, it retransmits the unacknowledged data. This is why TCP is reliable вҖ” data will eventually be delivered or the connection will fail.
- **Ordered delivery:** Segments can arrive out of order (different routing paths). TCP buffers them and delivers them in order to the application.
- **Flow control (windowing):** The receiver advertises a **window size** вҖ” how many bytes it can buffer. The sender won't send more than the window allows, preventing buffer overflow at the receiver.

**Connection teardown** is a four-way exchange:
```
Client sends FIN вҶ’ Server acknowledges вҶ’ Server sends FIN вҶ’ Client acknowledges
```
After teardown, the connection enters TIME_WAIT for 2x MSL (Maximum Segment Lifetime, typically 60-120 seconds) to handle any delayed packets from the closed connection.

<Callout type="deep-dive" title="TCP Congestion Control">
Beyond flow control (receiver-side), TCP implements **congestion control** to avoid overwhelming the network itself (not just the receiver).

**Slow start:** A new TCP connection starts by sending a small number of segments (initially 1 MSS вҖ” Maximum Segment Size, usually 1460 bytes for Ethernet). Each ACK increases the congestion window exponentially until either packet loss is detected or a threshold (ssthresh) is reached.

**Congestion avoidance:** After reaching ssthresh, the window grows linearly (one MSS per RTT). When packet loss is detected (timeout or three duplicate ACKs), the algorithm interprets this as network congestion and reduces the window dramatically.

**Why this matters in production:** A freshly established TCP connection has a small congestion window. Your first HTTP request after a connection is established will be slow while TCP "ramps up." HTTP/2 and QUIC address this with connection reuse and different congestion strategies. This is why connection pool reuse matters in high-throughput services.
</Callout>

### UDP: The Fast Protocol

**UDP (User Datagram Protocol)** is the opposite of TCP: no handshake, no acknowledgments, no ordering, no retransmission. The sender fires off datagrams and doesn't track whether they arrived.

| Feature | TCP | UDP |
|---------|-----|-----|
| Connection setup | 3-way handshake | No setup |
| Acknowledgments | Yes | No confirmation |
| Lost data | Retransmission | No retry |
| Delivery order | Ordered | No ordering |
| Flow control | Yes | No throttling |
| Header size | ~20 bytes | 8 bytes |
| State | Connection state in kernel | Stateless |

**When UDP is the right choice:**
- **DNS queries:** Small request/response. If the query is lost, just retry. TCP overhead would double the latency.
- **Video streaming and gaming:** Old data is worse than no data. A dropped video frame is better than a delayed one that disrupts playback. UDP lets the application decide whether to retransmit.
- **NTP (Network Time Protocol):** A lost time sync packet is harmless вҖ” the next one arrives in seconds.
- **QUIC (HTTP/3):** UDP is used as the foundation for QUIC, which implements its own reliability and ordering at the application layer with better performance than TCP (no head-of-line blocking).

### Ports: Multiplexing Connections

IP addresses identify hosts. **Ports** identify specific processes (services) on those hosts. The transport layer uses source and destination ports to route incoming data to the right process.

```
Port-based multiplexing on one host

  Host: 192.168.1.10
  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
  в”Ӯ                                         в”Ӯ
  в”Ӯ  Web server process  вҶ’ port 80, 443     в”Ӯ
  в”Ӯ  SSH daemon          вҶ’ port 22          в”Ӯ
  в”Ӯ  Postgres            вҶ’ port 5432        в”Ӯ
  в”Ӯ  Redis               вҶ’ port 6379        в”Ӯ
  в”Ӯ                                         в”Ӯ
  в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

Incoming packet: src=203.0.113.5:54321  dst=192.168.1.10:443
OS routes this to the web server process based on dst port 443.
```

**Well-known ports** (0-1023) require root to bind:

| Port | Protocol |
|------|----------|
| 22   | SSH      |
| 25   | SMTP     |
| 53   | DNS      |
| 80   | HTTP     |
| 443  | HTTPS    |
| 5432 | PostgreSQL |
| 6379 | Redis    |

**Ephemeral ports** (typically 32768-60999 on Linux) are assigned by the OS to the client side of a connection. When your browser connects to port 443, the OS picks an ephemeral source port for the connection.

A unique connection is identified by a 5-tuple: `(protocol, src_ip, src_port, dst_ip, dst_port)`. Two connections from the same client to the same server differ only in their ephemeral source port.

<Callout type="tip" title="ss -tlnp: Seeing the Transport Layer">
The `ss` command (socket statistics) shows you exactly which ports are listening and which process owns them вҖ” you are looking directly at the transport layer.

```bash
ss -tlnp
# -t: TCP sockets
# -l: listening sockets only
# -n: show port numbers (not names)
# -p: show process name and PID
```

When you troubleshoot "why can't I connect to port 8080?", `ss -tlnp` tells you whether anything is actually listening there. You will use this command extensively in the troubleshooting lesson.
</Callout>
