## Overview

Your application can't connect to the database at `db.internal.company.com` — is the server down, or is DNS failing to resolve the hostname? Understanding DNS tells you where the name lookup is breaking. Before any TCP connection can be established to a hostname, the OS must resolve that name to an IP address. If DNS is broken, nothing that uses hostnames works — even if the target server is perfectly healthy.

This lesson explains how domain names map to IP addresses: the resolution chain from your machine to authoritative nameservers, the record types that carry different kinds of information, how caching layers affect propagation, and how to debug the entire process with `dig`.

## How It Works

### What DNS Does

**DNS (Domain Name System)** is a distributed, hierarchical database that maps human-readable names (`example.com`) to machine-readable IP addresses (`93.184.216.34`). Every network request that uses a hostname starts with a DNS lookup — unless the destination is already an IP address.

Without DNS, you would need to memorize IP addresses for every service you connect to, and every address change would break every client. DNS decouples names from addresses, allowing IPs to change without updating clients.

### The Resolution Chain

DNS resolution is not a single lookup — it is a chain of queries that traverses the DNS hierarchy from root to authoritative nameserver:

```
DNS Resolution for "app.example.com"

  Your machine                             Internet DNS hierarchy
  ┌──────────┐  1. Is "app.example.com"    ┌────────────────────┐
  │          │     in browser cache?       │                    │
  │ Browser  │     No → ask OS resolver    │   Root Nameserver  │
  │          │                             │   (13 clusters,    │
  └────┬─────┘                             │    worldwide)      │
       │  2. Is it in OS cache?            │                    │
       ▼     No → ask recursive resolver   └──────────┬─────────┘
  ┌──────────┐                                        │
  │    OS    │                                        │ 3. Knows who runs .com
  │ Resolver │                                        │    → points to TLD
  │(/etc/    │                                        ▼
  │resolv.   │  ◄─────────────────── ┌────────────────────┐
  │conf)     │                       │   .com TLD Server  │
  └────┬─────┘                       │   (Verisign)       │
       │                             └──────────┬─────────┘
       ▼                                        │
  ┌────────────────┐                            │ 4. Knows example.com's NS
  │   Recursive    │ ◄──────────────────────────┘    → points to authoritative
  │   Resolver     │                             ┌────────────────────┐
  │ (your ISP,     │ ─────────────────────────► │  example.com NS    │
  │  8.8.8.8, etc) │                             │  (authoritative)   │
  └────────────────┘ ◄──────────────────────────│  Has A record for  │
       │               5. Returns answer         │  app.example.com   │
       │               "app.example.com =        └────────────────────┘
       ▼               93.184.216.34, TTL 300s"
  ┌──────────┐
  │    OS    │  Caches answer for TTL seconds
  │ Resolver │  Returns IP to your application
  └──────────┘
```

**Step-by-step:**
1. **Browser cache:** Chrome, Firefox maintain their own DNS cache. `chrome://net-internals/#dns` to inspect it.
2. **OS resolver cache:** The OS checks its local DNS cache (`/etc/hosts` first, then cached responses from previous queries).
3. **Recursive resolver:** Your configured DNS server (from `/etc/resolv.conf`). This server does the actual recursive lookup on your behalf — it queries the root, TLD, and authoritative nameservers.
4. **Root nameserver:** 13 logical nameserver clusters (operated by different organizations) that know where each TLD's servers live. They don't know `app.example.com` — they just know who's authoritative for `.com`.
5. **TLD nameserver:** Knows which nameservers are authoritative for each second-level domain (`example.com`). Returns NS records pointing to `example.com`'s nameservers.
6. **Authoritative nameserver:** The server that actually holds the DNS records for `example.com`. It answers definitively: "app.example.com is at 93.184.216.34, and you can cache that for 300 seconds."

The answer travels back through the recursive resolver to your OS, which caches it for the TTL duration. Your application never sees this chain — it just gets the IP address back from the OS `getaddrinfo()` syscall.

### DNS Record Types

DNS zones contain different record types for different purposes:

| Type  | Purpose                          | Example Value                        |
|-------|----------------------------------|--------------------------------------|
| A     | IPv4 address mapping             | `app.example.com → 93.184.216.34`  |
| AAAA  | IPv6 address mapping             | `app.example.com → 2606:2800::34`  |
| CNAME | Canonical name (alias)           | `www.example.com → app.example.com` |
| MX    | Mail exchange server             | `example.com → mail.example.com` (priority 10) |
| NS    | Nameserver for a zone            | `example.com → ns1.registrar.com`  |
| PTR   | Reverse DNS (IP → name)          | `34.216.184.93.in-addr.arpa → app.example.com` |
| TXT   | Arbitrary text data              | SPF records, domain verification, DKIM keys |
| SOA   | Start of authority for a zone    | Defines serial number, TTL, responsible email |

**Important CNAME behavior:** A CNAME points to another name (not an IP). The resolver must then look up that name too. `www.example.com → app.example.com → 93.184.216.34` requires two lookups. You **cannot** put a CNAME on a zone apex (the root domain `example.com` itself) because it conflicts with required SOA and NS records — this is why CDNs offer "ALIAS" or "ANAME" records as a workaround.

**MX records** have a priority number. Lower numbers mean higher priority. If `mail.example.com` is down, mail is retried at the next-highest priority MX.

**PTR records** (reverse DNS) live in the special `in-addr.arpa` zone. The IP address is written backwards: `93.184.216.34` becomes `34.216.184.93.in-addr.arpa`. PTR records are used by mail servers to verify that a sending IP maps back to the expected hostname.

### DNS Caching

Every response includes a **TTL (Time to Live)** in seconds. Resolvers cache the answer for that duration before querying again.

```
DNS Caching Layers

┌─────────────────────┐
│   Browser cache     │  ~60s typical, varies by browser
├─────────────────────┤
│   OS resolver cache │  Respects TTL from authoritative response
│   (nscd, systemd-   │  On macOS: managed by mDNSResponder
│    resolved)        │  On Linux: systemd-resolved or dnsmasq
├─────────────────────┤
│ Recursive resolver  │  ISP or 8.8.8.8 — caches for full TTL
│ cache               │  Shared across all users of that resolver
└─────────────────────┘
```

**TTL strategy:**
- **Low TTL (60-300s):** Use before migrations — lets you change the record and have clients pick up the new IP quickly. Downside: more DNS queries, more load on authoritative servers.
- **High TTL (3600-86400s):** Use for stable records — reduces DNS query volume and improves response time for end users (more cache hits). Downside: changes take hours to propagate.

**DNS propagation** is not instant because each resolver caches based on the old TTL. When you change a DNS record, resolvers that already have the old record cached won't re-query until that cache expires. A record with TTL=86400 can take up to 24 hours to "propagate" everywhere. This is why you lower the TTL *before* a migration, not during.

<Callout type="warning" title="DNS Propagation Is Not Instant">
When you change a DNS record, some resolvers may cache the old record for up to the previous TTL duration. If your A record had a TTL of 86400 seconds (24 hours), the change can take a full day to reach all resolvers. Best practice: lower the TTL to 300 seconds **several hours before** you make the change, wait for the old TTL to expire, make the change, then raise the TTL back after the migration is complete.
</Callout>

<Callout type="tip" title="Container DNS: Docker's Embedded Resolver">
In Docker, every container on a bridge network has a built-in DNS resolver at `127.0.0.11`. When a container does a DNS lookup for another container's service name (e.g., `db` or `api`), Docker's embedded resolver intercepts the query and returns the target container's IP address on the shared network.

This is the same DNS mechanism you are learning here — just running inside Docker's network stack. When you reach Phase 4, you will run `dig db @127.0.0.11` from inside a container and see exactly this resolver responding. The mental model is identical to production DNS.
</Callout>

<Callout type="deep-dive" title="DNSSEC: Authenticating DNS Responses">
Standard DNS has no authentication — a resolver cannot verify that a response actually came from the legitimate authoritative nameserver. This enables **DNS spoofing (cache poisoning)**: an attacker tricks a resolver into caching a false IP for a domain, redirecting all users to an attacker-controlled server.

**DNSSEC (DNS Security Extensions)** adds cryptographic signatures to DNS records. The authoritative server signs its records with a private key. Resolvers validate the signature using the public key, which is anchored at the root zone. A validating resolver rejects any response it cannot verify.

DNSSEC is not universally deployed — many domains do not sign their zones, and many resolvers do not validate signatures. It also does not encrypt DNS traffic (that is what DNS over HTTPS/DNS over TLS does). DNSSEC is beyond the scope of these exercises but understanding the threat model (unauthenticated DNS responses) explains why it exists.
</Callout>
