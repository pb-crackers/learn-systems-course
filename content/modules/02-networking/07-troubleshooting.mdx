## Overview

It's 2 AM and the monitoring alert says "service unreachable." You have 5 minutes to figure out if it's a DNS problem, a routing issue, a firewall rule, or the service itself being down. This lesson gives you the systematic approach to find the answer fast. Random guessing wastes time and can make things worse. The right approach is methodical: work up through the network stack layer by layer, using a specific tool at each layer to confirm or rule out that layer as the source of the problem.

This lesson covers the five essential diagnostic tools (ping, traceroute, ss, curl, tcpdump), explains what each one tells you and how to interpret its output, and provides a systematic checklist for reaching a diagnosis quickly.

## How It Works

### The Troubleshooting Methodology

Network problems manifest at a specific layer of the TCP/IP stack. The most efficient approach is to start at the bottom (physical/network) and work up — there's no point debugging DNS if you can't even ping the host.

```
Layer 7: Application     Can the application respond?
         Tool: curl      Does curl return the expected HTTP response?
         ─────────────────────────────────────────────────────
Layer 4: Transport       Is the port open and listening?
         Tool: ss, nc    Is the service bound to the expected port?
         ─────────────────────────────────────────────────────
Layer 3: Network         Can I reach the host? Is routing correct?
         Tool: ping,     Does ping get a reply? Does traceroute
               traceroute show the expected path?
         ─────────────────────────────────────────────────────
Layer 2: Data Link       Is the interface up? Any link errors?
         Tool: ip link   Is the network interface up? Is there
                         a valid MAC in ARP table?
         ─────────────────────────────────────────────────────
Layer 1: Physical        Is there a cable? Is the interface driver loaded?
         Tool: ip link   Interface shows UP with an IP address?
```

**The systematic checklist (use this in order):**

1. `ping <host>` — can I reach the host? (Layer 3)
2. `traceroute <host>` if ping fails — where does the path break?
3. `ss -tlnp` on the target — is the service actually listening on the expected port?
4. `curl <service>` — does the application respond?
5. `tcpdump -i <iface> port <N> -A -c 10` if everything looks right but it still doesn't work — what's actually on the wire?

### ping: Testing Basic Connectivity

`ping` sends ICMP echo request packets to a host and measures whether they come back and how long they take. It tests whether Layer 3 (IP routing) is working.

```bash
# Basic ping — sends until you press Ctrl-C
ping server.example.com

# Send exactly 5 packets then stop
ping -c 5 server.example.com

# Flood ping — send as fast as possible (requires root; use for throughput testing)
ping -f server.example.com
```

**Reading ping output:**

```
PING webserver (10.0.0.3) 56(84) bytes of data.
64 bytes from 10.0.0.3: icmp_seq=1 ttl=64 time=0.412 ms
64 bytes from 10.0.0.3: icmp_seq=2 ttl=64 time=0.387 ms
64 bytes from 10.0.0.3: icmp_seq=3 ttl=64 time=0.401 ms

--- webserver ping statistics ---
3 packets transmitted, 3 received, 0% packet loss, time 2003ms
rtt min/avg/max/mdev = 0.387/0.400/0.412/0.010 ms
```

- **TTL (Time to Live)**: Decremented by each router. A TTL of 64 means you're probably one hop away. Very low TTL (< 10) suggests many hops or a distant host.
- **RTT (Round-Trip Time)**: Latency in milliseconds. Local network: < 1ms. Same datacenter: 1-5ms. Cross-country: 30-80ms. Unusual spikes indicate congestion or routing problems.
- **Packet loss %**: Any loss indicates a problem. 100% loss means completely unreachable.

**What different failures mean:**

| Ping Result | Likely Cause |
|-------------|-------------|
| `Destination Host Unreachable` | Router knows the network but can't reach the specific host — bad routing or host is down |
| `Network Unreachable` | No route to the destination network — missing route or wrong gateway |
| `Request timeout` (no reply) | Host is down, firewall is DROP-ing (not REJECT-ing), or ICMP is blocked |
| Replies with high packet loss | Network congestion, flaky link, or overloaded host |
| Replies with consistently high RTT | Congested or distant path |

### traceroute: Tracing the Path

`traceroute` reveals the hop-by-hop path packets take to reach a destination. It works by sending packets with increasing TTL values — each router decrements TTL; when it reaches 0, the router sends back an ICMP "Time Exceeded" message, revealing its address.

```bash
traceroute server.example.com

# Use ICMP instead of UDP (more likely to pass through firewalls)
traceroute -I server.example.com

# Limit maximum hops
traceroute -m 20 server.example.com
```

**Reading traceroute output:**

```
traceroute to webserver (10.0.0.3), 30 hops max
 1  gateway (10.0.0.1)    0.312 ms  0.287 ms  0.301 ms
 2  router.isp.net        5.412 ms  5.398 ms  5.407 ms
 3  * * *                            ← router didn't respond (filtered)
 4  10.0.0.3              8.201 ms  8.188 ms  8.195 ms
```

- Each line is one hop (one router)
- Three RTT measurements are shown per hop
- `* * *` means the router at that hop didn't respond — this is common and doesn't necessarily indicate a problem (many routers filter ICMP Time Exceeded). If the **final destination** is reachable (last line has an IP and RTT), intermediate `* * *` hops are irrelevant.
- **Where to look for problems**: If traceroute stops at a specific hop with `* * *` and doesn't reach the destination, that hop is where routing breaks or a firewall is blocking traffic.

### ss: Socket Statistics

`ss` (Socket Statistics) replaces the older `netstat` command. It queries the kernel directly for socket state information — much faster than netstat for large numbers of connections.

The most useful flag combination for diagnosing service problems:

```bash
# Show all listening TCP sockets with process information
ss -tlnp

# t = TCP, l = listening, n = numeric (no DNS), p = process name
```

**Reading ss output:**

```
Netid State  Recv-Q Send-Q Local Address:Port  Peer Address:Port  Process
tcp   LISTEN  0      128    0.0.0.0:22          0.0.0.0:*          users:(("sshd",pid=1234))
tcp   LISTEN  0      511    0.0.0.0:80          0.0.0.0:*          users:(("nginx",pid=5678))
tcp   LISTEN  0      128    127.0.0.1:5432      0.0.0.0:*          users:(("postgres",pid=9012))
```

- **Local Address:Port** is where the service is listening
- `0.0.0.0:80` means listening on all interfaces (reachable from anywhere)
- `127.0.0.1:5432` means listening only on loopback (not reachable from outside — intentional for databases)
- **Process** column shows which process owns the socket

**What ss tells you for troubleshooting:**

```bash
# Is the service even running?
ss -tlnp | grep 8080
# If no output: the service isn't listening — it may have crashed or started on wrong port

# What's actually listening on port 80?
ss -tlnp | grep ':80 '

# Show all established connections to port 443
ss -tnp state established '( dport = :443 )'
```

### tcpdump: Packet Capture and Analysis

`tcpdump` is your last resort when everything looks right at the application and OS level but it still doesn't work. It captures actual packets from a network interface so you can see exactly what's on the wire.

**Always use filters.** Running bare `tcpdump` on a busy interface produces hundreds of lines per second — you won't be able to read them and it won't help you:

```bash
# Capture HTTP traffic (port 80) in ASCII, stop after 5 packets
tcpdump -i eth0 port 80 -A -c 5

# Capture all traffic to/from a specific host
tcpdump -i eth0 host webserver -n

# Capture TCP traffic on port 443 with verbose output
tcpdump -i eth0 tcp port 443 -v

# Capture and save to a file for later analysis (e.g., in Wireshark)
tcpdump -i eth0 port 80 -w /tmp/capture.pcap

# Common flags:
# -i eth0   interface to capture on
# -A        print packet payload as ASCII (readable for HTTP)
# -n        no DNS lookup (faster, shows IPs not hostnames)
# -v/-vv    verbose (more packet header detail)
# -c N      stop after N packets
# -w file   write to pcap file instead of stdout
```

<Callout type="warning" title="Always Filter tcpdump — Never Run It Bare">
Never run `tcpdump` without a filter on a busy network interface. On a production system handling thousands of connections per second, unfiltered tcpdump floods your terminal faster than you can read it, and the output is useless. Always specify at least a port (`port 80`) or host filter (`host webserver`). The `-c N` flag to capture a fixed number of packets is also your friend — it ensures tcpdump exits automatically.
</Callout>

**Reading tcpdump output:**

```
10:00:01.123456 IP client.52341 > webserver.80: Flags [S], seq 12345, win 65535
10:00:01.123789 IP webserver.80 > client.52341: Flags [S.], seq 98765, ack 12346, win 65535
10:00:01.124012 IP client.52341 > webserver.80: Flags [.], ack 98766
```

| Field | Description |
|-------|-------------|
| **Timestamp** | When the packet was captured |
| **IP src > dst** | Source and destination addresses with port after the dot |
| **Flags** | TCP flags (see table below) |

| Pattern | Meaning |
|---------|---------|
| `[S]` | SYN — connection attempt |
| `[S.]` | SYN-ACK — server accepting |
| `[.]` | ACK — acknowledgment |
| `[P.]` | PSH+ACK — data push |
| `[F.]` | FIN — connection close |
| `[R]` | RST — connection reset/refused |
| `[S]` then `[S.]` then `[.]` | Healthy three-way handshake |
| `[S]` then `[R]` | Connection refused — port not listening |
| `[S]` repeated, no response | Firewall DROP — SYN timeout |

### curl for Application Debugging

Once you've confirmed the service is listening (ss) and the network path is clear (ping), curl confirms whether the application is responding correctly:

```bash
# Full verbose — see headers, TLS, everything
curl -v http://webserver/api/health

# Just the HTTP status code
curl -s -o /dev/null -w '%{http_code}\n' http://webserver/

# Short timeout — don't wait forever for a slow service
curl --connect-timeout 5 --max-time 10 http://webserver/

# Test application is responding on a specific path
curl -I http://webserver/api/health
```

**Interpreting curl failures:**

| curl Error | Meaning |
|------------|---------|
| `Connection refused` | Nothing is listening on that port (check ss) |
| `Could not resolve host` | DNS failure (check dig, check /etc/resolv.conf) |
| `Connection timed out` | Firewall is DROP-ing packets (check iptables/ufw) |
| `SSL certificate error` | TLS cert is invalid/expired (use -k to test without validation) |
| HTTP 502 | Proxy reached backend but got bad response (backend is up but crashing) |
| HTTP 504 | Proxy timed out waiting for backend (backend is slow or down) |

### Systematic Checklist

When a service is unreachable, run these steps in order:

```
1. Can I reach the host at all?
   ping <host>
   → If NO: routing problem. Run traceroute to find where it breaks.
   → If YES: continue to step 2.

2. Is the service listening on the expected port?
   ss -tlnp | grep <port>  (run on the target host)
   → If NO: the service isn't running or is on the wrong port.
             Check: systemctl status <service>, journalctl -u <service>
   → If YES: continue to step 3.

3. Can I connect from outside?
   nc -zv <host> <port>  (run from the client)
   → If NO but service IS listening: firewall is blocking.
             Check: iptables -L -n, ufw status
   → If YES: continue to step 4.

4. Does the application respond correctly?
   curl -v http://<host>:<port>/health
   → If HTTP error: application-level issue. Check app logs.
   → If connection refused/timeout: inconsistency with step 3 — recheck.
   → If OK: check DNS, check the actual URL the client is using.

5. Still can't find it?
   tcpdump -i eth0 port <N> -A -c 10  (run on the server)
   Make a request from the client — does tcpdump see any packets?
   → If NO packets on the server: routing/firewall drops before the server
   → If packets arrive but app doesn't respond: application bug
```
