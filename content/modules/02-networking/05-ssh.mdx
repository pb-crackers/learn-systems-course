## Overview

You need to access a database server that's behind a firewall and only reachable from a bastion host. SSH tunneling lets you create a secure path through the bastion without exposing the database to the internet. SSH (Secure Shell) is the protocol that replaced telnet, rsh, and rlogin — every one of those sent credentials in plaintext. SSH encrypts the entire session, authenticates the server so you can't be redirected to an impostor, and provides multiple mechanisms to authenticate the client.

This lesson covers the full SSH model: how the protocol works, why key-based authentication is more secure than passwords, how to configure `~/.ssh/config` for convenience, the three flavors of SSH tunneling, and when agent forwarding is useful (and dangerous).

## How It Works

### How SSH Works

SSH operates on port 22 by default. When you run `ssh user@server`, here is what happens in order:

1. **TCP connection**: Your client opens a TCP connection to port 22 on the server.
2. **Protocol negotiation**: Client and server agree on the SSH protocol version (2 is current) and algorithm preferences.
3. **Server authentication**: The server presents its **host key** (a long-lived key pair unique to this server). If you've connected before, your `~/.ssh/known_hosts` file has the expected public key fingerprint — if it matches, you proceed. If it doesn't match, SSH warns you about a potential MITM attack.
4. **Key exchange**: Client and server use Diffie-Hellman (or ECDH) to derive a shared session key without either side transmitting the key over the wire. All subsequent communication is encrypted with this key.
5. **Client authentication**: With the encrypted channel established, the client proves its identity. This is either password authentication (the password is sent over the already-encrypted channel) or public key authentication (preferred).

### Key-Based Authentication

Password authentication has two problems: passwords can be guessed, and servers get hammered by brute-force attempts. Key-based authentication uses asymmetric cryptography: you hold a private key; the server has your public key. Only the holder of the private key can authenticate, and the private key never leaves your machine.

**Generating a key pair:**
```bash
# Generate an Ed25519 key pair (current best practice)
ssh-keygen -t ed25519 -C "your@email.com"

# Generates two files:
# ~/.ssh/id_ed25519      — private key (NEVER share this)
# ~/.ssh/id_ed25519.pub  — public key (safe to share; goes on servers)
```

**Copying the public key to a server:**
```bash
# ssh-copy-id appends your public key to ~/.ssh/authorized_keys on the remote
ssh-copy-id user@server

# Or manually — paste the public key content into ~/.ssh/authorized_keys on the server
cat ~/.ssh/id_ed25519.pub
# Copy the output, then on the server:
echo "ssh-ed25519 AAAAC3Nza..." >> ~/.ssh/authorized_keys
```

**How the authentication exchange works:**

```
Client                                     Server
  │                                           │
  │──── "I have key: ed25519-sha2 AAAA..." ► │
  │     (sends public key fingerprint)        │
  │                                           │
  │     (server checks ~/.ssh/authorized_keys)│
  │     (finds matching public key)           │
  │                                           │
  │  ◄── Challenge (random bytes encrypted ── │
  │       with your public key)               │
  │                                           │
  │──── Response (challenge decrypted with ► │
  │      your private key, re-signed)         │
  │                                           │
  │  ◄── Authentication successful ─────────  │
  │                                           │
  │  ════ Encrypted shell session ═══════════ │
```

Only the holder of the private key can decrypt the server's challenge and produce the correct response. This proves identity without the private key ever leaving the client.

### SSH Config File

Repeatedly typing `ssh -i ~/.ssh/my_key -p 2222 deploy@long.hostname.example.com` is error-prone. `~/.ssh/config` lets you define named hosts with their settings:

```
# ~/.ssh/config

# Production bastion host
Host bastion
    HostName bastion.example.com
    User ec2-user
    IdentityFile ~/.ssh/prod_key
    Port 22

# Internal app server — only reachable via bastion
Host app-server
    HostName 10.0.1.50
    User deploy
    IdentityFile ~/.ssh/prod_key
    ProxyJump bastion

# Dev server with non-standard port
Host dev
    HostName dev.example.com
    User developer
    Port 2222
    IdentityFile ~/.ssh/dev_key
```

With this config, `ssh app-server` automatically connects through the bastion with the correct key — no flags needed. The `ProxyJump` directive tells SSH to first connect to `bastion`, then from there SSH to the internal address.

### SSH Tunneling

SSH tunneling creates an encrypted TCP forwarding path between your machine and a remote network. There are three kinds:

**Local port forwarding (`-L`)**: A port on your local machine is forwarded to a port on (or reachable from) the remote server. Use case: access a remote database that's not publicly exposed.

```bash
# Forward localhost:5432 to the database server behind the firewall
ssh -L 5432:db.internal:5432 user@bastion

# Now in another terminal — connect as if the DB is local
psql -h localhost -p 5432 -U dbuser mydb
```

```
You           Bastion          DB Server
localhost:5432 ──SSH tunnel──► :22   ──TCP──► db.internal:5432
```

**Remote port forwarding (`-R`)**: A port on the remote server is forwarded back to a port on your local machine. Use case: expose a local dev server to a remote tester.

```bash
# Anyone connecting to port 8080 on the remote server gets forwarded to your local :3000
ssh -R 8080:localhost:3000 user@server

# Now someone at the remote server can test your local app:
# curl http://localhost:8080/ (run on the remote server) → your localhost:3000
```

```
You           Remote Server
localhost:3000 ◄──SSH tunnel── :22
                                ▲
                         :8080 ─┘
```

<Callout type="deep-dive" title="Dynamic Port Forwarding: SSH as a SOCKS Proxy">
`ssh -D 1080 user@server` creates a SOCKS proxy on localhost port 1080. Your applications can send TCP traffic of any kind through this proxy, and it exits at the remote server. This is useful for routing browser traffic through a remote machine (essentially a VPN-lite). Configure your browser's proxy settings to use SOCKS5 at `127.0.0.1:1080` and your browsing traffic exits from the SSH server's IP address.
</Callout>

### Agent Forwarding

When you're on a bastion host and need to SSH further to an internal server, you have two options: copy your private key to the bastion (bad — exposes your key on a shared machine), or use agent forwarding.

SSH agent forwarding (`-A`) makes your local SSH agent available on the remote machine. When the bastion needs to authenticate to an internal server, it asks your local machine's SSH agent to handle the authentication — your private key never leaves your laptop.

```bash
# Connect to bastion with agent forwarding enabled
ssh -A user@bastion

# Now from the bastion, SSH to an internal server
# The bastion's SSH client asks your local agent for the key
ssh user@internal-server    # Works! Your local key authenticates.
```

<Callout type="warning" title="Agent Forwarding Security Risk">
Never use agent forwarding (`-A`) to servers you don't fully trust. When agent forwarding is enabled, the remote machine's root user can use your forwarded agent to authenticate as you to any server that has your public key. An attacker who compromises root on the bastion can SSH anywhere your key has access, without ever seeing your private key. Use `ProxyJump` instead of agent forwarding — it proxies the connection through the bastion without exposing your agent to it.
</Callout>

### Host Key Verification

The first time you connect to a new server, SSH asks you to verify the host key fingerprint:

```
The authenticity of host 'server.example.com (10.0.0.5)' can't be established.
ED25519 key fingerprint is SHA256:AAAA...
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

After you accept, the fingerprint is stored in `~/.ssh/known_hosts`. On all future connections, SSH checks the server presents the same key. If it doesn't:

```
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
@    WARNING: REMOTE HOST IDENTIFICATION HAS CHANGED!     @
@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
```

This is a security warning — it means either:
1. The server was rebuilt and has new host keys (legitimate)
2. Someone is performing a man-in-the-middle attack (malicious)

For containers that get recreated frequently, this is a nuisance. You can remove the old entry: `ssh-keygen -R server.example.com`. In production, treat this warning seriously and verify through a separate channel before proceeding.
