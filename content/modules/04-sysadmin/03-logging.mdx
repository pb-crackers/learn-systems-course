## Overview

An application has been failing silently for three hours. No alerts fired. When you finally check the server, the process is down and the disk is 98% full — rotated over by uncontrolled log growth. Understanding the Linux logging stack is not just about reading logs — it is about knowing where logs go, how they are stored, and how to prevent them from becoming an operational hazard.

Modern Linux uses a layered logging architecture: journald captures structured logs from systemd services, rsyslog optionally forwards those logs to `/var/log/` text files, and logrotate manages retention. This lesson traces each layer.

## How It Works

### The Logging Pipeline

When a service writes to stdout or stderr, and its unit file has `StandardOutput=journal`, the logging pipeline looks like this:

```
Application stdout/stderr
         ↓
    journald (PID 1 subsystem)
    /run/log/journal/ or /var/log/journal/
    Binary structured storage, indexed
         ↓
    rsyslog (optional consumer)
    reads from journald, writes to /var/log/
    Plain text files: auth.log, kern.log, syslog
         ↓
    logrotate (scheduled by cron or systemd timer)
    Rotates, compresses, and purges old log files
```

Applications can also write directly to the **syslog socket** (`/dev/log`) — this is how older applications log without being started by systemd. When an application opens `/dev/log` and writes a syslog-format message, journald (or rsyslog, depending on configuration) picks it up.

### The syslog Protocol (RFC 5424)

The syslog protocol defines a **facility** (what kind of thing is logging) and a **severity** (how serious) for every log message. These two values encode context without requiring you to parse the message text.

**Facilities** (source type):
| Value | Keyword | Meaning |
|-------|---------|---------|
| 0 | kern | Kernel messages |
| 1 | user | User-level messages |
| 4 | auth | Authentication and authorization |
| 6 | lpr | Printer subsystem |
| 16–23 | local0–local7 | Local use (applications use these) |

**Severities** (priority levels, lower = more severe):
| Value | Keyword | Meaning |
|-------|---------|---------|
| 0 | emerg | System is unusable |
| 1 | alert | Action must be taken immediately |
| 2 | crit | Critical conditions |
| 3 | err | Error conditions |
| 4 | warning | Warning conditions |
| 5 | notice | Normal but significant event |
| 6 | info | Informational messages |
| 7 | debug | Debug-level messages |

journalctl's `-p` flag filters by severity: `journalctl -p err` shows errors and above (err, crit, alert, emerg).

### journald: The Primary Log Store

journald is the systemd logging daemon. It captures log output from:
- Services with `StandardOutput=journal` in their unit file
- The kernel (via `/proc/kmsg`)
- Applications writing to the syslog socket (`/dev/log` and `/run/systemd/journal/syslog`)
- Applications writing to the journal API directly

journald stores logs as **structured binary data** — not text files. Each log entry is a set of key=value fields: timestamp, hostname, unit name (`_SYSTEMD_UNIT`), PID (`_PID`), the message text (`MESSAGE`), and many more. Because these fields are indexed, filtering is fast — no grep required.

**Log persistence**: By default, journald stores logs in `/run/log/journal/` which is a tmpfs (volatile — cleared on reboot). To enable persistent logging across reboots, create the persistent directory:

```bash
mkdir -p /var/log/journal
systemd-tmpfiles --create --prefix /var/log/journal
systemctl restart systemd-journald
```

After this, logs accumulate in `/var/log/journal/<machine-id>/`. journald auto-rotates this directory when it reaches 10% of the filesystem or exceeds configured `SystemMaxUse`.

<Callout type="deep-dive" title="journald and /var/log/syslog">
On modern Ubuntu, `/var/log/syslog` is populated by rsyslog reading FROM journald — journald is the primary store. rsyslog connects to journald's socket and re-logs entries to the traditional text files in `/var/log/`. This is why `journalctl` gives you richer filtering than `grep` on `/var/log/syslog` — journalctl queries the structured binary data directly, while syslog is a plain-text re-export that has lost the structured fields.

You can verify this by checking rsyslog's configuration: `grep -r 'imjournal\|journal' /etc/rsyslog.conf /etc/rsyslog.d/` — on Ubuntu 22.04 you will find the `imjournal` module loading journald entries.
</Callout>

### /var/log/ Directory Structure

Even with journald as the primary store, `/var/log/` contains important text logs:

| File | Content |
|------|---------|
| `/var/log/syslog` | General system messages (Ubuntu) or `/var/log/messages` (RHEL/CentOS) |
| `/var/log/auth.log` | Authentication events: SSH logins, sudo usage, PAM |
| `/var/log/kern.log` | Kernel messages only |
| `/var/log/dpkg.log` | Package manager activity (apt installs, removals) |
| `/var/log/apt/` | Detailed apt-get/apt transaction history |
| `/var/log/nginx/` | Nginx access and error logs (created by nginx package) |
| `/var/log/mysql/` | MySQL error log (created by mysql package) |

### logrotate: Preventing Disk Exhaustion

logrotate is a utility that rotates, compresses, and purges log files on a schedule. Without it, `/var/log/nginx/access.log` on a busy server would grow indefinitely until the disk fills.

logrotate configuration files live in:
- `/etc/logrotate.conf` — global defaults
- `/etc/logrotate.d/` — per-application overrides (e.g., `/etc/logrotate.d/nginx`)

A typical logrotate configuration:

```
/var/log/myapp/*.log {
    daily           # rotate every day
    rotate 14       # keep 14 rotated files (2 weeks of history)
    compress        # gzip old log files
    delaycompress   # do not compress the most recent rotated file (still being written to)
    missingok       # do not error if log file is missing
    notifempty      # do not rotate if file is empty
    create 0640 www-data adm   # create new empty log with these permissions and ownership
    postrotate
        nginx -s reopen         # tell nginx to open the new log file
    endscript
}
```

**Key directives:**

| Directive | Meaning |
|-----------|---------|
| `daily` / `weekly` / `monthly` | Rotation frequency |
| `rotate N` | Keep N old log files before deleting |
| `compress` | gzip rotated files |
| `delaycompress` | Skip compressing the most recent rotation (may still be open) |
| `missingok` | Do not fail if the log file is absent |
| `notifempty` | Skip rotation if the current log is empty |
| `postrotate ... endscript` | Script to run after rotation (typically signal the app to reopen logs) |
| `copytruncate` | Copy the log file then truncate it (for apps that cannot reopen files) |

logrotate runs daily via a cron job at `/etc/cron.daily/logrotate`. You can force rotation manually:

```bash
logrotate --force /etc/logrotate.d/nginx   # force-rotate one config
logrotate --debug /etc/logrotate.conf      # dry-run to see what would happen
```

### Centralized Logging: Why Single-Machine Logs Break at Scale

When you operate a single server, SSH and `journalctl` work fine. When you have a fleet of 50 servers, multiple problems emerge:

1. **No correlation**: An error on server A triggers a cascade on server B — but you have to SSH into each server separately and manually correlate timestamps.
2. **Lost logs**: If a server dies, its logs may die with it (unless stored persistently off-machine).
3. **No aggregation**: "How many 500 errors per minute across all API servers?" requires log data from all servers in one place.

The solution is **log shipping** — forwarding logs from each host to a central store:

```
Server A → [Filebeat/Fluentd agent] → Central Store ← Server B
                                       (ELK, Loki)    [Filebeat/Fluentd agent]
                                           ↑
                                       Server C
                                       [agent]
```

**Common patterns:**
- **rsyslog forwarding**: Configure rsyslog to forward to a central syslog server over TCP/UDP. Simple but limited filtering.
- **Log agents (Filebeat, Fluentd)**: Lightweight agents that tail log files or read from journald and ship to Elasticsearch (ELK stack) or Loki (Grafana stack). More flexible — can parse, filter, and enrich log data before shipping.
- **Centralized stores**: Elasticsearch stores logs as JSON documents for full-text search. Loki stores log streams indexed by labels (more efficient for large volumes). Both expose query APIs consumed by Grafana dashboards.

This architecture is the foundation for Phase 7's monitoring and observability module — where you will build a Prometheus + Grafana + Loki stack to collect metrics and logs from a multi-service application.
