## Overview

Your Node.js API is crashing under load. Every time it crashes, someone has to SSH in and manually restart it. The correct fix is not a shell script in a while loop — it is a systemd service unit that automatically restarts the process, logs output to the journal, and starts on boot. Understanding systemd is the difference between a fragile process and a managed service.

systemd is PID 1 — the first process the kernel starts. Everything else on the system is a child or descendant of systemd. It manages service dependencies, parallelizes boot, and provides structured logging. This lesson explains how it works before showing you the commands.

<Callout type="warning" title="Practicing systemd in Docker Containers">
In this course we practice systemd inside Docker containers using a special systemd-enabled image (`jrei/systemd-ubuntu:22.04`). This image requires the `--privileged` flag and specific `--tmpfs` mounts so systemd can manage cgroups correctly inside the container. In production, you manage systemd on real Linux hosts — the container gives you a safe sandbox to practice the commands without risking a real system.

On Apple Silicon Macs, add `--platform linux/amd64` to the docker run command — the image is amd64-only and runs via Rosetta 2 emulation.
</Callout>

## How It Works

### What PID 1 Does

When the Linux kernel finishes booting, it needs to hand control to user space. It does this by executing one process: PID 1. That process is responsible for starting everything else — every service, every daemon, every user session. If PID 1 dies, the kernel panics.

Before systemd, this was handled by **SysV init** — a set of shell scripts in `/etc/init.d/` that ran sequentially. Sequential startup was slow: each service waited for the previous one to finish. Dependency management was manual and fragile. A broken init script could hang the entire boot process.

**systemd** replaced this with:
- **Parallel startup**: services with no dependencies start simultaneously
- **Declarative dependency management**: you describe what depends on what, systemd calculates the order
- **Socket activation**: systemd can listen on a port and start a service on demand when the first connection arrives
- **Unified logging**: all service output goes to journald (structured binary log), accessible via `journalctl`
- **Automatic restart**: services can be configured to restart on failure with configurable backoff

### Unit File Anatomy

systemd manages resources through **unit files** — declarative configuration files describing what to start, how to start it, and what it depends on. Unit files live in:

- `/lib/systemd/system/` — installed by packages (do not edit)
- `/etc/systemd/system/` — your custom units and overrides

A service unit file has three sections:

```ini
# /etc/systemd/system/hello.service
[Unit]
Description=Hello World Service
After=network.target
Wants=network.target

[Service]
Type=simple
ExecStart=/usr/local/bin/hello.sh
Restart=on-failure
RestartSec=5s
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

**`[Unit]` section** — metadata and dependencies:

| Directive | Meaning |
|-----------|---------|
| `Description=` | Human-readable name shown in `systemctl status` |
| `After=` | Start this unit AFTER these units are active (ordering only) |
| `Requires=` | Hard dependency — if the required unit fails, this unit fails too |
| `Wants=` | Soft dependency — if the wanted unit fails, this unit still starts |

**`[Service]` section** — how to run the service:

| Directive | Meaning |
|-----------|---------|
| `Type=simple` | Process stays in foreground (default for most services) |
| `Type=forking` | Process daemonizes — forks and parent exits |
| `Type=oneshot` | Process runs once and exits (scripts, setup tasks) |
| `Type=notify` | Process notifies systemd when it is ready via `sd_notify()` |
| `ExecStart=` | Command to start the service (must be an absolute path) |
| `ExecStop=` | Command to stop it gracefully (default: SIGTERM) |
| `Restart=on-failure` | Restart only if the process exits with non-zero status |
| `Restart=always` | Restart regardless of exit status |
| `RestartSec=5s` | Wait this long before restarting |
| `StandardOutput=journal` | Send stdout to journald |
| `StandardError=journal` | Send stderr to journald |

**`[Install]` section** — when to start this service:

| Directive | Meaning |
|-----------|---------|
| `WantedBy=multi-user.target` | Enable this service as part of normal multi-user boot |
| `WantedBy=graphical.target` | Enable for graphical desktop boot |

### Targets: Replacing Runlevels

SysV init had numbered runlevels (0=halt, 1=single-user, 3=multi-user, 5=graphical, 6=reboot). systemd replaces these with **targets** — named groups of units that represent a system state:

| Target | SysV Equivalent | Meaning |
|--------|----------------|---------|
| `poweroff.target` | runlevel 0 | Shutdown |
| `rescue.target` | runlevel 1 | Single-user rescue mode |
| `multi-user.target` | runlevel 3 | Multi-user without GUI |
| `graphical.target` | runlevel 5 | Multi-user with GUI |
| `reboot.target` | runlevel 6 | Reboot |

When you "enable" a service with `WantedBy=multi-user.target`, systemd creates a symlink in `/etc/systemd/system/multi-user.target.wants/` pointing to your unit file. When the system reaches `multi-user.target`, it starts all units in that directory.

### Service Lifecycle

A unit moves through states:

```
loaded
  └─ inactive (not running)
       └─ activating (starting up — ExecStartPre, ExecStart running)
            └─ active (running normally)
                 └─ deactivating (stopping — ExecStop running)
                      └─ inactive
                           └─ failed (exited with error, not configured to restart)
```

`systemctl status` shows the current state plus recent journal entries for that unit.

### journald: Structured Logging

journald is systemd's logging daemon. Unlike traditional syslog (which wrote text files), journald stores **structured binary logs** with indexed metadata: timestamp, unit name, PID, priority level, hostname, and the message. This enables fast queries by any of those fields.

journald stores logs in `/run/log/journal/` (volatile, cleared on reboot) or `/var/log/journal/` (persistent, if that directory exists). Access logs with `journalctl`:

```bash
journalctl -u hello.service          # logs for specific unit
journalctl -u hello.service -f       # follow (like tail -f)
journalctl -b                        # logs since current boot
journalctl -b -1                     # logs from the previous boot
journalctl --since "1 hour ago"      # time-based filter
journalctl --since "09:00" --until "10:00"
journalctl -p err                    # only errors and above
journalctl -p err..crit              # range of priorities
journalctl -u nginx.service --since "2026-01-01" -p warning
```

Priority levels (from lowest to highest): `debug`, `info`, `notice`, `warning`, `err`, `crit`, `alert`, `emerg`.
