## Overview

Your nightly backup script has been running at 2:30 AM — except on the three nights it silently failed and nobody noticed for a week. Your monitoring system later revealed that systemd could have logged the failure and retried the job automatically. Scheduled tasks are deceptively tricky: they run unattended, they fail silently if not configured properly, and they compete with production workloads for CPU time. Understanding the scheduling mechanisms before writing a crontab is what separates a reliable system from a fragile one.

## How It Works

### cron: The Classic Daemon

The **cron daemon** (`crond` on RHEL/CentOS, `cron` on Ubuntu) runs continuously in the background. Every minute, it wakes up, scans the schedule tables, and executes any jobs whose schedule matches the current time.

cron reads job schedules from multiple locations:

| Location | Who controls it | Purpose |
|----------|----------------|---------|
| `/var/spool/cron/crontabs/<username>` | Each user (via `crontab -e`) | Per-user jobs |
| `/etc/cron.d/` | System (packages and admins) | System-level jobs in crontab format |
| `/etc/cron.daily/` | System | Shell scripts run once per day |
| `/etc/cron.weekly/` | System | Shell scripts run once per week |
| `/etc/cron.monthly/` | System | Shell scripts run once per month |
| `/etc/crontab` | Root only | System crontab with an extra username field |

### crontab Schedule Syntax

A crontab line has five time fields followed by the command:

```
# ┌───────────── minute        (0-59)
# │ ┌─────────── hour          (0-23)
# │ │ ┌───────── day of month  (1-31)
# │ │ │ ┌─────── month         (1-12 or JAN-DEC)
# │ │ │ │ ┌───── day of week   (0-7, both 0 and 7 = Sunday, or SUN-SAT)
# │ │ │ │ │
  * * * * *  command to execute
```

Common patterns:

```bash
30 2 * * *         # 2:30 AM every day
0 9 * * 1-5        # 9:00 AM on weekdays (Monday=1 through Friday=5)
*/15 * * * *       # every 15 minutes
0 0 1 * *          # midnight on the first of every month
0 */6 * * *        # every 6 hours (midnight, 6am, noon, 6pm)
30 2 * * 0         # 2:30 AM every Sunday
```

**Special strings** (shortcuts for common schedules):

| String | Equivalent |
|--------|------------|
| `@reboot` | Once at system startup |
| `@daily` | `0 0 * * *` |
| `@weekly` | `0 0 * * 0` |
| `@monthly` | `0 0 1 * *` |

**Editing crontabs:**

```bash
crontab -e         # edit your crontab in $EDITOR
crontab -l         # list your current crontab
crontab -r         # remove your crontab entirely (destructive — no confirmation)
crontab -l -u alice  # list another user's crontab (requires root)
```

**Critical cron pitfall:** cron runs jobs in a minimal environment. The `$PATH` is typically `/usr/bin:/bin` — not your full login shell `$PATH`. Scripts that work in your terminal may fail silently in cron because a command is not in cron's `$PATH`. Always use absolute paths in crontab entries or set `PATH` at the top of the crontab.

### systemd Timers: The Modern Alternative

A **systemd timer** is a unit file with a `.timer` extension that activates a corresponding `.service` unit at a scheduled time. Unlike cron:

- Output goes to journald — failures are logged and visible with `journalctl`
- `systemctl status backup.timer` shows when it last ran and whether it succeeded
- `Persistent=true` catches up missed runs after the system was off
- Resource limits and dependencies work exactly as with regular services

A systemd timer requires **two unit files**:

**The timer unit** (`/etc/systemd/system/backup.timer`):

```ini
[Unit]
Description=Daily Backup Timer

[Timer]
OnCalendar=*-*-* 02:30:00
Persistent=true

[Install]
WantedBy=timers.target
```

**The service unit** (`/etc/systemd/system/backup.service`):

```ini
[Unit]
Description=Daily Backup

[Service]
Type=oneshot
ExecStart=/usr/local/bin/backup.sh
StandardOutput=journal
StandardError=journal
```

The timer activates the service. The service does the actual work. Note that `Type=oneshot` is used for tasks that run once and exit (unlike `simple` which expects a long-running process).

**`OnCalendar=` syntax** — human-readable calendar expressions:

```bash
OnCalendar=*-*-* 02:30:00      # every day at 2:30 AM (year-month-day hour:min:sec)
OnCalendar=weekly               # every week (Monday 00:00:00)
OnCalendar=Mon *-*-* 09:00:00  # every Monday at 9 AM
OnCalendar=*-*-01 00:00:00     # first of every month at midnight
OnCalendar=hourly               # every hour at :00
```

Validate calendar expressions before deploying:

```bash
systemd-analyze calendar "*-*-* 02:30:00"
# Output shows next fire times:
#   Next elapse: Thu 2026-03-19 02:30:00 UTC
#   (in 13h 29min)
```

**Timer activation workflow:**

```bash
systemctl daemon-reload
systemctl enable --now backup.timer    # enable and start the timer immediately
systemctl list-timers                  # show all active timers and next fire time
systemctl status backup.timer          # when last triggered, next trigger
journalctl -u backup.service           # view output from previous runs
```

### cron vs systemd Timers: When to Use Each

| Feature | cron | systemd timer |
|---------|------|--------------|
| Logging | Silent by default (errors go to email if configured) | Always logs to journald |
| Failure visibility | `crontab -l` shows schedule, no run history | `systemctl status` shows last run time and result |
| Missed run recovery | No catch-up | `Persistent=true` runs once on next boot |
| Resource limits | Not supported | Full systemd resource controls |
| Dependencies | Not supported | Can depend on other units (network, mounts) |
| Simplicity | One line per job | Two unit files per job |
| Portability | Works on any Unix | systemd only (most modern Linux) |

**Use cron when:** the job is simple, you need one-liner syntax, or you are on a non-systemd system.

**Use systemd timers when:** you need reliable failure detection, catch-up after downtime, resource limits, or dependencies on other system state.

### at: One-Time Scheduled Tasks

`at` schedules a command to run once at a specific future time — not recurring. Useful for maintenance tasks at a specific time without setting up a cron entry:

```bash
at 03:00 tomorrow          # opens interactive prompt, Ctrl+D to submit
echo "/usr/local/bin/maintenance.sh" | at 03:00 tomorrow
at now + 2 hours           # 2 hours from now
at 23:00 2026-12-31        # specific date and time

atq                        # list pending at jobs
atrm 3                     # remove at job number 3
```

### nice and renice: Process Priority

All processes compete for CPU time. The kernel scheduler uses **priority values** (nice values) to determine how much CPU time each process gets.

Nice values range from **-20 (highest priority) to 19 (lowest priority)**. The default is 0.

```
-20  ──────────────────────────────────  19
High Priority                      Low Priority
(gets more CPU)                  (gets less CPU)
(requires root to set below 0)
```

**`nice`** starts a new command with a specific priority:

```bash
nice -n 10 /usr/local/bin/heavy-job.sh    # start with priority 10 (lower priority)
nice -n -5 /usr/local/bin/critical.sh     # start with priority -5 (higher — requires root)
```

**`renice`** changes the priority of a running process:

```bash
renice 15 -p 12345         # set PID 12345 to priority 15 (lower — be nice to others)
renice -5 -p 12345         # set to -5 (higher — requires root)
renice 10 -u nobody        # change all processes owned by user 'nobody'
```

**Practical use:** Background batch jobs (backups, log processing, report generation) should run at positive nice values (10-19). This ensures they yield CPU time to interactive processes and production services when load increases. A backup running at nice 0 can compete with database queries; at nice 15, the database always wins.
