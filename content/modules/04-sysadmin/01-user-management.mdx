## Overview

A developer accidentally runs `sudo rm -rf /` and destroys the server. The post-mortem reveals that every engineer had unrestricted sudo access because "it was easier to set up." Understanding how Linux identity and privilege actually work is not optional for system administration — it is the first thing you configure when a server is provisioned.

Linux does not think in usernames. The kernel thinks in numbers. Every access decision at the kernel level uses UIDs and GIDs. Usernames are a human-readable convenience layer on top of those numbers. This lesson explains the mechanism first, then shows you how to configure it correctly.

## How It Works

### The Identity Database: /etc/passwd, /etc/shadow, /etc/group

**`/etc/passwd`** is the primary user database. Despite the name, it has not stored passwords since the 1980s. It has seven colon-separated fields:

```
root:x:0:0:root:/root:/bin/bash
alice:x:1001:1001:Alice Smith:/home/alice:/bin/bash
daemon:x:1:1:daemon:/usr/sbin:/usr/sbin/nologin
```

| Field | Example | Meaning |
|-------|---------|---------|
| username | `alice` | Human-readable name (kernel does not use this) |
| password | `x` | Placeholder — actual hash is in `/etc/shadow` |
| UID | `1001` | Numeric user identity — what the kernel uses |
| GID | `1001` | Primary group ID |
| GECOS | `Alice Smith` | Comment field — full name, contact info |
| home | `/home/alice` | Home directory created at login |
| shell | `/bin/bash` | Default shell (use `/usr/sbin/nologin` for service accounts) |

The `x` in the password field means "check `/etc/shadow`." The shadow file is readable only by root (mode 640, owned by root:shadow):

```
alice:$6$rounds=656000$salt$hash...:19000:0:99999:7:::
```

Fields: `username : hashed-password : last-change : min-days : max-days : warn-days : inactive : expire : reserved`

The hash prefix `$6$` identifies SHA-512 (the current standard). `$5$` is SHA-256. `$y$` is yescrypt (Ubuntu 22.04+, more resistant to GPU cracking). A `!` prefix means the account is locked.

**`/etc/group`** maps group names to GIDs and member lists:

```
sudo:x:27:alice,bob
docker:x:998:alice
developers:x:1002:alice,carol,dave
```

Fields: `groupname : password : GID : member-list`

### UID Ranges and Their Meaning

The kernel does not distinguish "user" from "system account" — that is a convention:

| Range | Convention | Examples |
|-------|------------|---------|
| 0 | Root | `root` — unrestricted kernel access |
| 1–99 | Static system accounts | `daemon`, `bin`, `sys` (allocated by distro) |
| 100–999 | Dynamic system accounts | `www-data`, `mysql`, `docker` (created by packages) |
| 1000+ | Regular user accounts | `alice`, `bob` — interactive human users |
| 65534 | Nobody | `nobody` — processes that should have no access |

When `ls -l` shows `alice` as owner, it is looking up UID 1001 in `/etc/passwd` and printing the username. If the file was created on a different system where UID 1001 belonged to someone else, you will see that different username — the UID is what the kernel stored, not the name.

<Callout type="deep-dive" title="UIDs, GIDs, and File Permissions">
In the Linux Fundamentals lesson on file permissions, you learned that permission checks compare UIDs and GIDs — not names. That same principle applies here: when the kernel evaluates whether process UID 1001 can read a file owned by UID 1001, it sees a match and grants access. The mapping from UID 1001 to username `alice` is only used by tools like `ls -l` to display human-readable output.

This has a practical consequence: if you copy files between systems with different `/etc/passwd` configurations, ownership can appear to change — because the same UID now maps to a different username. Always verify ownership by UID number when migrating systems.
</Callout>

### The sudo Mechanism

`sudo` allows a user to run a command with elevated privileges (typically as root). The mechanism:

1. You run `sudo systemctl restart nginx`
2. `sudo` reads `/etc/sudoers` to check if your user is allowed to run that command
3. PAM authenticates you (typically by checking your password against `/etc/shadow`)
4. If both checks pass, `sudo` forks a child process with UID 0 (root) and executes the command
5. The parent (`sudo`) logs the action to syslog

The `/etc/sudoers` file controls exactly who can do what:

```
# Format: who  host=(as-user:as-group) commands

# Root can do anything
root    ALL=(ALL:ALL) ALL

# Members of sudo group can run anything with password
%sudo   ALL=(ALL:ALL) ALL

# alice can restart nginx without a password
alice   ALL=(root) NOPASSWD: /bin/systemctl restart nginx

# Include drop-in files from this directory
#includedir /etc/sudoers.d
```

<Callout type="warning" title="Never Edit /etc/sudoers Directly">
Always use `visudo` to edit `/etc/sudoers`. `visudo` opens the file in your editor and validates syntax before saving. A syntax error in `/etc/sudoers` causes `sudo` to refuse ALL commands — including attempts to fix the file. Without `sudo`, you cannot become root unless you have direct root access or can boot into recovery mode. The `visudo` validation step is what prevents this lockout scenario.
</Callout>

The preferred pattern for managing sudo grants is drop-in files in `/etc/sudoers.d/`. Each file in that directory is included automatically. This avoids touching the main sudoers file:

```bash
# Create a drop-in file for the deploy user
echo "deploy ALL=(root) NOPASSWD: /bin/systemctl restart myapp" > /etc/sudoers.d/deploy
chmod 440 /etc/sudoers.d/deploy
```

### useradd vs adduser

**`useradd`** is the low-level system call wrapper — available on all Linux distributions, configurable, but requires explicit flags:

```bash
useradd -m -s /bin/bash -c "Alice Smith" alice
# -m: create home directory
# -s: set login shell
# -c: GECOS comment field
```

**`adduser`** is a Perl/Python wrapper available on Debian/Ubuntu that provides interactive prompts and sane defaults:

```bash
adduser alice   # prompts for password, GECOS, creates home dir automatically
```

For automation and scripts, use `useradd` with explicit flags. For interactive admin tasks on Debian/Ubuntu, `adduser` is faster.
