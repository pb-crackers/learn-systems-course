## Overview

Users are reporting timeouts. The service is running — systemctl shows it active — but responses are taking 30 seconds instead of 300 milliseconds. You have 10 minutes before the incident escalates. Knowing exactly which monitoring command to run, and what each number means, is the difference between finding the problem and staring at a wall of output.

There are four resources to monitor: CPU, memory, disk I/O, and network. Each has different tools and different signatures when saturated. This lesson teaches the interpretation pattern first, then the commands.

## How It Works

### The Four Resources

Every performance problem on a Linux server falls into one of four categories:

| Resource | Saturated when... | Primary tools |
|----------|------------------|---------------|
| CPU | Processes are waiting for CPU time | `top`, `vmstat`, `mpstat` |
| Memory | System is using swap | `free`, `top`, `vmstat` |
| Disk I/O | Processes are waiting for disk reads/writes | `iostat`, `vmstat`, `iotop` |
| Network | Bandwidth exhausted or high packet loss | `iftop`, `netstat`, `ss`, `ping` |

The key diagnostic pattern: **load average tells you something is wrong; the other tools tell you what**.

### Load Average: The Starting Point

Load average is the most prominent indicator in system monitoring. It appears in `top`, `uptime`, and `cat /proc/loadavg`. It reports three numbers:

```
load average: 2.45, 1.83, 1.21
              ↑      ↑      ↑
           1-min  5-min  15-min
```

**What load average measures:** the average number of processes that are either:
1. Running (currently using a CPU), or
2. In the "uninterruptible sleep" state (waiting for I/O — cannot be interrupted by a signal)

**The key reference point is the number of CPU cores.** A load of 1.00 on a single-core system means the CPU is exactly fully utilized. A load of 1.00 on a 4-core system means it is 25% utilized.

```
Single-core system:
  load 0.5  = 50% utilized (CPU is idle half the time)
  load 1.0  = 100% utilized (CPU is always busy, no queue)
  load 2.0  = 200% load (1 running + 1 waiting — there is a queue)

4-core system:
  load 4.0  = 100% utilized across all 4 cores
  load 8.0  = 200% load (4 running + 4 waiting — there is a queue)
```

**Reading the three values:**
- 1-min > 5-min > 15-min: load is **increasing** — problem is getting worse
- 1-min < 5-min < 15-min: load is **decreasing** — problem is resolving
- 1-min >> 15-min: sudden spike — something just happened

**Critical distinction:** High load with high `%wa` (I/O wait) is an I/O bottleneck. High load with high `%us` (user CPU) is a CPU bottleneck. The load average alone cannot tell you which — you need vmstat or top.

### top: The Overview Dashboard

`top` is the Swiss Army knife — it shows all four resource categories simultaneously.

```
top - 14:23:05 up 2 days,  3:14,  2 users,  load average: 2.45, 1.83, 1.21
Tasks: 187 total,   2 running, 185 sleeping,   0 stopped,   0 zombie
%Cpu(s): 45.2 us, 12.3 sy,  0.0 ni, 38.5 id,  3.8 wa,  0.0 hi,  0.2 si
MiB Mem :  7823.5 total,   456.2 free,  5234.1 used,  2133.2 buff/cache
MiB Swap:  2048.0 total,  1024.0 free,  1024.0 used.  2134.2 avail Mem

  PID USER      PR  NI    VIRT    RES    SHR S  %CPU  %MEM     TIME+ COMMAND
12345 www-data  20   0  678944 456782  12344 R  87.3   5.7   4:32.12 node
 9876 mysql     20   0 1234567 890123  23456 S   8.1  11.1  12:45.67 mysqld
```

**CPU line interpretation:**
| Field | Meaning |
|-------|---------|
| `us` | User space CPU — your application code |
| `sy` | System (kernel) CPU — system calls, I/O scheduling |
| `ni` | Nice — CPU used by niced (low-priority) processes |
| `id` | Idle — unused CPU capacity |
| `wa` | I/O wait — CPU is idle but waiting for disk/network I/O |
| `hi` | Hardware interrupts |
| `si` | Software interrupts |

**Memory line:** `used` = actively used memory. `buff/cache` = disk cache (the kernel uses free RAM for caching — this is normal and healthy). `avail Mem` is how much memory is actually available without touching swap — this is the number to watch.

**Process column interpretation:**

| Column | Meaning |
|--------|---------|
| `PID` | Process ID |
| `PR` | Priority as seen by the kernel scheduler |
| `NI` | Nice value (-20 to 19) |
| `VIRT` | Virtual memory — all mapped address space (includes memory not yet loaded) |
| `RES` | Resident set size — RAM actually used right now |
| `SHR` | Shared memory — portions shared with other processes (libraries) |
| `S` | State: R=running, S=sleeping, D=uninterruptible sleep (I/O wait), Z=zombie |
| `%CPU` | CPU usage percentage |
| `%MEM` | Percentage of total RAM |

**Useful top interactive commands:** `M` sort by memory, `P` sort by CPU, `k` kill a process, `q` quit.

### vmstat: Sampling System-Wide Activity

`vmstat` shows aggregate resource utilization per sample interval. Unlike `top` (which shows a point-in-time snapshot), running `vmstat 1 5` takes 5 samples one second apart, showing trends:

```bash
$ vmstat 1 5
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa
 2  0      0 456000  12000 2100000    0    0     0    45  320  890 45 12 38  4
 3  1      0 450000  12000 2100000    0    0   200  1245  450 1200 52 15 25  8
 1  0      0 455000  12000 2100000    0    0     0    30  300  780 43 11 42  4
```

**Column interpretation:**

| Section | Column | Meaning |
|---------|--------|---------|
| procs | `r` | Run queue — processes waiting for CPU (> number of CPUs = CPU saturated) |
| procs | `b` | Blocked processes in uninterruptible sleep (> 0 usually = I/O bottleneck) |
| memory | `swpd` | Amount of swap in use (> 0 = memory pressure) |
| memory | `free` | Free memory |
| memory | `buff/cache` | Kernel buffer/cache (NOT wasted — it speeds up disk access) |
| swap | `si` | Swap in — pages read from swap disk to RAM (> 0 = serious memory pressure) |
| swap | `so` | Swap out — pages written from RAM to swap (> 0 = serious memory pressure) |
| io | `bi` | Blocks read from disk per second |
| io | `bo` | Blocks written to disk per second |
| cpu | `us/sy/id/wa` | Same as top's CPU line (user/system/idle/iowait) |

**Diagnosis signals:**
- `r` consistently > number of CPU cores: **CPU bottleneck**
- `b` > 0, `wa` > 10%: **I/O bottleneck**
- `si` or `so` > 0: **memory pressure** (thrashing)

### iostat: Disk I/O Detail

`iostat` (from the `sysstat` package) shows per-device disk I/O statistics:

```bash
$ iostat -x 1 3
Device    r/s    w/s  rMB/s  wMB/s  r_await  w_await  %util
sda      45.2   12.3    2.4    1.1      2.3     12.4   23.5
sdb       0.1  234.5    0.0   58.6      0.8     85.3   98.7  ← SATURATED
```

**Key columns:**

| Column | Meaning |
|--------|---------|
| `r/s`, `w/s` | Read/write operations per second |
| `rMB/s`, `wMB/s` | Read/write throughput in MB/s |
| `r_await`, `w_await` | Average time (ms) for read/write requests to complete |
| `%util` | Percentage of time the device was busy — > 80% indicates saturation |

`%util` near 100% means the disk cannot keep up with the I/O requests. `await` values above 10-20ms for SSD or 100ms for spinning disk indicate queue buildup.

### The Bottleneck Identification Pattern

When a system is slow, run this sequence:

```
Step 1: uptime or top header
  → Check load average vs number of CPUs

Step 2: vmstat 1 5
  → r > CPU count?  → CPU bottleneck
  → b > 0, wa > 10? → I/O bottleneck
  → si or so > 0?   → Memory pressure

Step 3 (if I/O): iostat -x 1 3
  → Which device has %util near 100%?
  → What is the await time?

Step 4 (if CPU): top (sort by CPU with P)
  → Which process is consuming the CPU?
  → Is it expected behavior or a runaway process?

Step 5 (if memory): free -h
  → How much swap is in use?
  → What is the avail Mem number?
  → top (sort by MEM with M) — which process is the largest?
```

Common bottleneck signatures:

| Observation | Diagnosis |
|-------------|-----------|
| High load + high `%us` + high CPU process | CPU-bound: one process consuming all CPU |
| High load + high `%wa` + `b > 0` | I/O-bound: processes waiting for disk |
| High load + `si`/`so` > 0 + low `free` | Memory exhaustion: system is swapping |
| Low load + slow responses | Not resource-bound: check network, application, external dependencies |

<Callout type="tip" title="What's Next: Prometheus and Grafana">
These command-line tools show you a snapshot of one machine at one moment. `vmstat 1 5` tells you what the system was doing in the last 5 seconds. But production questions sound like: "Was the disk I/O high before the deployment at 2:30 PM, or did it start after?" That requires storing metrics over time.

In Phase 7 (Monitoring and Observability), you will learn **Prometheus** for collecting and storing metrics continuously, and **Grafana** for visualizing trends over time — turning these point-in-time snapshots into a monitoring system that can show you load average graphs, alert on `%util` thresholds, and correlate deployments with performance changes.
</Callout>
